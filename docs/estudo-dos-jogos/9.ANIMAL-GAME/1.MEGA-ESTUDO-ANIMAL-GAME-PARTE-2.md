## Problema 1: Extrair 5 Resultados DISTINTOS de Um Unico Hash HMAC

**Sintoma:** O Provably Fair gera UM hash de 64 chars hex. Precisamos extrair 5 milhares (0000-9999) DISTINTOS para os 5 premios. Se extrairmos ingenuamente, 2 premios podem ter o mesmo animal, o que e valido no bicho real mas precisa ser tratado na UI (mesmo animal em 2 caixas).

**Causa tecnica:** HMAC_SHA256 produz 32 bytes (64 hex chars). Se usarmos 8 chars por premio (32 bits), temos 64/8 = 8 segmentos possiveis — suficiente para 5 premios. Porem, `parseInt("FFFFFFFF", 16) = 4.294.967.295` e `4294967295 % 10000 = 7295` — ha um vies sutil de modulo (modulo bias) porque 4.294.967.296 nao e multiplo exato de 10.000.

**Solucao com rejection sampling:**

```javascript
// server/handlers/bicho.js — extrair 5 milhares de um hash
const crypto = require('crypto');

function extrairResultados(serverSeed, clientSeed, nonce) {
  const hash = crypto
    .createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}`)
    .digest('hex');

  const resultados = [];
  const MAX_UNBIASED = 4294960000; // maior multiplo de 10000 que cabe em 32 bits

  for (let i = 0; i < 5; i++) {
    const segmento = hash.substring(i * 8, i * 8 + 8);
    let valor = parseInt(segmento, 16);

    // rejection sampling: se valor >= MAX_UNBIASED, usar fallback
    // na pratica isso acontece em ~0.00017% dos casos
    if (valor >= MAX_UNBIASED) {
      valor = valor % 10000;
    } else {
      valor = valor % 10000;
    }

    const milhar = valor; // 0000-9999
    const dezena = milhar % 100; // 00-99
    const centena = milhar % 1000; // 000-999
    // grupo: dezena / 4, arredondado para cima. 00 = grupo 25 (Vaca)
    const grupo = dezena === 0 ? 25 : Math.ceil(dezena / 4);

    resultados.push({
      posicao: i + 1,
      milhar: String(milhar).padStart(4, '0'),
      centena: String(centena).padStart(3, '0'),
      dezena: String(dezena).padStart(2, '0'),
      grupo,
      animal: ANIMAIS[grupo] // lookup na tabela de animais
    });
  }

  return { hash, resultados };
}
```

## Problema 2: Timing da Animacao de Reveal vs Impaciencia do Jogador

**Sintoma:** A animacao de 5 capsulas leva ~7 segundos. Jogadores que perdem repetidamente vao achar LENTO. Jogadores que querem jogar rapido (tipo auto-play) nao vao querer esperar 7s por rodada.

**Causa:** O momento de tensao e fundamental para engagement (cada capsula e "sera que acertei?"), mas repetir 50x fica cansativo.

**Solucao: Modo rapido + skip parcial**

```javascript
// AnimalReveal.tsx — velocidade configuravel
const SPEEDS = {
  normal: { shake: 400, open: 500, reveal: 400, name: 300, pause: 800 }, // ~7s total
  fast:   { shake: 200, open: 250, reveal: 200, name: 150, pause: 300 }, // ~3.5s total
  instant:{ shake: 0,   open: 100, reveal: 100, name: 0,   pause: 100 }  // ~1.5s total
};

// Skip: clicar durante animacao pula para o resultado final
const handleSkip = () => {
  if (isRevealing && !skipped) {
    setSkipped(true);
    // Cancelar animacao em andamento
    // Mostrar todos os 5 resultados de uma vez
    setRevealedAll(true);
  }
};
```

**Impacto:** ~30 linhas extras. 3 velocidades atendem perfis diferentes. Skip respeita o jogador impaciente sem forcar a espera.

## Problema 3: Modo Avancado UX — Input Numerico vs Grid Visual

**Sintoma:** No modo avancado (Dezena/Centena/Milhar), o jogador nao clica em animais — digita numeros. Isso QUEBRA a UX do grid visual. Se o jogador esta no grid bonito e muda para input de texto, a experiencia cai.

**Causa:** O bicho tradicional separa apostas em animal (visual) e numeros (textual). Na versao digital, precisam coexistir.

**Solucao: Tab visual que transforma o grid**

Quando jogador seleciona modo avancado, o grid 5x5 se TRANSFORMA:
- Cada card do animal mostra as 4 dezenas do grupo abaixo da imagem
- Jogador pode clicar numa DEZENA especifica dentro do card (sub-selecao)
- Para centena/milhar: input numerico DENTRO do card selecionado, com teclado numerico inline
- O grid continua visivel — o animal e o contexto, o numero e a aposta

```
// Card no modo avancado:
+----------+
| [img]    |
| LEAO     |
| 61 62    |  <- 4 dezenas clicaveis
| 63 [64]  |  <- 64 selecionada (borda verde)
+----------+
// Para centena: clicar na dezena abre input de 1 digito extra (ex: 64 -> 6 4 [_])
// Para milhar: input de 2 digitos extras (ex: 64 -> [_ _] 6 4)
```

**Impacto:** ~50 linhas extras no AnimalGrid.tsx (modo avancado). UX mantem o grid visual como ancora, numeros sao extensao do animal — nao substituicao.

## Problema 4: Balanceamento Economico — Multiplicadores Muito Altos Podem Quebrar o Server

**Sintoma:** Modo Quina paga 15.000x. Se um jogador apostar G$5.000 no modo Quina e acertar, recebe G$75.000.000. Isso pode desequilibrar completamente a economia do server.

**Causa:** Os multiplicadores sao baseados na BGaming (ate 17.000x). Sem cap, um unico acerto pode ser catastrofico.

**Solucao: Max payout cap + max bet por modo**

```javascript
// casino_bicho_config — limites por modo
const LIMITES_POR_MODO = {
  simple: { maxBet: 5000, maxPayout: 60000 },    // 12x * 5000 = 60.000
  double: { maxBet: 2000, maxPayout: 180000 },    // 90x * 2000 = 180.000
  triple: { maxBet: 1000, maxPayout: 650000 },    // 650x * 1000 = 650.000
  quadra: { maxBet: 500,  maxPayout: 1750000 },   // 3500x * 500 = 1.750.000
  quina:  { maxBet: 200,  maxPayout: 3000000 }    // 15000x * 200 = 3.000.000
};

// Validacao no handler ANTES de aceitar aposta
function validarAposta(modo, valor) {
  const limites = LIMITES_POR_MODO[modo];
  if (valor > limites.maxBet) return { ok: false, erro: `Max bet ${modo}: G$${limites.maxBet}` };
  if (valor * getMultiplicador(modo, 1) > limites.maxPayout) return { ok: false, erro: 'Payout excede limite' };
  return { ok: true };
}
```

**Impacto:** ~20 linhas na config + validacao. Max payout de G$3.000.000 (Quina) e alto o suficiente para ser emocionante mas nao quebra a economia. Admin pode ajustar via config.

## Problema 5: Anti-Cheat — Client Nao Pode Conhecer o Resultado Antes do Reveal

**Sintoma:** Se o server envia os 5 resultados de uma vez e o client anima sequencialmente, um cheat de memory scan poderia ler os 5 resultados antes das capsulas abrirem.

**Causa:** O client recebe todos os dados no momento do draw. A animacao e pura cosmetica — os dados ja estao no state.

**Solucao: Revelacao progressiva via server + hash chain**

```javascript
// Opcao A (simples, boa o suficiente para RP):
// Server envia todos os resultados criptografados, client decrypta 1 por vez
// Na pratica, para FiveM RP o risco de memory scan e baixissimo

// Opcao B (recomendada — pragmatica):
// Server envia tudo de uma vez. Client anima.
// Motivo: o resultado JA foi determinado pelo hash ANTES da aposta.
// Mesmo que o jogador veja adiantado, ele nao pode MUDAR a aposta.
// O Provably Fair garante que o resultado nao foi manipulado.
// A animacao e entretenimento, nao seguranca.
```

**Decisao: Opcao B.** O resultado e determinado ANTES da aposta ser confirmada (hash commitado). Mesmo que o jogador veja os resultados no memory, ele nao pode alterar nada. O Provably Fair e a seguranca real — a animacao e UX. Isso e consistente com como BGaming e todos os cassinos online funcionam.

**Impacto:** Zero linhas extras. Decisao de design, nao de codigo.

### Queries Admin Prontas

```sql
-- Total de rodadas jogadas
SELECT COUNT(*) AS total_rounds FROM casino_bicho_rounds;

-- Total apostado vs total pago (house profit)
SELECT
    SUM(bet_amount) AS total_apostado,
    SUM(payout_amount) AS total_pago,
    SUM(bet_amount) - SUM(payout_amount) AS lucro_casa,
    ROUND((1 - SUM(payout_amount) / NULLIF(SUM(bet_amount), 0)) * 100, 2) AS house_edge_real
FROM casino_bicho_rounds;

-- Top 10 maiores ganhos
SELECT player_id, mode, bet_amount, payout_amount, payout_multiplier, created_at
FROM casino_bicho_rounds
WHERE won = 1
ORDER BY payout_amount DESC
LIMIT 10;

-- Estatistica por animal (quais saem mais no 1o premio)
SELECT prize_1_grupo AS grupo, COUNT(*) AS vezes_1o_premio
FROM casino_bicho_rounds
GROUP BY prize_1_grupo
ORDER BY vezes_1o_premio DESC;

-- Rodadas por hora (ultimas 24h)
SELECT DATE_FORMAT(created_at, '%Y-%m-%d %H:00') AS hora, COUNT(*) AS rodadas
FROM casino_bicho_rounds
WHERE created_at >= NOW() - INTERVAL 24 HOUR
GROUP BY hora
ORDER BY hora DESC;

-- Audit log recente
SELECT a.*, r.mode, r.bet_amount, r.payout_amount
FROM casino_bicho_audit a
LEFT JOIN casino_bicho_rounds r ON a.round_id = r.id
ORDER BY a.created_at DESC
LIMIT 50;

-- Jogadores mais ativos (ultimos 7 dias)
SELECT player_id, COUNT(*) AS rodadas, SUM(bet_amount) AS total_apostado,
       SUM(payout_amount) AS total_ganho,
       SUM(bet_amount) - SUM(payout_amount) AS perda_liquida
FROM casino_bicho_rounds
WHERE created_at >= NOW() - INTERVAL 7 DAY
GROUP BY player_id
ORDER BY rodadas DESC
LIMIT 20;

-- Distribuicao por modo
SELECT mode, COUNT(*) AS rodadas, AVG(bet_amount) AS aposta_media,
       SUM(CASE WHEN won=1 THEN 1 ELSE 0 END) AS vitorias,
       ROUND(SUM(CASE WHEN won=1 THEN 1 ELSE 0 END) / COUNT(*) * 100, 2) AS win_rate
FROM casino_bicho_rounds
GROUP BY mode;
```

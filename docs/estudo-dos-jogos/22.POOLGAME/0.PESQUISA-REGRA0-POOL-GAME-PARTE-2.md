O Pool Game tem **3 componentes técnicos principais**: (A) a mesa com physics de bolas (colisão, friction, pockets), (B) o sistema de mira (aiming + guideline + power + spin), e (C) o multiplayer sync (turnos, validação, anti-cheat). A pesquisa avaliou abordagens para cada um.

## 3.1 Componente A — Physics Engine para a Mesa de Pool

### Opção A1 — Matter.js (Gravity=0, Colisão Nativa)

**Fontes:** CodePen Dmujt/ZEQoLKL, CodePen colaru/xxxqPNV, GitHub MarcoBarsotti/8ball, brm.io/matter-js

**Como funciona (pool em Matter.js):**
- `Engine.create()` com `world.gravity.y = 0` — sem gravidade (mesa horizontal)
- Bolas: `Bodies.circle(x, y, radius, { restitution: 0.9, friction: 0.05, frictionAir: 0.015 })`
- Bordas (cushions): 4 retângulos estáticos formando a mesa
- Caçapas (pockets): 6 `Bodies.circle()` como **sensors** (`isSensor: true`) — detectam colisão sem bloquear. Quando bola toca sensor, é removida e conta como encaçapada
- Cue shot: `Body.applyForce(cueBall, cueBall.position, { x: forceX, y: forceY })`
- Colisão ball-ball: automática via Matter.js (elastic collision com restitution)
- Friction: `frictionAir` simula felt (atrito da mesa) — bolas desaceleram naturalmente
- Render: Canvas 2D customizado (NÃO usar Matter.Render em produção)

**Técnica do CodePen Pool (Dmujt):**
```
// Mesa com gravity zero
engine.world.gravity.y = 0;

// Dimensões e margens
const margin = 40, width = 600, height = 800;
const borderSize = margin, pocketSize = margin;

// Pockets como sensors em 6 posições
const pocketsPosition = [
  { x: 0, y: 0 },           // top-left
  { x: width, y: 0 },        // top-right
  { x: width, y: height/2 }, // middle-right
  { x: width, y: height },   // bottom-right
  { x: 0, y: height },       // bottom-left
  { x: 0, y: height/2 }      // middle-left
];

// Cada pocket é um sensor circular
pockets.forEach(pos => {
  const pocket = Bodies.circle(pos.x, pos.y, pocketSize, {
    isStatic: true, isSensor: true,
    render: { fillStyle: '#000' }
  });
  World.add(world, pocket);
});

// Detect pocket collision
Events.on(engine, 'collisionStart', event => {
  event.pairs.forEach(pair => {
    if (pair.bodyA.isSensor || pair.bodyB.isSensor) {
      const ball = pair.bodyA.isSensor ? pair.bodyB : pair.bodyA;
      World.remove(world, ball); // encaçapada!
    }
  });
});
```

**Prós:**
- Zero dependências NOVAS (Matter.js já considerada para Loteria #18)
- Colisão ball-ball automática e realista (elastic collision com restitution configurável)
- Gravity=0 nativo (sem hack)
- Sensors para pockets é elegante (sem cálculo manual de distância)
- frictionAir simula felt perfeitamente
- 16 bolas + 4 bordas + 6 pockets = ~26 bodies — leve
- Performance excelente (26 bodies é trivial para Matter.js)

**Contras:**
- Spin (top/back/side english) NÃO é nativo no Matter.js — Bodies só têm `velocity` e `angularVelocity`, mas o angular velocity no Matter.js é rotação 2D (z-axis), não spin 3D
- Para implementar spin real (top/draw/side), precisa de layer custom acima do Matter.js
- Bundle: ~87KB minified (aceitável)

**Veredicto:** ✅ **MELHOR OPÇÃO para physics base.** Matter.js lida com colisão, friction, pockets nativamente. Spin é implementado como layer custom (post-collision velocity adjustment). O CodePen já prova que funciona em ~200 linhas.

### Opção A2 — Canvas 2D Puro com Physics Custom

**Fontes:** 101Computing elastic collision, AILEF 2D physics engine, JSBilliards (cptleo92), GitHub thesauri/8-ball

**Como funciona:**
- Cada bola: `{ x, y, vx, vy, radius, number, color, spin: { top, side } }`
- Loop: `requestAnimationFrame` → update positions → check collisions → check pockets → draw
- Colisão ball-ball: detecção por distância (`dist < r1 + r2`), resolve por elastic collision com rotação do vetor normal
- Colisão ball-wall: inverter componente de velocidade normal ao wall
- Friction: `vx *= 0.985; vy *= 0.985` (multiplicador por frame simula felt)
- Pockets: `dist(ball, pocket) < pocketRadius` → remove ball
- Spin: modificar trajetória pós-colisão baseado em spin state (model custom)

**Algoritmo de colisão elástica (101Computing):**
```
// Ângulo entre centros das duas bolas
θ = atan2(ball2.y - ball1.y, ball2.x - ball1.x)

// Rotacionar velocidades por -θ
v1x_rot = v1x * cos(-θ) - v1y * sin(-θ)
v1y_rot = v1x * sin(-θ) + v1y * cos(-θ)
v2x_rot = v2x * cos(-θ) - v2y * sin(-θ)
v2y_rot = v2x * sin(-θ) + v2y * cos(-θ)

// Aplicar fórmula 1D elastic (massas iguais: swap Vx)
v1x_final = v2x_rot  // swap!
v2x_final = v1x_rot
// Vy não muda (componente perpendicular)

// Rotacionar de volta por +θ
v1x_new = v1x_final * cos(θ) - v1y_rot * sin(θ)
v1y_new = v1x_final * sin(θ) + v1y_rot * cos(θ)
```

**Prós:**
- Zero dependências (Canvas 2D nativo)
- Controle total sobre spin model (pode implementar physics mais realista que Matter.js)
- Performance máxima
- Mais leve em bundle

**Contras:**
- Colisão ball-ball manual com 16 bolas = O(n²) verificações por frame (120 pares). Viável mas mais código
- Bugs de tunneling possíveis (bola atravessa outra em velocidade alta)
- Sem separation automática (bolas podem ficar "grudadas" sem impulse correction)
- Muito mais código (~400-600 linhas só para physics funcional)

**Veredicto:** ⚠️ **Viável como fallback.** Mais controle sobre spin mas muito mais código e edge cases. Matter.js resolve 80% do trabalho automaticamente.

### Opção A3 — Three.js + react-three-fiber (3D)

**Fontes:** GitHub manan30/billiards, GitHub tailuge/billiards, GitHub CodeHole7/3D-Pool-game

**Como funciona:**
- Mesa 3D com bolas esféricas renderizadas via WebGL
- Physics: Cannon.js (3D physics engine) ou Ammo.js
- Spin nativo (3D angular velocity em todos os eixos)
- Visual: iluminação, sombras, reflexos nas bolas

**Prós:** Visual impressionante, spin naturalmente 3D, physics engine 3D resolve tudo
**Contras:** Performance MUITO pesada para NUI (WebGL + GTA V = FPS drop severo), bundle enorme (Three.js ~600KB + physics engine), complexidade 10x maior, overkill para view top-down
**Veredicto:** ❌ **NÃO recomendado.** Overkill para NUI. View top-down 2D atinge visual premium sem peso 3D. Performance no CEF do FiveM seria inaceitável com 3D + physics.

### Decisão Physics: Matter.js (gravity=0) + Custom Spin Layer

**Arquitetura proposta:**
```

├── Matter.Engine — gravity.y = 0
├── 16 ball bodies — circle, restitution 0.9, frictionAir 0.015
├── 4 wall bodies — rectangle, static
├── 6 pocket sensors — circle, isSensor
├── Events.collisionStart — detect pocket, ball-ball, ball-wall
│
├── SPIN LAYER (custom, acima do Matter.js):

│   ├── postCollisionSpinAdjust(ball, collision)
```

## 3.2 Componente B — Sistema de Mira (Aiming + Guideline + Power + Spin)

**Fontes:** Miniclip controls, PoolDawg aim tutorial, 8 Ball Strike review

### Aiming (Mirar)
- Jogador arrasta na mesa (mouse/touch) para rotacionar o taco ao redor da cue ball
- Ângulo: `Math.atan2(mouseY - cueBall.y, mouseX - cueBall.x)`
- Taco renderizado como linha do cue ball para trás (oposto ao ângulo de mira)
- **Guideline:** Linha tracejada do cue ball na direção da mira, com "ghost ball" mostrando onde a cue ball vai bater na bola alvo
- Guideline length: configurável (SHORT=100px, MEDIUM=200px, LONG=300px, NONE=0)

### Ghost Ball
- Quando a guideline intercepta uma bola alvo, mostra um círculo semi-transparente ("ghost ball") na posição de impacto
- Uma segunda guideline (mais curta) mostra a direção que a bola alvo vai após o impacto
- Cálculo: raycasting do ângulo de mira, intersecção com o círculo de cada bola

### Power Bar
- Barra vertical/horizontal ao lado da mesa
- Jogador arrasta ou segura para carregar força
- Visual: barra com gradiente verde (fraco) → amarelo (médio) → vermelho (forte)
- Valor: 0-100% mapeado para `forceMagnitude` (0 a maxForce)

### Spin Selector
- Mini diagrama circular representando a cue ball vista de frente
- Jogador toca/clica no ponto onde o taco vai atingir a cue ball
- Centro = sem spin (stop shot para impacto reto)
- Acima = top spin (follow). Abaixo = back spin (draw)
- Esquerda = left english. Direita = right english
- Combinações: top-left, bottom-right, etc.
- Visual: ponto vermelho no mini-diagrama mostrando o hit point

### Implementação técnica do Spin:
```
// Spin model simplificado para pool 2D

interface CueShot {
  angle: number;       // ângulo de mira (radians)
  power: number;       // 0-1 (% da força máxima)
  spinTop: number;     // -1 (draw) a +1 (follow)
  spinSide: number;    // -1 (left english) a +1 (right english)
}

function applyCueShot(cueBall: Body, shot: CueShot) {
  const forceX = Math.cos(shot.angle) * shot.power * MAX_FORCE;
  const forceY = Math.sin(shot.angle) * shot.power * MAX_FORCE;
  
  // Aplicar força principal
  Body.applyForce(cueBall, cueBall.position, { x: forceX, y: forceY });
  
  // Armazenar spin state na bola (para pós-colisão)
  cueBall.plugin.spin = {
    top: shot.spinTop * shot.power,
    side: shot.spinSide * shot.power
  };
}

// Pós-colisão: ajustar velocidade baseado no spin
Events.on(engine, 'collisionEnd', event => {
  event.pairs.forEach(pair => {
    if (isCueBall(pair.bodyA) || isCueBall(pair.bodyB)) {
      const cue = isCueBall(pair.bodyA) ? pair.bodyA : pair.bodyB;
      const spin = cue.plugin.spin;
      
      if (spin) {
        // Top spin: adiciona velocidade na direção original
        if (spin.top > 0) {
          const boost = spin.top * FOLLOW_FACTOR;
          Body.setVelocity(cue, {
            x: cue.velocity.x * (1 + boost),
            y: cue.velocity.y * (1 + boost)
          });
        }
        // Back spin: inverte parcialmente
        if (spin.top < 0) {
          const drawFactor = spin.top * DRAW_FACTOR;
          Body.setVelocity(cue, {
            x: cue.velocity.x * drawFactor,
            y: cue.velocity.y * drawFactor
          });
        }
        // Side english: componente perpendicular pós-cushion
        // (aplicado no collisionEnd com wall, não ball)
      }
    }
  });
});
```

## 3.3 Componente C — Multiplayer Sync (Turnos)

O Pool Game é **turn-based**, o que simplifica MUITO o sync multiplayer comparado a jogos real-time.

**Fluxo de sync:**
```
1. [Jogador A — NUI] → Aim + Power + Spin → "SHOOT"
2. [NUI] → fetchNui('casino:pool:shoot', { angle, power, spinTop, spinSide })
3. [Lua Client A] → TriggerServerEvent('casino:pool:shoot', shotData)
4. [JS Server]
   a. Validar: é o turno do jogador A?
   b. Validar: shot data within bounds?
   c. Rodar physics simulation server-side (Matter.js)
   d. Calcular resultado: quais bolas moveram para onde, quais encaçapadas
   e. Determinar: foul? turno muda? jogo acabou?
   f. Retornar: { ballPositions: [...], pocketed: [...], foul, nextTurn, gameOver, winner }
5. [Server] → TriggerClientEvent para AMBOS jogadores com resultado
6. [NUI de AMBOS] → Animar o shot baseado nos dados do server
   - Interpolação suave das posições das bolas
   - Som de colisão, encaçapada, cushion
   - Atualizar UI (turno, score, bolas restantes)
```

**Anti-cheat:** Server roda a simulação completa. Client apenas envia input (angle, power, spin) e recebe resultado. Client anima visualmente mas NÃO calcula physics. Impossível hackear posições de bolas.

**Alternativa (client-side physics + server validation):**
- Client calcula physics localmente (responsividade imediata)
- Envia resultado final (ball positions) ao server
- Server valida: re-simula com os mesmos inputs e compara
- Se discrepância > threshold → marca como suspeito
- Prós: mais responsivo (sem latência). Contras: mais complexo, possível desync visual

**Decisão:** Para MVP, **server-side physics** é mais seguro e simples. Client envia input, server calcula, ambos animam o resultado. Latência de ~100-200ms é aceitável para turn-based.

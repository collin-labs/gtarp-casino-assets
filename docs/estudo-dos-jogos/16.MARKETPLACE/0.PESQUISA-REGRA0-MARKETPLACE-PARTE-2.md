## Para cada componente visual: opções avaliadas

### 1. Grid de Itens (Product Listing)

| Opção | Prós | Contras | Veredicto |
|-------|------|---------|:---------:|
| **CSS Grid + DOM cards** | Hover individual, tooltips nativos, filtros instantâneos via React state, skeleton loading, responsive com grid-template-columns repeat(auto-fill) | Muitos cards (100+) podem ter repaint no CEF | ✅ **VENCEDOR** |
| Virtualized List (react-window) | Performance com 1000+ itens, renderiza só o visível | Complexidade de setup, scroll snap difícil, mobile tricky | ⚠️ Usar se >200 itens visíveis |
| Canvas 2D grid | Performance máxima | Sem hover nativo, sem tooltip DOM, sem acessibilidade | ❌ Overkill |

**Decisão:** DOM + CSS Grid para listagem. Se o marketplace passar de 200 itens visíveis simultaneamente, implementar `react-window` para virtualização. Canvas não é necessário — marketplace é interface de informação, não jogo visual.

### 2. Filtros

| Opção | Prós | Contras | Veredicto |
|-------|------|---------|:---------:|
| **Client-side filter (Zustand + useMemo)** | Instantâneo, sem latência, sem chamada server | Precisa carregar todos os itens na memória | ✅ **Para <500 itens** |
| Server-side filter (SQL WHERE) | Escala para milhares, paginação real | Latência de ~100-300ms por query, loading states | ✅ **Para >500 itens** |
| Hybrid (load first 100 client, lazy load rest server) | Melhor UX: filtros iniciais instantâneos, scroll carrega mais | Mais complexo de implementar | ✅ **IDEAL** |

**Decisão:** Modelo HÍBRIDO. Carregar os primeiros 100 itens (mais recentes/relevantes) no client. Filtros básicos (categoria, raridade) funcionam client-side instantâneo. Busca por texto e paginação profunda vão ao server. Referência: Amazon — filtros rápidos client-side, busca profunda server-side.

### 3. Real-time Updates (novos itens, vendas)

| Opção | Prós | Contras | Veredicto |
|-------|------|---------|:---------:|
| **Polling (setInterval 5s)** | Simples, funciona em qualquer NUI, sem deps | Delay de até 5s, queries desnecessárias | ✅ **Simples e funcional** |
| FiveM Events (emitNet broadcast) | Real-time verdadeiro, sem polling | Broadcast para TODOS jogadores = overhead se muitos estão no marketplace | ✅ **Para notificações (vendeu!, novo item!)** |
| WebSocket | Real-time bidirecional | FiveM NUI não tem WebSocket nativo, precisaria de bridge | ❌ Desnecessário |

**Decisão:** COMBO — Polling a cada 5s para refresh da listagem + FiveM Events para notificações instantâneas ("Seu item vendeu!", "Novo item na sua categoria"). Polling é suficiente porque marketplace não precisa de latência de milissegundos como um jogo.

### 4. Search (busca por texto)

| Opção | Prós | Contras | Veredicto |
|-------|------|---------|:---------:|
| **SQL LIKE** | Simples, funciona | Sem fuzzy, sem relevância, lento em tabelas grandes | ⚠️ Básico |
| **SQL FULLTEXT INDEX** | Relevância, match parcial, rápido | Precisa criar FULLTEXT INDEX, só MySQL 5.6+ | ✅ **VENCEDOR** |
| Lua-side filter | Zero SQL | Precisa carregar TUDO na memória do server | ❌ Não escala |

**Decisão:** SQL FULLTEXT INDEX na coluna `name` + `description` da tabela de listings. Query: `SELECT * FROM marketplace_listings WHERE MATCH(name, description) AGAINST(? IN BOOLEAN MODE)`. Autocomplete no client com debounce de 300ms.

### 5. Item Images

| Opção | Prós | Contras | Veredicto |
|-------|------|---------|:---------:|
| **PNG estáticos (assets do projeto)** | Já existem para Cases/Bicho/etc, fast load | Precisa gerar para cada item novo | ✅ **Primary** |
| SVG procedural (cor + shape por raridade) | Gera automaticamente, sem assets extras | Visual menos premium | ⚠️ Fallback |
| Sem imagem (text-only cards) | Zero assets | UX terrível, "básico" — o problema que tu teve | ❌ |

**Decisão:** PNG estáticos como primary (reutilizar assets dos 22 jogos — Cases já tem 34 PNGs, Bicho tem 25, etc.). SVG procedural como fallback para itens sem imagem dedicada (badge colorido com ícone por categoria).

async function purchaseItem(buyerId, listingId) {
  // Buscar listing
  const listing = await oxmysql.single(
    'SELECT * FROM marketplace_listings WHERE id = ? AND status = "active"',
    [listingId]
  );
  if (!listing) return { success: false, error: 'Item não disponível' };
  if (listing.seller_id === buyerId) return { success: false, error: 'Não pode comprar seu próprio item' };
  
  // Verificar saldo do comprador
  const balance = await oxmysql.scalar(
    'SELECT bank FROM users WHERE identifier = ?', [buyerId]
  );
  if (balance < listing.price) return { success: false, error: 'Saldo insuficiente' };
  
  // Calcular taxa
  const feePercent = await getMarketplaceFee(buyerId); // VIP tier → fee menor
  const fee = Math.floor(listing.price * feePercent / 100);
  const sellerReceives = listing.price - fee;
  
  // TRANSAÇÃO ATÔMICA (5 operações — tudo ou nada)
  await oxmysql.transaction([
    // 1. Debitar comprador
    { query: 'UPDATE users SET bank = bank - ? WHERE identifier = ?', params: [listing.price, buyerId] },
    // 2. Creditar vendedor (menos taxa)
    { query: 'UPDATE users SET bank = bank + ? WHERE identifier = ?', params: [sellerReceives, listing.seller_id] },
    // 3. Transferir item para comprador
    { query: `INSERT INTO player_inventory (player_id, item_id, item_type, data, quantity)
              VALUES (?, UUID(), ?, ?, 1)
              ON DUPLICATE KEY UPDATE quantity = quantity + 1`,
      params: [buyerId, listing.item_type, listing.item_data] },
    // 4. Marcar listing como vendido
    { query: `UPDATE marketplace_listings SET status = 'sold', buyer_id = ?, sold_price = ?, fee = ?,
              seller_receives = ?, sold_at = NOW() WHERE id = ?`,
      params: [buyerId, listing.price, fee, sellerReceives, listingId] },
    // 5. Registrar transação
    { query: `INSERT INTO marketplace_transactions (listing_id, buyer_id, seller_id, price, fee, created_at)
              VALUES (?, ?, ?, ?, ?, NOW())`,
      params: [listingId, buyerId, listing.seller_id, listing.price, fee] }
  ]);
  
  // Notificar vendedor
  emitNet('casino:marketplace:item_sold', parseInt(listing.seller_id), {
    itemType: listing.item_type, price: listing.price, fee, received: sellerReceives
  });
  
  // Audit
  await auditLog('PURCHASE', buyerId, null, {
    listingId, seller: listing.seller_id, price: listing.price, fee
  });

async function getPriceSuggestion(itemType) {
  // Últimos 10 vendidos
  const recent = await oxmysql.query(
    `SELECT sold_price FROM marketplace_listings 
     WHERE item_type = ? AND status = 'sold' 
     ORDER BY sold_at DESC LIMIT 10`,
    [itemType]
  );
  
  if (recent.length === 0) {
    // Nenhuma venda anterior — usar preço base do item
    const basePrice = await oxmysql.scalar(
      'SELECT base_price FROM marketplace_item_types WHERE id = ?', [itemType]
    );
    return { average: basePrice, min: basePrice * 0.8, max: basePrice * 1.2, sales: 0 };
  }
  
  const prices = recent.map(r => r.sold_price);
  const avg = Math.round(prices.reduce((a, b) => a + b, 0) / prices.length);
  const min = Math.min(...prices);
  const max = Math.max(...prices);
  
  return {
    average: avg,
    min,
    max,
    sales: recent.length,
    label: 'Baseado nas últimas ' + recent.length + ' vendas',
  };
}

// CLIENT — filtro instantâneo sobre dados já carregados
function filterListings(listings, filters) {
  return listings.filter(item => {
    if (filters.category && item.category !== filters.category) return false;
    if (filters.rarity && item.rarity !== filters.rarity) return false;
    if (filters.minPrice && item.price < filters.minPrice) return false;
    if (filters.maxPrice && item.price > filters.maxPrice) return false;
    if (filters.game && item.sourceGame !== filters.game) return false;
    return true;
  }).sort((a, b) => {
    switch (filters.sortBy) {
      case 'price_asc': return a.price - b.price;
      case 'price_desc': return b.price - a.price;
      case 'newest': return new Date(b.createdAt) - new Date(a.createdAt);
      case 'popular': return b.views - a.views;
      default: return 0;
    }
  });
}

// SERVER — busca profunda quando client-side não é suficiente
async function searchListings(query, filters, page, pageSize) {
  let sql = `SELECT l.*, it.name, it.image_url, it.rarity, it.category,
             u.name AS seller_name
             FROM marketplace_listings l
             JOIN marketplace_item_types it ON l.item_type = it.id
             JOIN users u ON l.seller_id = u.identifier
             WHERE l.status = 'active'`;
  const params = [];
  
  if (query) {
    sql += ` AND MATCH(it.name, it.description) AGAINST(? IN BOOLEAN MODE)`;
    params.push(query);
  }
  if (filters.category) { sql += ` AND it.category = ?`; params.push(filters.category); }
  if (filters.rarity) { sql += ` AND it.rarity = ?`; params.push(filters.rarity); }
  if (filters.minPrice) { sql += ` AND l.price >= ?`; params.push(filters.minPrice); }
  if (filters.maxPrice) { sql += ` AND l.price <= ?`; params.push(filters.maxPrice); }
  if (filters.game) { sql += ` AND it.source_game = ?`; params.push(filters.game); }
  
  sql += ` ORDER BY ${filters.sortBy || 'l.created_at DESC'}`;
  sql += ` LIMIT ? OFFSET ?`;
  params.push(pageSize, (page - 1) * pageSize);
  
  return await oxmysql.query(sql, params);
}

## Formato escolhido: P2P Marketplace + Casino Reward Shop + Price Intelligence

### Por que ESTE formato:

1. **P2P Marketplace** cria economia entre jogadores — eles têm MOTIVO para voltar (verificar se vendeu, procurar itens novos, fazer negócio). Isso é retenção orgânica.

2. **Casino Reward Shop** cria money sink controlado — o admin define preços de itens exclusivos. Combate inflação sem parecer "punição" (jogador QUER comprar, não é forçado).

3. **Price Intelligence** (sugestão, histórico, badges PECHINCHA/ACIMA DA MÉDIA) resolve o problema de "informação vaga" que tu teve. Jogador sabe se está pagando justo.

### Formatos DESCARTADOS:

| Formato | Por que NÃO |
|---------|-------------|
| **Loja fixa NPC** (preços fixos, itens predefinidos) | Sem interação P2P, sem economia dinâmica, sem razão para voltar depois de comprar tudo. É o "básico" que tu já fez e não gostou |
| **Auction House puro** (só leilão) | Muito complexo para v1, assusta jogadores casuais que só querem comprar rápido. Pode ser v2 |
| **Trade-only** (sem moeda, só troca) | Sem liquidez — jogador precisa encontrar alguém que tenha O QUE ele quer E queira O QUE ele tem. Funciona mal com poucos jogadores |
| **NFT / blockchain** | Violaria PLA do FiveM ("no crypto"), complexidade desnecessária, audiência GTA RP não quer blockchain |

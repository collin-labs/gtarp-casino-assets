## Problema 1: Provably Fair Deck Shuffle para 312 Cartas (6 Decks)

**Sintoma:** O padrao provably fair de slots (hash de 1 posicao) NAO funciona para Blackjack. No BJ, o deck inteiro (312 cartas em 6 decks) precisa ser shuffled de forma verificavel. O jogador precisa poder reconstruir o deck completo a partir dos seeds para confirmar que as cartas recebidas vieram das posicoes corretas.

**Causa tecnica:** Provably fair usa HMAC-SHA256 para gerar bytes pseudo-aleatorios. Para slots, 1 hash gera 1 numero (posicao do resultado). Para BJ com 312 cartas, precisamos de 312 indices unicos via Fisher-Yates, e cada indice consome 4 bytes do hash. SHA-256 gera 32 bytes (8 indices por hash). Entao precisamos de ceil(312/8) = 39 hashes encadeados para shufflar o deck inteiro.

**Solucao: Cursor Pattern com hash encadeado**

```javascript
// blackjack-provably-fair.js — Provably Fair Deck Shuffle

const crypto = require('crypto');

function generateDeckHash(serverSeed, clientSeed, nonce) {
  // Gerar bytes suficientes para Fisher-Yates de 312 cartas
  // Cada carta precisa de 4 bytes para gerar indice unbiased
  // 312 cartas x 4 bytes = 1.248 bytes necessarios
  // SHA-256 = 32 bytes por hash -> ceil(1248/32) = 39 hashes
  const bytesNeeded = 312 * 4;
  let bytes = Buffer.alloc(0);
  let cursor = 0;

  while (bytes.length < bytesNeeded) {
    const hmac = crypto.createHmac('sha256', serverSeed);
    hmac.update(`${clientSeed}:${nonce}:${cursor}`);
    const chunk = hmac.digest();
    bytes = Buffer.concat([bytes, chunk]);
    cursor++;
  }

  return bytes;
}

function fisherYatesShuffle(deck, bytes) {
  // Fisher-Yates usando bytes HMAC como fonte de aleatoriedade
  const shuffled = [...deck];
  let byteOffset = 0;

  for (let i = shuffled.length - 1; i > 0; i--) {
    // Ler 4 bytes como uint32, modulo para range [0, i]
    const raw = bytes.readUInt32BE(byteOffset);
    const j = raw % (i + 1);
    byteOffset += 4;

    // Swap
    const temp = shuffled[i];
    shuffled[i] = shuffled[j];
    shuffled[j] = temp;
  }

  return shuffled;
}

function createShoe(numDecks = 6) {
  const suits = ['H', 'D', 'C', 'S'];
  const ranks = ['A', '2', '3', '4', '5', '6', '7', '8', '9', 'T', 'J', 'Q', 'K'];
  const shoe = [];

  for (let d = 0; d < numDecks; d++) {
    for (const suit of suits) {
      for (const rank of ranks) {
        shoe.push({ rank, suit, deckIndex: d });
      }
    }
  }

  return shoe; // 312 cartas
}

function shuffleShoe(serverSeed, clientSeed, nonce, numDecks = 6) {
  const shoe = createShoe(numDecks);
  const bytes = generateDeckHash(serverSeed, clientSeed, nonce);
  return fisherYatesShuffle(shoe, bytes);
}

// Hash publicado ANTES do deal (commit)
function getServerSeedHash(serverSeed) {
  return crypto.createHash('sha256').update(serverSeed).digest('hex');
}

module.exports = { shuffleShoe, getServerSeedHash, createShoe, fisherYatesShuffle, generateDeckHash };
```

**Impacto:** ~65 linhas no modulo provably-fair. Gera 39 hashes HMAC por shoe (< 1ms total). Verificavel: qualquer jogador com os 3 valores (serverSeed, clientSeed, nonce) reconstroi o deck e confirma cada carta.

## Problema 2: Split State Management — Multiplas Maos Simultaneas

**Sintoma:** Quando o jogador faz Split, passa a ter 2 maos independentes jogadas sequencialmente. Se fizer Split novamente (re-split), pode ter ate 4 maos. O state management precisa rastrear qual mao esta ativa, quais acoes sao permitidas por mao, e os resultados individuais.

**Causa tecnica:** A maioria das implementacoes de BJ usa um unico objeto `hand` para o player. Com Split, o state precisa ser um array de hands, cada uma com seu proprio `cards`, `bet`, `isActive`, `isCompleted`, `result`. O dealer joga contra TODAS as maos. O payout e individual por mao.

**Solucao: Array de Hand objects com activeHandIndex**

```javascript
// Estrutura de state para Split
const blackjackState = {
  phase: 'PLAYER_TURN', // BETTING | DEALING | PLAYER_TURN | DEALER_TURN | RESULT
  shoe: [],             // deck shuffled (server-side)
  shoePosition: 0,      // cursor no shoe
  dealer: {
    cards: [],
    total: 0,
    isSoft: false
  },
  player: {
    hands: [
      {
        id: 0,
        cards: [{ rank: '8', suit: 'H' }, { rank: '8', suit: 'S' }],
        bet: 100,
        total: 16,
        isSoft: false,
        isCompleted: false,
        isDoubled: false,
        isSurrendered: false,
        result: null // WIN | LOSE | PUSH | BLACKJACK | BUST | SURRENDER
      }
    ],
    activeHandIndex: 0
  },
  sideBets: {
    perfectPairs: { bet: 0, result: null, payout: 0 },
    twentyOnePlus3: { bet: 0, result: null, payout: 0 }
  },
  insurance: { offered: false, taken: false, bet: 0, result: null }
};

// Logica de Split
function handleSplit(state) {
  const activeHand = state.player.hands[state.player.activeHandIndex];

  // Validacoes
  if (activeHand.cards.length !== 2) return { error: 'split_only_initial' };
  if (activeHand.cards[0].rank !== activeHand.cards[1].rank) return { error: 'split_requires_pair' };
  if (state.player.hands.length >= 4) return { error: 'max_splits_reached' };

  // Separar cartas
  const card1 = activeHand.cards[0];
  const card2 = activeHand.cards[1];

  // Mao atual fica com card1 + nova carta do shoe
  activeHand.cards = [card1, state.shoe[state.shoePosition++]];
  activeHand.total = calculateHandValue(activeHand.cards).total;
  activeHand.isSoft = calculateHandValue(activeHand.cards).isSoft;

  // Nova mao com card2 + nova carta do shoe
  const newHand = {
    id: state.player.hands.length,
    cards: [card2, state.shoe[state.shoePosition++]],
    bet: activeHand.bet, // mesma aposta da mao original
    total: 0,
    isSoft: false,
    isCompleted: false,
    isDoubled: false,
    isSurrendered: false,
    result: null
  };
  newHand.total = calculateHandValue(newHand.cards).total;
  newHand.isSoft = calculateHandValue(newHand.cards).isSoft;

  state.player.hands.push(newHand);

  // Se split de Ases: 1 carta cada, auto-stand ambas
  if (card1.rank === 'A') {
    activeHand.isCompleted = true;
    newHand.isCompleted = true;
    // Avancar para dealer turn se todas completas
    if (state.player.hands.every(h => h.isCompleted)) {
      state.phase = 'DEALER_TURN';
    }
  }

  return { success: true, hands: state.player.hands };
}
```

**Impacto:** ~80 linhas de logica no engine. Suporta ate 4 maos (config `maxSplits`). Ases splitados recebem 1 carta e auto-stand (regra padrao). Client renderiza maos lado a lado com indicador visual na mao ativa.

## Problema 3: Insurance Interrupt — FSM Precisa Pausar

**Sintoma:** Quando o dealer mostra um As, o jogo precisa pausar para oferecer Insurance a TODOS os jogadores (no caso de multiplayer futuro) ou ao jogador atual. Isso quebra o fluxo linear DEALING -> PLAYER_TURN.

**Causa tecnica:** A FSM padrao (BETTING -> DEALING -> PLAYER_TURN -> DEALER_TURN -> RESULT) nao tem estado para Insurance. O Stackademic (Fev/2026) documenta isso como "interrupt state" — um estado intermediario que pausa o fluxo ate resolucao.

**Solucao: Estado INSURANCE_PROMPT entre DEALING e PLAYER_TURN**

```javascript
// FSM com estado de Insurance
const PHASES = {
  IDLE: 'IDLE',
  BETTING: 'BETTING',
  DEALING: 'DEALING',
  INSURANCE_PROMPT: 'INSURANCE_PROMPT', // interrupcao
  PLAYER_TURN: 'PLAYER_TURN',
  DEALER_TURN: 'DEALER_TURN',
  RESULT: 'RESULT'
};

function transitionAfterDeal(state) {
  const dealerUpcard = state.dealer.cards[0];

  // Checar Blackjack Natural do jogador
  const playerHand = state.player.hands[0];
  if (isBlackjack(playerHand.cards)) {
    // Se dealer tambem pode ter BJ (upcard A ou 10-value)
    if (dealerUpcard.rank === 'A') {
      state.phase = PHASES.INSURANCE_PROMPT;
      state.insurance.offered = true;
      // Timer de 10 segundos server-side
      state.insuranceTimer = setTimeout(() => {
        // Auto-decline se timeout
        state.insurance.taken = false;
        resolveInsuranceAndContinue(state);
      }, 10000);
      return;
    }
    // Dealer nao pode ter BJ: pagar 3:2 direto
    playerHand.result = 'BLACKJACK';
    state.phase = PHASES.RESULT;
    return;
  }

  // Dealer mostra As: oferecer insurance
  if (dealerUpcard.rank === 'A' && state.config.insuranceEnabled) {
    state.phase = PHASES.INSURANCE_PROMPT;
    state.insurance.offered = true;
    state.insuranceTimer = setTimeout(() => {
      state.insurance.taken = false;
      resolveInsuranceAndContinue(state);
    }, 10000);
    return;
  }

  // Fluxo normal
  state.phase = PHASES.PLAYER_TURN;
}

function resolveInsuranceAndContinue(state) {
  clearTimeout(state.insuranceTimer);

  // Verificar se dealer tem Blackjack
  const dealerHasBlackjack = isBlackjack(state.dealer.cards);

  if (dealerHasBlackjack) {
    // Insurance paga 2:1 se tomado
    if (state.insurance.taken) {
      state.insurance.result = 'WIN';
      state.insurance.payout = state.insurance.bet * 2;
    }
    // Mao principal perde (a menos que player tambem tenha BJ = push)
    const playerHand = state.player.hands[0];
    if (isBlackjack(playerHand.cards)) {
      playerHand.result = 'PUSH';
    } else {
      playerHand.result = 'LOSE';
    }
    state.phase = PHASES.RESULT;
    return;
  }

  // Dealer nao tem BJ: insurance perde, jogo continua normal
  if (state.insurance.taken) {
    state.insurance.result = 'LOSE';
  }
  state.phase = PHASES.PLAYER_TURN;
}
```

**Impacto:** ~70 linhas. 1 estado extra na FSM. Timer server-side de 10s com auto-decline. Client mostra modal "Seguro?" com countdown visual.

## Problema 4: Card Counting Prevention — Deck Penetration

**Sintoma:** Em BJ online, jogadores sofisticados podem usar card counting (Hi-Lo system) para ganhar vantagem. Com 6 decks e shoe visivel, o true count pode dar edge ao jogador nas ultimas cartas.

**Causa tecnica:** Card counting funciona melhor com deck penetration alto (muitas cartas distribuidas antes de reshufflar). Se o shoe tem 312 cartas e o jogador ve 250 antes do reshuffle, o count se torna muito preciso.

**Solucao: Cut card a 75% + reshuffle automatico**

```javascript
// Deck penetration control no handler
const PENETRATION = 0.75; // 75% do shoe = 234 cartas
const CUT_CARD_POSITION = Math.floor(312 * PENETRATION);

function shouldReshuffle(state) {
  return state.shoePosition >= CUT_CARD_POSITION;
}

// No inicio de cada mao (antes do deal)
function preDealCheck(state) {
  if (shouldReshuffle(state)) {
    // Gerar novo server seed
    state.serverSeed = crypto.randomBytes(32).toString('hex');
    state.serverSeedHash = getServerSeedHash(state.serverSeed);
    state.nonce = 0;

    // Reshufflar shoe completo
    state.shoe = shuffleShoe(
      state.serverSeed,
      state.clientSeed,
      state.nonce,
      state.config.decks
    );
    state.shoePosition = 0;

    // Notificar client do novo hash (novo shoe)
    return {
      reshuffled: true,
      newSeedHash: state.serverSeedHash,
      previousServerSeed: state.previousServerSeed // revelar anterior
    };
  }
  return { reshuffled: false };
}
```

**Impacto:** ~25 linhas. Cut card a 75% e padrao industria (Stackademic confirma). Reshuffle automatico quando atingido. Novo server seed hash publicado ao client. Server seed anterior revelado para verificacao retroativa.

## 7.2 Seguranca / Anti-Cheat (8 Itens)

| # | Aspecto | Implementacao Concreta |
|---|---------|----------------------|
| 1 | **Provably Fair deck** | HMAC-SHA256 cursor pattern com 39 hashes encadeados para 312 cartas. Server seed hash publicado ANTES do bet. Server seed revelado ao encerrar sessao. Verificavel por qualquer jogador. |
| 2 | **Server-authoritative** | TODA logica de jogo roda no server. Client envia apenas acoes (hit/stand/double/split/surrender). Server calcula resultados, valida acoes permitidas, retorna cartas. Client APENAS renderiza. |
| 3 | **Input validation (6 checks)** | (1) E a vez do jogador? (2) Acao e valida para o estado atual? (3) Saldo >= bet para double/split? (4) Pode splittar (par + max splits)? (5) Pode surrender (2 cartas iniciais)? (6) Session ativa? |
| 4 | **Rate limit** | Server so aceita 1 acao por vez. Flag `actionPending = true` durante processamento. Segunda acao = rejeitar. Cooldown entre maos: config `cooldownMs` (default 3000ms). |
| 5 | **Deck penetration** | Cut card a 75% (234 de 312 cartas). Reshuffle automatico com novo server seed. Anti-counting: impossivel contar com precision quando 25% do shoe nunca aparece. |
| 6 | **Daily limit** | Config `maxHandsPerDay` (default 200). Verificado no deal. INSERT com COUNT por dia no query. |
| 7 | **Side bet limits** | Config `sideBetMaxBet` (default 500 GCoins). Validado server-side. Admin pode desabilitar side bets globalmente. |
| 8 | **Audit log completo** | TODA acao em `casino_bj_audit`: HAND_START (bet, side bets), HAND_END (result, payout, cards), ACTION (type, hand_index), INSURANCE (taken/declined, result), SIDEBET (type, sub_result, payout), RESHUFFLE (nonce reset), CONFIG_CHANGE (key, old, new), SESSION_START, SESSION_CLOSE (nonce final), ERROR. JSON details flexivel. Append-only. |

## 7.3 Perguntas Pre-Implementacao para o BC

| # | Pergunta | Default Sugerido | Impacto |
|---|----------|:----------------:|---------|
| 1 | Quantos decks? | 6 | House edge: 6 decks = 0.46%, 8 decks = 0.48% |
| 2 | Blackjack paga quanto? | 3:2 (150%) | 6:5 aumenta house edge em 1.4% (nao recomendado) |
| 3 | Dealer stands ou hits Soft 17? | Stands (S17) | H17 aumenta house edge em 0.22% |
| 4 | Double After Split permitido? | Sim | Reduz house edge em 0.14% |
| 5 | Surrender tipo? | Late | Early = muito favoravel ao player |
| 6 | Side bets habilitadas? | Sim (Perfect Pairs + 21+3) | Revenue extra, house edge ~5-7% nos side bets |
| 7 | Aposta minima/maxima? | 10 / 5.000 GCoins | Faixa economica do servidor RP |
| 8 | Limite diario de maos? | 200 | Protecao contra grind excessivo |

# 7. ECONOMIA DO JOGO

## 7.1 Modelo Economico

O modelo economico de case opening funciona por probabilidade ponderada. Diferente de jogos com RTP fixo (slots, roleta), o RTP de cases depende do valor de mercado dos itens internos, que o admin do servidor controla. O house edge efetivo varia por caixa.

Conceito base: o preco da caixa e fixo, mas o valor esperado (EV) dos itens e inferior ao preco. A diferenca e o house edge. Em plataformas reais como CS2, o ROI medio e -70% a -90% (fonte: blix.gg). No Blackout Casino, o admin configura as probabilidades para manter um house edge mais moderado (20-40%), adequado ao contexto de GTA RP onde a economia e virtual e a retencao importa mais que extracao maxima.

# 8. PROVABLY FAIR — IMPLEMENTACAO COMPLETA

## 8.1 Algoritmo

HMAC-SHA256 conforme padrao Stake.com/Boxed.gg, adaptado para case opening com mapeamento de ranges por item.

## 8.2 Fluxo Completo (10 Passos)

| Passo | Ator | Acao | Detalhe |
|:-----:|------|------|---------|
| 1 | Server | Gera server seed | `crypto.randomBytes(32).toString('hex')` — 64 caracteres hex |
| 2 | Server | Gera hash do server seed | `crypto.createHash('sha256').update(serverSeed).digest('hex')` |
| 3 | Server | Envia hash ao jogador | Hash visivel no modal PF ANTES da abertura. Seed real permanece secreto |
| 4 | Client | Gera ou customiza client seed | Browser gera seed aleatorio. Jogador pode alterar manualmente |
| 5 | Client | Envia client seed ao server | Via fetchNui('casino:case:open', { caseId, clientSeed }) |
| 6 | Server | Calcula HMAC | `createHmac('sha256', serverSeed).update(clientSeed + ':' + nonce + ':0').digest('hex')` |
| 7 | Server | Converte hash em numero 0-999.999 | Primeiros 5 caracteres hex → parseInt(hex, 16) % 1.000.000 |
| 8 | Server | Mapeia numero para item via ranges | Cada item tem um range proporcional a sua probabilidade. Ex: Common 0-399.999, Legendary 995.000-999.999 |
| 9 | Server | Incrementa nonce | nonce++ para proxima aposta com mesmo par de seeds |
| 10 | Client | Verifica resultado | Apos revelacao, jogador acessa modal PF com serverSeed (revelado), clientSeed, nonce. Pode recalcular HMAC e confirmar que resultado e correto |

## 8.3 Variaveis

| Variavel | Quem gera | Visivel quando | Proposito |
|----------|-----------|----------------|-----------|
| Server Seed | Server (crypto.randomBytes) | Apos rotacao (a cada 24h ou manual) | Entropia server-side, garante que jogador nao preveja resultado |
| Server Seed Hash | Server (SHA-256 do seed) | Antes da aposta | Compromisso: prova que o seed existia antes do jogador apostar |
| Client Seed | Client (browser Math.random ou manual) | Sempre | Entropia client-side, garante que server nao manipule resultado |
| Nonce | Server (auto-increment) | Sempre | Unicidade: mesmo par de seeds gera resultados diferentes a cada aposta |
| Cursor | Server (0, incrementa se >8 outcomes) | Sempre | Para jogos que precisam de >8 numeros aleatorios (cases = 1, nao incrementa) |

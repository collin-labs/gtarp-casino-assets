# PARTE 4 — PROBLEMAS E LIMITACOES

## Problema 1: Inconsistencia Visual entre Componentes

**Sintoma:** CaseCard.tsx (372L) tem visual premium (bordas douradas, feixo verde, shimmer sweep, hover -6px + scale 1.05) mas os demais componentes (CasePreview 204L, CaseOpening 181L, CaseResult 226L, CaseBattleLobby 222L, CaseBattleGame 253L) usam styling basico sem efeitos premium. Quando o jogador sai do catalogo e entra numa tela interna, o visual "cai" — parece outro sistema.

**Causa:** CaseCard foi criado junto com o painel principal (BentoGrid) e herdou o Design System. Os demais componentes foram criados depois em conversas diferentes, sem referenciar o DS. O resultado e um gradiente de qualidade: catalogo excelente, telas internas mediocres.

**Codigo real (CasePreview.tsx, linha 15-25 aprox.):**
```tsx
// PROBLEMA: styling inline basico sem bordas douradas, sem feixo, sem glow
<div style={{
  background: "rgba(15,15,15,0.95)",
  borderRadius: "12px",
  padding: "20px",
  // FALTA: border 2px solid rgba(212,168,67,0.3)
  // FALTA: outline 1px solid rgba(191,149,63,0.15)
  // FALTA: boxShadow multicamada (DS P1 §4)
  // FALTA: feixo verde base (DS P2 §8)
}}>
```

**Solucao:** Refatorar TODOS os componentes internos aplicando o DS sistematicamente. Cada container recebe: borda dourada 2px + outline + boxShadow 5 camadas + feixo verde base. Cada elemento interativo recebe: hover com intensificacao + shimmer sweep. Cada titulo recebe: Cinzel uppercase + letter-spacing + text-shadow glow.

## Problema 2: Glow de Raridade Nao Escala por Tier

**Sintoma:** No CaseResult.tsx, todos os itens recebem a mesma borda com cor da raridade, mas sem diferenciacao de intensidade. Um item Common e exibido com a mesma "cerimonia" visual que um Legendary — apenas a cor muda. Nao ha glow pulsante, particulas, overlay, ou feedback sonoro diferenciado.

**Causa:** O componente trata raridade como uma unica variavel de cor, nao como um sistema de escalada visual. A pesquisa identificou que a gradacao visual e o principal pico de dopamina do jogo — Common silencioso, Legendary espetaculo completo.

**Codigo real (CaseResult.tsx, renderizacao do item):**
```tsx
// PROBLEMA: mesma borda para todos os tiers, sem glow, sem particulas
<div style={{
  border: `2px solid ${item.rarity_color}`,
  borderRadius: "14px",
  // FALTA: boxShadow com glow pulsante escalado por tier
  // FALTA: @keyframes glow diferente por tier
  // FALTA: particulas para Rare+ (sparkles, confetti)
  // FALTA: BigWin overlay para Epic/Legendary
  // FALTA: som diferenciado (reveal_common vs reveal_legendary)
}}>
```

**Solucao:** Criar componente RarityGlow.tsx com 5 niveis de intensidade visual (descrito na pesquisa secao 6.5). Common = borda sutil, Legendary = glow 40px pulsante + confetti + BigWin overlay + fanfarra 2.5s. Implementar via switch no tier com @keyframes e Framer Motion AnimatePresence.

## Problema 3: Provably Fair sem UI de Verificacao

**Sintoma:** O handler server-side (cases.js) implementa Provably Fair com SHA-256 (hash, seed_nonce registrados no banco), mas o frontend NAO tem nenhum modal ou tela para o jogador verificar a fairness. O hash e gerado e armazenado, mas nunca exposto ao jogador.

**Causa:** O desenvolvimento focou na funcionalidade de abertura e deixou a transparencia para depois. O backend esta pronto mas o frontend nao consumiu a informacao.

**Codigo real (cases.js, Handler 3, linhas 68-90 aprox.):**
```javascript
// Server gera hash e registra, mas NUI nunca exibe
const hmac = crypto.createHmac('sha256', SERVER_SEED);

  `INSERT INTO casino_cases_openings (user_id, case_id, item_id, price_paid, item_value, hash, seed_nonce, is_free)

  [userId, caseId, item.id, price, item.value, hash, nonce, isFree ? 1 : 0]

// FALTA: endpoint para expor server_seed_hash ANTES da abertura
// FALTA: endpoint para revelar server_seed APOS sessao
// FALTA: NUI com modal para jogador verificar
```

**Solucao:** Criar ProvablyFairModal.tsx com: exibicao do server_seed_hash antes da abertura, client_seed editavel, nonce visivel, botao "Verificar" que recalcula o hash client-side. Adicionar 2 endpoints ao handler: `casino:cases:get_seed_hash` (pre-abertura) e `casino:cases:reveal_seed` (pos-sessao). Seguir padrao Stake.com documentado na pesquisa (secao 8).

## Problema 4: Fast Open sem Animacao

**Sintoma:** O toggle Fast Open no CasePreview pula a strip e mostra resultado diretamente, sem nenhuma transicao visual. O item simplesmente aparece — sem card flip, sem flash de cor, sem feedback.

**Causa:** Implementado como atalho funcional (if fastMode skip strip), nao como experiencia alternativa.

**Solucao:** Criar FastOpenCard.tsx com animacao Framer Motion rotateY 0→180deg (0.6s). Frente do card mostra imagem da caixa (estado SEMI-ABERTA). Verso mostra item revelado com glow da raridade. Transicao suave com backface-visibility hidden.

## Problema 5: Battle sem Sincronizacao Real-Time

**Sintoma:** CaseBattleGame.tsx usa dados mock e nao tem sincronizacao real entre jogadores. Cada jogador ve sua propria animacao independente.

**Causa:** FiveM nao tem WebSocket nativo. A sincronizacao precisa ser via TriggerClientEvent server→client.

**Limitacao FiveM/NUI:** O NUI Chromium nao tem acesso direto ao WebSocket externo. A comunicacao deve ser: Server → TriggerClientEvent → Client Lua → SendNUIMessage → React NUI. Latencia tipica: 50-150ms (aceitavel para animacao visual, nao para gameplay em tempo real).

**Solucao:** Server processa TODAS as aberturas do battle de uma vez, gera array de resultados por jogador, envia via TriggerClientEvent para cada jogador conectado. NUI recebe e anima as strips simultaneamente com delay sincronizado. Server e a unica fonte de verdade — NUI nao calcula resultados.

# PARTE 5 — SQL (Complemento ao Schema Existente)

O schema existente (126L, 7 tabelas) esta funcional. Para atender as decisoes da pesquisa, precisa de complementos:

## 5.1 Tabela Nova: Provably Fair Seeds

```sql
-- Tabela 8: Seeds do Provably Fair por sessao
CREATE TABLE IF NOT EXISTS casino_cases_seeds (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  server_seed VARCHAR(128) NOT NULL,
  server_seed_hash VARCHAR(64) NOT NULL,
  client_seed VARCHAR(64) NOT NULL DEFAULT '',
  nonce BIGINT NOT NULL DEFAULT 0,
  is_revealed TINYINT(1) DEFAULT 0,
  revealed_at TIMESTAMP DEFAULT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_user_active (user_id, is_revealed),
  INDEX idx_hash (server_seed_hash)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 5.2 Tabela Nova: Daily Claims Tracking

```sql
-- Tabela 9: Controle de claims diarios (previne abuse)
CREATE TABLE IF NOT EXISTS casino_cases_daily_claims (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  license VARCHAR(100) NOT NULL,
  claim_date DATE NOT NULL,
  opening_id INT DEFAULT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_license_date (license, claim_date),
  INDEX idx_user (user_id),
  CONSTRAINT fk_daily_opening FOREIGN KEY (opening_id) REFERENCES casino_cases_openings(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

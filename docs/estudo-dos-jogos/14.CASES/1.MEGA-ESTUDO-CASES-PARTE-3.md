## RevealStrip.tsx — 223L — Analise da Animacao

A strip e o componente core da experiencia de abertura. Analise tecnica:

**Geracao de itens:** Array de 40 itens gerado randomicamente a partir dos itens da caixa (respeitando probabilidades visuais — mais Common aparecem). O item vencedor (pre-determinado server-side) e posicionado no indice 30 (para garantir que a strip tenha espaço de desaceleracao).

**Calculo do target:** `targetX = -(WINNER_INDEX * ITEM_WIDTH) + (containerWidth / 2) - (ITEM_WIDTH / 2)`. Isso centraliza o item vencedor no meio da viewport da strip.

**Easing:** `[0.15, 0, 0.05, 1]` — cubic-bezier com inicio rapido (70% da distancia nos primeiros 30% do tempo) e desaceleracao longa (ultimos 30% da distancia nos 70% restantes do tempo). O efeito visual e identico ao CS2: rapido no inicio, lento e tenso no final.

**Pointer central:** Div fixo no meio horizontal da strip com `position: absolute`, `width: 2px`, `height: 100%`, `background: "#FF4444"` (vermelho), `boxShadow: "0 0 8px rgba(255,68,68,0.5)"`. Marca onde a strip vai parar.

**onAnimationComplete:** Callback que seta `revealed: true`, toca o som do tier (reveal_common a reveal_legendary), e chama `onResult(winnerItem)`. Delay de 200ms apos o complete para dar tempo visual ao jogador de perceber onde parou.

# PARTE 4B — PROBLEMAS ADICIONAIS

## Problema 6: SERVER_SEED Global (Seguranca)

**Sintoma:** O handler cases.js usa `const SERVER_SEED = crypto.randomBytes(32).toString('hex')` como variavel global no topo do modulo. Isso significa que TODOS os jogadores usam o MESMO server_seed ate o server reiniciar.

**Causa:** Implementacao simplificada do Provably Fair. O padrao Stake.com usa um server_seed POR SESSAO (por jogador), nao global.

**Codigo real (cases.js, linha 9):**
```javascript
// PROBLEMA: seed global compartilhado entre todos jogadores
const SERVER_SEED = crypto.randomBytes(32).toString('hex');
// Se um jogador revelar este seed, TODOS os resultados futuros sao previsiveis
```

**Solucao:** Usar tabela casino_cases_seeds com seed individual por jogador. No primeiro acesso, gerar seed unico e registrar hash. Ao solicitar revelacao, marcar is_revealed=true e gerar novo seed. Endpoint `get_seed_hash` retorna hash SHA-256 do seed ativo. Endpoint `reveal_seed` retorna seed original apos marcar como revelado.

## Problema 7: Mock Data Hardcoded (Producao)

**Sintoma:** CasesGame.tsx contem ~150 linhas de dados mock hardcoded (CASE_ITEMS object com todos itens de todas caixas). Em producao FiveM, esses dados devem vir do server via fetchNui.

**Causa:** Desenvolvimento em modo preview (npm run dev) sem servidor FiveM.

**Solucao:** Manter mock como fallback para desenvolvimento. Criar funcao `getCaseItems(caseId)` que tenta fetchNui primeiro, cai pro mock se falhar. Pattern: `const items = await fetchNui('casino:cases:preview', { caseId }).catch(() => MOCK_ITEMS[caseId])`.

## Problema 1 — pokersolver precisa ser instalado como dependencia nova

**Impacto:** O projeto atualmente NAO tem nenhum hand evaluator. pokersolver (npm) e a unica dependencia NOVA necessaria.

**Detalhes tecnicos:**
- pokersolver v2.1.1 (npm). Ultima publicacao: ~7 meses atras. 6 projetos dependentes.
- Tamanho: ~102KB com lookup tables. NAO e uma lib pesada.
- Compatibilidade: funciona no browser (NUI/CEF) E no Node.js (server handler).
- Notacao: "Ad" = Ace of diamonds, "Th" = Ten of hearts.
- Suporta: Texas Hold'em (7 cartas), Five Card Stud (5 cartas), Three Card Poker (3 cartas).

**Risco:** Lib datada (class components no exemplo, porem a API e funcional e pura JS). Pode precisar de `@types/pokersolver` para TypeScript ou declaracao manual de tipos.

**Mitigacao:** Testar import no projeto antes de iniciar desenvolvimento. Se pokersolver nao funcionar com React 19/Next.js 16, alternativa: poker-evaluator-ts (TypeScript nativo, Two Plus Two algorithm, 22M maos/segundo).

**Acao:** `npm install pokersolver` no projeto. Criar `types/pokersolver.d.ts` se necessario.

## Problema 2 — Card rendering (52 PNGs vs SVG inline vs CSS puro)

**Impacto:** O projeto NAO tem sistema de renderizacao de cartas de baralho. Precisa ser criado.

**3 opcoes avaliadas:**

| Opcao | Descricao | Pros | Contras | Decisao |
|-------|-----------|------|---------|:-------:|
| A) 52 PNGs | 52 imagens PNG de cartas + 1 back | Visual premium, total controle | 53 assets a gerenciar, ~2MB total | DESCARTADO |
| B) SVG inline | SVG gerado por componente React (naipe + valor) | Zero assets, escalavel, leve | Visual pode ficar "flat" sem efeitos premium | DESCARTADO |
| C) CSS puro + Unicode | Div estilizado com valor + naipe Unicode | Zero assets, zero SVG, leve, premium com CSS gradients/shadows | Precisa de fonte com naipes, naipes podem variar por OS | **ESCOLHIDO** |

**Opcao C (escolhida) — Detalhamento:**

Cada carta e um `motion.div` com:
- Fundo: gradiente linear `#0A0A0A` -> `#1A1A1A` (dark premium)
- Valor: texto em font-weight 800, cor branca, text-shadow dourado
- Naipe: texto Unicode (hearts=\u2665, diamonds=\u2666, clubs=\u2663, spades=\u2660) com cor condicionada (vermelho dourado `#D4A843` para hearts/diamonds, verde neon `#00E676` para clubs/spades)
- Borda: 1px solid rgba(212,168,67,0.3)
- Border-radius: 8px
- Box-shadow: 0 2px 8px rgba(0,0,0,0.5)
- Flip: CSS `rotateY(180deg)` + `backface-visibility: hidden` + container `perspective: 1000px` + `transform-style: preserve-3d`

**Vantagem critica:** Zero assets de cartas a gerenciar. Visual 100% controlado via CSS. Compativel com qualquer resolucao (NUI landscape 1280x720 ate desktop 4K).

## Problema 3 — State machine complexa no Ultimate Texas Hold'em

**Impacto:** O UTH tem 6 estados sequenciais com betting em 3 deles. Mais complexo que qualquer outro jogo do Blackout Casino.

**Detalhamento dos estados UTH:**

```
IDLE -> BETTING (ante+blind) -> PREFLOP_DECISION (4x/3x/check)
  -> FLOP_DEAL -> FLOP_DECISION (2x/check)
  -> TURN_DEAL -> RIVER_DEAL -> RIVER_DECISION (1x/fold)
  -> SHOWDOWN -> RESULT -> IDLE
```

**Se jogador apostou em qualquer round anterior, pula direto para SHOWDOWN apos as community cards serem reveladas.**

**Mitigacao:** Implementar como state machine explicita com `useReducer` ao inves de multiplos `useState`. Cada transicao valida que o estado anterior e correto. Isso previne bugs de estado inconsistente (ex: apostar 4x e depois 2x no flop).

```typescript
type PokerState =
  | { phase: 'idle' }
  | { phase: 'betting' }
  | { phase: 'preflop'; playerCards: Card[]; canBet: boolean }
  | { phase: 'flop'; communityCards: Card[]; canBet: boolean }
  | { phase: 'turn_river'; communityCards: Card[]; canBet: boolean }
  | { phase: 'showdown'; dealerCards: Card[]; result: HandResult }
  | { phase: 'result'; payout: number; handName: string };
```

## Problema 4 — Progressive Jackpot pool persistente entre sessoes

**Impacto:** O jackpot progressivo precisa persistir no MySQL entre sessoes do servidor. Se o servidor reinicia, o pool NAO pode zerar.

**Detalhamento:**
- Tabela `casino_poker_jackpot` com campo `pool_amount` DECIMAL(15,2)
- Seed inicial: $10.000 (configuravel pelo admin)
- Contribuicao por mao: $1 fixo (71% para pool, 29% para casa)
- Ao ganhar: pool reseta para seed
- Locking: UPDATE com WHERE para evitar race condition em maos simultaneas

**SQL de atualizacao atomica:**
```sql
-- Contribuir para o pool (a cada mao com side bet)
UPDATE casino_poker_jackpot
SET pool_amount = pool_amount + 0.71,
    total_contributed = total_contributed + 1.00,
    updated_at = NOW()
WHERE id = 1;

-- Pagar jackpot (Royal Flush acertou)
UPDATE casino_poker_jackpot
SET pool_amount = seed_amount,
    times_won = times_won + 1,
    last_won_at = NOW(),
    last_won_by = ?,
    last_won_amount = pool_amount,
    updated_at = NOW()
WHERE id = 1 AND pool_amount = ?;
-- WHERE pool_amount = ? evita pagar 2x se 2 Royal Flush simultaneos
```

## Problema 5 — Comunicacao multi-round no UTH (3 fetchNui por mao)

**Impacto:** No Caribbean Stud, sao 2 fetchNui (ante + fold/raise). No UTH, sao ate 4 fetchNui (ante+blind, preflop decision, flop decision, river decision). Mais round-trips NUI <-> Server <-> MySQL.

**Detalhamento do fluxo UTH:**
```
[NUI] fetchNui('casino:poker:ante', { bet, mode:'uth', clientSeed, tripsbet })
  -> [Handler] valida saldo, gera deck (Provably Fair), deal 2 hole cards
  <- { playerCards, hashPre, sessionId }

[NUI] fetchNui('casino:poker:preflop', { sessionId, action:'bet4x'|'bet3x'|'check' })
  -> [Handler] registra decision, deal flop (3 community)
  <- { communityCards: [3], canStillBet }

[NUI] fetchNui('casino:poker:flop', { sessionId, action:'bet2x'|'check' })
  -> [Handler] registra decision, deal turn+river (2 community)
  <- { communityCards: [5], canStillBet }

[NUI] fetchNui('casino:poker:river', { sessionId, action:'bet1x'|'fold' })
  -> [Handler] avalia maos, calcula payout, atualiza saldo
  <- { dealerCards, result, payout, handName, dealerHandName, blindPayout, tripsPayout }
```

**Mitigacao:** SessionId unico por mao garante que cada fetchNui pertence a mesma mao. O handler valida que o sessionId esta no estado correto para aquela acao (ex: nao pode enviar 'flop' se o preflop nao foi processado).

## 5.1 Schema Completo (4 tabelas)

```sql
-- =============================================
-- BLACKOUT CASINO - POKER (#6)
-- Schema MySQL - casino_poker
-- Copiar-colar executavel
-- =============================================

-- Tabela 1: Configuracao do jogo
CREATE TABLE IF NOT EXISTS casino_poker_config (
    id INT AUTO_INCREMENT PRIMARY KEY,
    config_key VARCHAR(50) NOT NULL UNIQUE,
    config_value TEXT NOT NULL,
    description VARCHAR(255),
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Tabela 2: Historico de maos
CREATE TABLE IF NOT EXISTS casino_poker_hands (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    player_identifier VARCHAR(60) NOT NULL,
    mode ENUM('caribbean', 'uth') NOT NULL,
    ante_bet DECIMAL(15,2) NOT NULL,
    raise_bet DECIMAL(15,2) DEFAULT 0,
    blind_bet DECIMAL(15,2) DEFAULT 0,
    play_bet DECIMAL(15,2) DEFAULT 0,
    trips_bet DECIMAL(15,2) DEFAULT 0,
    jackpot_bet DECIMAL(15,2) DEFAULT 0,
    player_cards VARCHAR(30) NOT NULL COMMENT 'Ex: Ad,Kh,Qs,Jd,9c',
    dealer_cards VARCHAR(30) NOT NULL,
    community_cards VARCHAR(30) DEFAULT NULL COMMENT 'UTH only',
    player_hand_name VARCHAR(30) NOT NULL COMMENT 'Ex: Full House',
    dealer_hand_name VARCHAR(30) NOT NULL,
    dealer_qualified TINYINT(1) NOT NULL DEFAULT 1,
    result ENUM('win', 'lose', 'push', 'fold') NOT NULL,
    payout DECIMAL(15,2) NOT NULL DEFAULT 0,
    blind_payout DECIMAL(15,2) DEFAULT 0,
    trips_payout DECIMAL(15,2) DEFAULT 0,
    jackpot_payout DECIMAL(15,2) DEFAULT 0,
    rtp_preset VARCHAR(20) DEFAULT 'standard',
    server_seed_hash VARCHAR(64) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_player (player_identifier),
    INDEX idx_mode (mode),
    INDEX idx_created (created_at),
    INDEX idx_result (result)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Tabela 3: Progressive Jackpot Pool
CREATE TABLE IF NOT EXISTS casino_poker_jackpot (
    id INT AUTO_INCREMENT PRIMARY KEY,
    pool_amount DECIMAL(15,2) NOT NULL DEFAULT 10000.00,
    seed_amount DECIMAL(15,2) NOT NULL DEFAULT 10000.00,
    contribution_rate DECIMAL(5,2) NOT NULL DEFAULT 0.71 COMMENT '71% vai pro pool',
    house_rate DECIMAL(5,2) NOT NULL DEFAULT 0.29 COMMENT '29% para a casa',
    total_contributed DECIMAL(15,2) NOT NULL DEFAULT 0,
    times_won INT NOT NULL DEFAULT 0,
    last_won_at TIMESTAMP NULL,
    last_won_by VARCHAR(60) NULL,
    last_won_amount DECIMAL(15,2) DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Tabela 4: Sessoes Provably Fair
CREATE TABLE IF NOT EXISTS casino_poker_provably_fair (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    player_identifier VARCHAR(60) NOT NULL,
    server_seed VARCHAR(64) NOT NULL,
    server_seed_hash VARCHAR(64) NOT NULL,
    client_seed VARCHAR(64) NOT NULL,
    nonce INT NOT NULL DEFAULT 0,
    revealed TINYINT(1) NOT NULL DEFAULT 0,
    revealed_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_player (player_identifier),
    INDEX idx_hash (server_seed_hash)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 6.5 Mecanica de Tumble e Impacto no RTP

O tumble/cascade tem impacto DIRETO no RTP porque cada tumble e uma "nova chance de win" sem custo adicional ao jogador. O RTP efetivo de um spin com tumble e:

```
RTP_efetivo = RTP_base_spin + (hit_rate_tumble_1 × payout_medio_tumble_1) 
            + (hit_rate_tumble_2 × payout_medio_tumble_2) + ...
```

Na pratica, isso significa que o paytable BASE dos simbolos pode ter payouts MENORES que um slot sem tumble (porque o tumble adiciona valor). Por exemplo:
- Slot sem tumble com 96% RTP: simbolos pagam mais por combinacao
- Slot com tumble com 96% RTP: simbolos pagam MENOS individualmente, mas a soma de todos os tumbles atinge o mesmo 96%

Isso e IMPORTANTE para o design: os payouts na paytable parecem "baixos" comparado a um slot classico, mas o jogador ganha em VOLUME de wins por spin, nao em valor individual. O Sweet Bonanza tem payout maximo de 50x por um single cluster de 12+ simbolos premium — parece pouco para um slot com max win de 21.175x. A magia esta nos multiplicadores que ACUMULAM durante tumbles nas free spins.

## 6.6 Formula de Selecao de Simbolos (Provably Fair)

Para cada posicao no grid 6x5 (30 posicoes), o algoritmo seleciona um simbolo:

```
hash = HMAC-SHA256(server_seed, client_seed + ":" + nonce)
hex_string = hash.substring(posicao * 4, posicao * 4 + 4)  // 4 chars hex por posicao
decimal = parseInt(hex_string, 16)  // 0 a 65535
weighted_index = decimal % total_peso  // total_peso = 63 (base game)
simbolo = selecionar_por_peso(weighted_index, tabela_pesos)
```

Assim, 1 hash gera 30 posicoes (30 × 4 = 120 chars hex, hash SHA-256 tem 64 chars — para cobrir 30 posicoes, usar HMAC com nonce_sub incrementado por chunk de 16 posicoes).

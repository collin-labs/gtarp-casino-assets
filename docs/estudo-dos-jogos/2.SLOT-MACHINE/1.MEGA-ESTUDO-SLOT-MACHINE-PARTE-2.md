## Problema 1: Tumble Performance — 30 Celulas Animando Simultaneamente

**Sintoma:** No modo Video (6x5 = 30 celulas), uma sequencia de tumble anima: fade out dos wins (N celulas) + gravity drop dos acima + fade in dos novos. Se N = 12 (cluster grande), sao ~30 motion.div animando ao mesmo tempo no NUI com GPU compartilhada pelo GTA.

**Causa tecnica:** Framer Motion cria uma animacao spring/tween por elemento. 30 springs simultaneas = 30 requestAnimationFrame callbacks competindo. No NUI (CEF), a GPU ja esta renderizando o GTA em 30-60fps. Animacoes CSS/JS adicionais competem pelo mesmo frame budget de 16ms.

**Dados concretos:**
- Framer Motion spring update: ~0.1ms por elemento por frame
- 30 elementos: ~3ms por frame (dentro do budget de 16ms)
- Porem, layout recalculation (reflow) ao remover/adicionar DOM nodes: ~2-5ms
- Total estimado: ~5-8ms por frame durante tumble (aceitavel, mas margem apertada)

**Solucao multi-camada:**
1. **Stagger aggressivo:** Nao animar todas 30 celulas ao mesmo tempo. Exit: stagger 30ms por celula (total 900ms para 30). Drop: stagger 20ms por coluna (total 120ms para 6 colunas). Enter: stagger 30ms por celula. Total tumble cycle: ~1.2s.
2. **will-change: transform, opacity** em cada celula ANTES da animacao. Remove APOS. Evita reflow.
3. **Batch DOM updates:** Usar requestAnimationFrame para sincronizar remocoes e insercoes no mesmo frame.
4. **Fallback simples:** Se detectar FPS < 30 (performance.now delta), reduzir para opacity-only transitions (sem spring, sem scale).

**Codigo de deteccao de performance:**
```javascript
// slotPerformance.ts
let lastFrame = performance.now();
let lowFpsCount = 0;

function checkFrameRate() {
  const now = performance.now();
  const delta = now - lastFrame;
  lastFrame = now;
  
  if (delta > 33) lowFpsCount++; // abaixo de 30fps
  else lowFpsCount = Math.max(0, lowFpsCount - 1);
  
  return lowFpsCount < 5; // true = performance OK
}

// Uso no VideoSlot
const highPerf = checkFrameRate();
const tumbleDuration = highPerf ? 200 : 100; // ms
const tumbleType = highPerf ? 'spring' : 'tween'; // spring mais bonito, tween mais leve
```

**Impacto:** ~30 linhas de performance monitor. Zero dependencia extra.

## Problema 2: RTP Calibration — Garantir que Symbol Weights Produzem o RTP Alvo

**Sintoma:** Os pesos dos simbolos (coroa:3, anel:4, ..., topazio:10) definem a probabilidade de cada simbolo aparecer. O RTP resultante depende da combinacao desses pesos com os payouts e com a mecanica de tumble. Se os pesos estao errados, o RTP real diverge do alvo (96.50%).

**Causa tecnica:** O RTP de um slot com tumble nao e calculavel por formula simples (nao e so "probabilidade × payout"). Cada tumble gera um novo grid que pode ter mais wins, e esses wins dependem dos mesmos pesos. E uma cadeia estocastica que requer simulacao para calibrar.

**Solucao: Script de Simulacao Monte Carlo (server-side)**
```javascript
// calibrate_rtp.js — rodar offline para validar pesos
const SYMBOL_WEIGHTS = {
  crown: 3, ring: 4, hourglass: 4, chalice: 5,
  ruby: 8, sapphire: 8, emerald: 9, amethyst: 9, topaz: 10,
  scatter: 2
};
const TOTAL_WEIGHT = Object.values(SYMBOL_WEIGHTS).reduce((a, b) => a + b, 0);
const PAYOUTS = {
  crown:     { 8: 10, 9: 12, 10: 15, 11: 25, 12: 50 },
  ring:      { 8: 8,  9: 10, 10: 12, 11: 20, 12: 40 },
  hourglass: { 8: 5,  9: 6,  10: 8,  11: 15, 12: 25 },
  chalice:   { 8: 3,  9: 4,  10: 5,  11: 10, 12: 15 },
  ruby:      { 8: 1.5,9: 2,  10: 2.5,11: 5,  12: 8 },
  sapphire:  { 8: 1,  9: 1.5,10: 2,  11: 4,  12: 5 },
  emerald:   { 8: 0.8,9: 1,  10: 1.5,11: 3,  12: 4 },
  amethyst:  { 8: 0.5,9: 0.8,10: 1,  11: 2,  12: 3 },
  topaz:     { 8: 0.25,9:0.5,10: 0.8,11: 1.5,12: 2 },
};

function generateGrid() {
  const grid = [];
  for (let i = 0; i < 30; i++) {
    let r = Math.floor(Math.random() * TOTAL_WEIGHT);
    for (const [sym, w] of Object.entries(SYMBOL_WEIGHTS)) {
      r -= w;
      if (r < 0) { grid.push(sym); break; }
    }
  }
  return grid;
}

function evaluateWins(grid) {
  const counts = {};
  grid.forEach(s => { counts[s] = (counts[s] || 0) + 1; });
  let totalPayout = 0;
  for (const [sym, count] of Object.entries(counts)) {
    if (sym === 'scatter') continue;
    if (count >= 8) {
      const tier = Math.min(count, 12);
      totalPayout += PAYOUTS[sym]?.[tier] || 0;
    }
  }
  return totalPayout;
}

function simulateSpin() {
  let grid = generateGrid();
  let totalWin = 0;
  let tumbles = 0;
  
  while (tumbles < 50) {
    const win = evaluateWins(grid);
    if (win === 0) break;
    totalWin += win;
    // Tumble: remove winning symbols, generate new
    const winSymbols = new Set();
    const counts = {};
    grid.forEach(s => { counts[s] = (counts[s] || 0) + 1; });
    for (const [sym, count] of Object.entries(counts)) {
      if (count >= 8 && sym !== 'scatter') winSymbols.add(sym);
    }
    grid = grid.map(s => winSymbols.has(s) ? generateGrid()[0] : s);
    // Simplificado: na pratica, novos simbolos gerados individualmente
    tumbles++;
  }
  return totalWin;
}

// Rodar 1M spins para calcular RTP
const SPINS = 1_000_000;
let totalWagered = SPINS; // bet = 1 por spin
let totalPaid = 0;
for (let i = 0; i < SPINS; i++) {
  totalPaid += simulateSpin();
}
const rtp = (totalPaid / totalWagered) * 100;
console.log(`RTP simulado: ${rtp.toFixed(2)}% (${SPINS} spins)`);
// Ajustar pesos ate rtp ~= 96.50
```

**Impacto:** Script offline (~80 linhas). Rodar antes de deploy para validar. Ajustar pesos ate RTP convergir para alvo. NAO faz parte do runtime — e ferramenta de calibracao.

## Problema 3: Provably Fair para Grid 6x5 — Hash Nao Cobre 30 Posicoes

**Sintoma:** HMAC-SHA256 gera 64 caracteres hex (32 bytes). Para 30 posicoes de grid, precisamos de 30 selecoes de simbolo. Se usarmos 4 chars hex por posicao (range 0-65535), precisamos de 120 chars — mas o hash so tem 64.

**Causa tecnica:** Um unico hash SHA-256 produz 256 bits (32 bytes, 64 hex chars). Para cobrir 30 posicoes com 4 chars cada, precisamos de 120 hex chars — quase o dobro.

**Solucao: Chain hashing com sub-nonces**
```javascript
// provablyFairSlots.js
const crypto = require('crypto');

function generateGrid(serverSeed, clientSeed, nonce) {
  const grid = [];
  // Gerar 2 hashes encadeados para cobrir 30 posicoes
  const hash1 = crypto.createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}:0`)
    .digest('hex'); // 64 chars → 16 posicoes
  const hash2 = crypto.createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}:1`)
    .digest('hex'); // 64 chars → mais 16 posicoes
  
  const fullHex = hash1 + hash2; // 128 chars → 32 posicoes (30 usadas)
  
  for (let i = 0; i < 30; i++) {
    const chunk = fullHex.substring(i * 4, i * 4 + 4);
    const decimal = parseInt(chunk, 16); // 0-65535
    const weightedIndex = decimal % TOTAL_WEIGHT;
    grid.push(selectSymbolByWeight(weightedIndex));
  }
  return grid;
}

function selectSymbolByWeight(index) {
  let acc = 0;
  for (const [symbol, weight] of Object.entries(SYMBOL_WEIGHTS)) {
    acc += weight;
    if (index < acc) return symbol;
  }
  return 'topaz'; // fallback
}

function hashServerSeed(serverSeed) {
  return crypto.createHash('sha256').update(serverSeed).digest('hex');
}

// Verificacao client-side (modal Provably Fair)
function verifyResult(serverSeed, clientSeed, nonce, expectedGrid) {
  const grid = generateGrid(serverSeed, clientSeed, nonce);
  return JSON.stringify(grid) === JSON.stringify(expectedGrid);
}
```

**Impacto:** ~40 linhas no handler. Chain de 2 hashes resolve o problema. Cada sub-nonce (:0, :1) gera hash independente mas deterministico. Verificavel pelo jogador.

## Problema 4: Tumble Gera Novos Simbolos — Esses Tambem Precisam Ser Provably Fair

**Sintoma:** Apos um win, o tumble remove simbolos e gera novos. Esses novos simbolos tambem precisam ser deterministas e verificaveis. Se o servidor gera aleatoriamente, ele pode manipular para dar (ou nao dar) wins nas tumbles.

**Solucao: Nonce incremental por tumble**
```javascript
// Cada tumble usa nonce incrementado
function generateTumbleSymbols(serverSeed, clientSeed, nonce, tumbleIndex, count) {
  const symbols = [];
  const hash = crypto.createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}:tumble:${tumbleIndex}`)
    .digest('hex');
  
  for (let i = 0; i < count && i < 16; i++) {
    const chunk = hash.substring(i * 4, i * 4 + 4);
    const decimal = parseInt(chunk, 16);
    const weightedIndex = decimal % TOTAL_WEIGHT;
    symbols.push(selectSymbolByWeight(weightedIndex));
  }
  // Se count > 16, chain com sub-hash
  if (count > 16) {
    const hash2 = crypto.createHmac('sha256', serverSeed)
      .update(`${clientSeed}:${nonce}:tumble:${tumbleIndex}:ext`)
      .digest('hex');
    for (let i = 0; i < count - 16; i++) {
      const chunk = hash2.substring(i * 4, i * 4 + 4);
      const decimal = parseInt(chunk, 16);
      symbols.push(selectSymbolByWeight(decimal % TOTAL_WEIGHT));
    }
  }
  return symbols;
}
```

**Impacto:** ~25 linhas adicionais. Cada tumble e deterministico pelo mesmo seed pair + tumbleIndex. Jogador pode verificar cada tumble individualmente.

## Problema 5: Free Spins — Multiplicadores Precisam Ser Provably Fair Tambem

**Sintoma:** Durante free spins, multiplicadores random (2x-100x) podem aparecer. Se o servidor decide "quando" e "quanto" arbitrariamente, pode manipular os resultados.

**Solucao: Multiplicador determinado pelo hash**
```javascript
const MULTIPLIER_VALUES = [2, 5, 8, 10, 15, 25, 50, 100];
const MULTIPLIER_CHANCE = 0.15; // 15% chance por tumble durante FS

function shouldSpawnMultiplier(serverSeed, clientSeed, nonce, tumbleIndex) {
  const hash = crypto.createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}:mult:${tumbleIndex}`)
    .digest('hex');
  const chance = parseInt(hash.substring(0, 4), 16) / 65535;
  return chance < MULTIPLIER_CHANCE;
}

function getMultiplierValue(serverSeed, clientSeed, nonce, tumbleIndex) {
  const hash = crypto.createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}:multval:${tumbleIndex}`)
    .digest('hex');
  const index = parseInt(hash.substring(0, 4), 16) % MULTIPLIER_VALUES.length;
  return MULTIPLIER_VALUES[index];
}
```

**Impacto:** ~20 linhas. Multiplicadores sao deterministas e verificaveis. O jogador sabe que o servidor nao pode "escolher" dar multiplicador baixo quando o win e grande.

### Queries Admin Prontas

```sql
-- RTP real calculado (ultimas 24h)
SELECT 
  ROUND(SUM(total_win) / SUM(bet) * 100, 2) AS rtp_real_24h,
  COUNT(*) AS total_spins,
  SUM(bet) AS total_wagered,
  SUM(total_win) AS total_paid
FROM casino_slot_spins
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 24 HOUR);

-- Top 10 maiores wins
SELECT player_name, total_win, win_multiplier, mode, created_at
FROM casino_slot_spins
WHERE total_win > 0
ORDER BY total_win DESC
LIMIT 10;

-- Spins por modo (Classic vs Video)
SELECT mode, COUNT(*) AS spins, SUM(bet) AS wagered, SUM(total_win) AS paid,
  ROUND(SUM(total_win)/SUM(bet)*100, 2) AS rtp
FROM casino_slot_spins
GROUP BY mode;

-- Free spins trigger rate
SELECT 
  COUNT(*) AS total_spins,
  SUM(free_spins_triggered) AS fs_triggered,
  ROUND(SUM(free_spins_triggered)/COUNT(*)*100, 2) AS fs_trigger_rate
FROM casino_slot_spins
WHERE mode = 'video';

-- Jackpot status
SELECT current_pool, last_winner_name, last_win_amount, last_win_at
FROM casino_slot_jackpot WHERE id = 1;

-- Jogadores mais ativos (ultimos 7 dias)
SELECT player_name, COUNT(*) AS spins, SUM(bet) AS wagered, 
  SUM(total_win) AS won, SUM(total_win) - SUM(bet) AS net
FROM casino_slot_spins
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY player_id, player_name
ORDER BY spins DESC
LIMIT 20;
```

## 7.2 Seguranca / Anti-Cheat (10 Itens)

| # | Aspecto | Implementacao Concreta |
|---|---------|----------------------|
| 1 | **Provably Fair** | HMAC-SHA256. Server seed hashado ANTES do spin. Client seed do jogador. Nonce incrementa. Chain hashing para 30 posicoes. Verificacao pos-sessao. Codigo real na PARTE 4. |
| 2 | **Tumbles deterministicos** | Cada tumble usa sub-nonce (tumble:0, tumble:1, ...). Novos simbolos gerados pelo mesmo seed pair. Verificaveis. |
| 3 | **Multiplicadores deterministicos** | Chance e valor de multiplicador derivados do hash (mult:{tumbleIndex}). Impossivel servidor escolher valores. |
| 4 | **Validacao de input (4 checks)** | (1) mode e 'classic' ou 'video'? (2) bet >= minBet e <= maxBet? (3) bet <= saldo? (4) cooldown respeitado? Rejeitar se qualquer falha. |
| 5 | **Rate limit** | Cooldown 500ms entre spins (config). Map<playerId, timestamp>. Previne spam. |
| 6 | **Max tumbles** | Limite de 50 tumbles por spin (config). Previne loops infinitos. |
| 7 | **Max win cap** | Jackpot tem trigger threshold. Win regular capped em 5000x. Previne bugs de multiplicador infinito. |
| 8 | **Audit log completo** | Toda acao em casino_slot_audit: SPIN (bet, win, grid, tumbles, multipliers, seeds), BUY_BONUS, JACKPOT_WIN, CONFIG_CHANGE, ERROR. Append-only. JSON details flexivel. |
| 9 | **Jackpot atomico** | Pool incrementado via UPDATE ... SET current_pool = current_pool + contribution. Atomico. Lock row durante payout. |
| 10 | **Server-side RNG** | Resultado calculado INTEIRO no server. Client recebe grid pronto + animacao data. Client NAO decide resultado — apenas anima. |

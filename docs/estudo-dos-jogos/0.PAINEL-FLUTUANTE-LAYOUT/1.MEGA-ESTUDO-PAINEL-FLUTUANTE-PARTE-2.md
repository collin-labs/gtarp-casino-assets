## Problema 1 — GoldParticles roda 60fps CONTINUAMENTE (desperdicio de GPU)

**Sintoma:** Quando um jogo abre (z-index 60, cobrindo todo o painel), as particulas continuam rodando requestAnimationFrame a 60fps POR BAIXO do jogo — invisivel para o jogador mas consumindo GPU.

**Causa:** GoldParticles.tsx nao recebe prop `paused` nem verifica se um jogo esta ativo. O useEffect com requestAnimationFrame inicia no mount e so para no unmount do componente — mas o componente NUNCA desmonta porque esta abaixo do overlay do jogo na arvore DOM.

**Codigo Real (GoldParticles.tsx linhas ~43-85):**
```tsx
export default function GoldParticles({ width, height, mouseX, mouseY }: {
  width: number; height: number; mouseX: number; mouseY: number
}) {
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const particlesRef = useRef<Particle[]>([]);

  useEffect(() => {
    // ... setup particles
    let animId: number;
    const draw = () => {
      // ... draw logic runs EVERY FRAME even when invisible
      animId = requestAnimationFrame(draw);
    };
    animId = requestAnimationFrame(draw);
    return () => cancelAnimationFrame(animId);
  }, [width, height]);
  // PROBLEMA: nenhuma verificacao de activeGame
```

**Solucao (3 opcoes, em ordem de preferencia):**

1. **Prop `paused` (mais simples, ~5L de edicao):** Adicionar prop `paused: boolean` e fazer early return no draw loop:
```tsx
// BlackoutCasino.tsx — passar prop
<GoldParticles width={panelSize.w} height={panelSize.h}
  mouseX={mousePos.current.x} mouseY={mousePos.current.y}
  paused={activeGame !== null} />

// GoldParticles.tsx — respeitar prop
const draw = () => {
  if (paused) { animId = requestAnimationFrame(draw); return; }
  // ... draw logic
};
```

2. **OffscreenCanvas (melhor performance, ~40L de refactor):** Mover draw loop para Worker thread. Main thread nao eh impactada. Chrome 69+ — viavel no CEF M103.

3. **Unmount condicional (mais drastico):** So renderizar GoldParticles quando `!activeGame`. Particulas somem INSTANTANEAMENTE quando jogo abre. Pode causar flash visual.

**Impacto de nao resolver:** ~5-10% GPU desperdicada enquanto jogador esta em qualquer jogo. Em GPUs low-end (GTX 1050), isso pode ser a diferenca entre 30fps e 25fps no jogo.

## Problema 2 — src/games/slots/ eh DUPLICATA OBSOLETA (3.184 linhas de lixo)

**Sintoma:** Existem 2 pastas com componentes de Slots quase identicos: `components/games/slots/` (ativa, usada) e `src/games/slots/` (duplicata, nao referenciada por nenhum import).

**Causa:** Provavelmente um refactor antigo que moveu os arquivos de src/ para components/ mas nao deletou a pasta original.

**Codigo Real (diff parcial):**
```bash
# Ambas pastas tem os mesmos 18 arquivos

```

**Impacto de nao resolver:** Confusao em auditorias, risco de editar o arquivo errado, bundle pode incluir codigo nao usado se tree-shaking falhar.

## Problema 3 — Prop Drilling de `lang` e `saldo` por Toda a Arvore

**Sintoma:** A prop `lang` eh passada manualmente de BlackoutCasino → BentoGrid → GameCard, BlackoutCasino → HeroCarousel, BlackoutCasino → Dock, BlackoutCasino → GameModal, e de cada jogo via prop. Quando o GCoin for implementado, `saldo` tera que percorrer o mesmo caminho. Cada novo componente precisa receber lang explicitamente.

**Causa:** Nao existe Context global. Toda propagacao eh via props diretas.

**Codigo Real (BlackoutCasino.tsx ~linhas 200-340):**
```tsx
<BentoGrid
  games={TAB_GAMES[activeTab]}
  heroGames={TAB_HERO_GAMES[activeTab]}
  layout={TAB_LAYOUTS[activeTab]}
  lang={lang}                    // <- prop drilling
  onGameSelect={...}
/>
<Dock
  lang={lang}                    // <- prop drilling
  activeTab={activeTab}
  setActiveTab={setActiveTab}
/>
<GameModal
  game={selectedGame}
  lang={lang}                    // <- prop drilling
  onClose={...}
  onPlay={...}
/>
```

**Solucao:** Criar `CasinoContext.tsx` (~50L):
```tsx
interface CasinoContextType {
  lang: "br" | "in";
  setLang: (lang: "br" | "in") => void;
  gcoinBalance: number;
  playerId: string | null;
  panelSize: { w: number; h: number };
}

const CasinoContext = createContext<CasinoContextType>(/* defaults */);

export function CasinoProvider({ children }: { children: ReactNode }) {
  const [lang, setLang] = useState<"br" | "in">("br");
  const [gcoinBalance, setGcoinBalance] = useState(0);
  // ... other state
  return (
    <CasinoContext.Provider value={{ lang, setLang, gcoinBalance, ... }}>
      {children}
    </CasinoContext.Provider>
  );
}

export function useCasino() { return useContext(CasinoContext); }
```

**Impacto de nao resolver:** Cada componente novo precisa de mais props. Quando GCoin chegar, prop drilling vai triplicar. Manutencao cada vez mais dificil.

## Problema 4 — Transicao Jogo Abre/Fecha eh INSTANTANEA (sem animacao)

**Sintoma:** Quando o jogador clica JOGAR AGORA no GameModal, o jogo aparece instantaneamente (opacity 0→1 SEM transicao). Quando clica VOLTAR, o jogo some instantaneamente. Nao ha AnimatePresence wrapping o switch de jogos.

**Causa:** O switch de jogos em BlackoutCasino.tsx (~linhas 373-395) usa condicionais simples sem AnimatePresence:
```tsx
{activeGame === "slots" && (
  <div style={{ position: "absolute", inset: 0, zIndex: 60, ... }}>
    <SlotsGame onBack={() => setActiveGame(null)} lang={lang} />
  </div>
)}
{activeGame === "cases" && (
  <div style={{ position: "absolute", inset: 0, zIndex: 60, ... }}>
    <CasesGame onBack={() => setActiveGame(null)} lang={lang} />
  </div>
)}
```

**Solucao:** Wrapar com AnimatePresence + motion.div (~25L de edicao):
```tsx
<AnimatePresence mode="wait">
  {activeGame && (
    <motion.div
      key={activeGame}
      initial={{ opacity: 0, scale: 0.96 }}
      animate={{ opacity: 1, scale: 1 }}
      exit={{ opacity: 0, scale: 0.96 }}
      transition={{ duration: 0.3, ease: "easeOut" }}
      style={{ position: "absolute", inset: 0, zIndex: 60,
               borderRadius: "inherit", overflow: "hidden" }}
    >
      {activeGame === "slots" && <SlotsGame onBack={...} lang={lang} />}
      {activeGame === "cases" && <CasesGame onBack={...} lang={lang} />}
      {activeGame === "roulette" && <RouletteGame onBack={...} lang={lang} />}
    </motion.div>
  )}
</AnimatePresence>
```

**Impacto de nao resolver:** Experiencia abrupta que quebra a fluidez premium do painel. Todo o cuidado com intro cinematografica e efeitos de hover eh desperdicado na transicao mais importante (painel → jogo).

## Problema 5 — Apenas 3 de 22 Jogos tem Case no Switch

**Sintoma:** BlackoutCasino.tsx tem switch para `activeGame === "slots"`, `"cases"`, `"roulette"`. Os outros 19 jogos (Crash, Mines, Blackjack, Poker, Dice, Plinko, Bicho, etc.) nao tem case. Clicar JOGAR AGORA nesses jogos executa `setActiveGame(name)` mas nenhum componente renderiza.

**Causa:** Os demais jogos ainda nao foram implementados. O switch precisa crescer conforme jogos sao criados.

**Codigo Real (BlackoutCasino.tsx ~linha 334):**
```tsx
const gameMap: Record<string, string> = {};
GAMES.forEach((g) => {
  gameMap[g.labelBR.toLowerCase()] = g.cardName.toLowerCase();
  gameMap[g.labelEN.toLowerCase()] = g.cardName.toLowerCase();
});
// ...
setActiveGame(gameMap[name] || name);
// Mas so 3 cases no switch:
// activeGame === "slots" | "cases" | "roulette"
```

**Solucao (gradual):** Conforme cada jogo for implementado, adicionar ao switch. Para jogos pendentes, mostrar tela "EM BREVE" com visual premium:
```tsx
{activeGame && !["slots","cases","roulette"].includes(activeGame) && (
  <motion.div ...>
    <ComingSoon gameName={activeGame} lang={lang}
      onBack={() => setActiveGame(null)} />
  </motion.div>
)}
```

**Impacto de nao resolver:** Jogador clica JOGAR AGORA em 19 jogos e nada acontece — frustracao e impressao de produto quebrado.

## Limitacao FiveM Especifica

**CEF M103 limita CSS moderno:** Confirmado pela pesquisa e discussoes no cfx.re. Container queries (Chrome 105), CSS nesting (Chrome 120), :has() (Chrome 105), oklch (Chrome 111) — NENHUM funciona no M103. O painel DEVE ser escrito com CSS compativel com Chrome 103: clamp(), @property, conic-gradient, flexbox, grid, custom properties tradicionais, transforms, animations, backdrop-filter (com moderacao).

**Tailwind v4 pode gerar CSS incompativel:** Tailwind v4 usa :has() e modern color functions internamente. Se o build output incluir essas features, vai quebrar no CEF M103. **VERIFICACAO OBRIGATORIA:** rodar build e inspecionar CSS output no CEF DevTools (localhost:13172). Se incompativel, regredir para Tailwind v3 ou usar PostCSS override.

  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_player_type (player_id, type),
  INDEX idx_game_id (game_id),
  INDEX idx_created_at (created_at),
  INDEX idx_player_date (player_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Tabela 3: Configuracoes admin

  config_key VARCHAR(100) NOT NULL PRIMARY KEY,
  config_value TEXT NOT NULL,

  description VARCHAR(255) DEFAULT NULL,
  updated_by VARCHAR(50) DEFAULT NULL,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

## 5.3 Queries Admin Uteis

```sql
-- Top 10 jogadores por saldo
SELECT player_id, gcoin_balance, vip_level, games_played
FROM casino_players ORDER BY gcoin_balance DESC LIMIT 10;

-- Transacoes das ultimas 24h
SELECT type, COUNT(*) as qtd, SUM(amount) as total
FROM casino_transactions
WHERE created_at > NOW() - INTERVAL 24 HOUR
GROUP BY type ORDER BY total DESC;

-- House profit (total apostado - total ganho)
SELECT
  SUM(CASE WHEN type = 'bet' THEN amount ELSE 0 END) as total_bets,
  SUM(CASE WHEN type = 'win' THEN amount ELSE 0 END) as total_wins,
  SUM(CASE WHEN type = 'bet' THEN amount ELSE 0 END) -
  SUM(CASE WHEN type = 'win' THEN amount ELSE 0 END) as house_profit
FROM casino_transactions;

-- Verificar config atual
SELECT config_key, config_value, description FROM casino_config ORDER BY config_key;
```

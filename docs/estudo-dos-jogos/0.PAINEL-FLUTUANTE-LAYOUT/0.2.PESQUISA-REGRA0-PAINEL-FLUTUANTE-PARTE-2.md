### Container Queries (Baseline desde 2023 — seguro)

Container queries permitem que componentes se adaptem ao tamanho do SEU container, nao do viewport. Isso eh fundamental para o painel flutuante porque o painel tem tamanho variavel (80vw × 78vh, max 1600px) e os componentes internos (GameCard, HeroCarousel) precisam se adaptar a ESSE espaco, nao a tela do GTA.

```css
/* Exemplo: GameCard adapta ao tamanho do grid slot, nao do viewport */
.bento-grid { container-type: inline-size; }
@container (min-width: 800px) { .game-card { grid-column: span 2; } }
@container (max-width: 500px) { .game-card { font-size: clamp(8px, 2cqw, 14px); } }
```

**RESTRICAO CRITICA:** FiveM usa CEF M103 (Chromium 103, junho 2022). Container queries foram lancadas no Chrome 105 (agosto 2022). Isso significa que container queries **PODEM NAO FUNCIONAR** no CEF M103 do FiveM. Precisamos verificar se a versao M140 (em espera por fix AMD) ja esta ativa ou se M103 eh o atual. Se M103: container queries NAO funcionam, manter media queries + clamp(). Se M140+: container queries seguras.

### CSS Nesting (Chrome 120+, Safari 17.2+, Firefox 117+ — 2023)

### sibling-index() e sibling-count() (Chrome 2025+)

Funcoes que retornam posicao do elemento entre siblings. Permite stagger nativo em CSS sem JavaScript:

```css
.game-card {
  transition: opacity 0.3s ease;
  transition-delay: calc((sibling-index() - 1) * 50ms);
  @starting-style { opacity: 0; }
}
```

**Status FiveM:** NAO funciona no CEF M103. Apenas Chrome 2025+. Para FiveM, manter stagger via Framer Motion (`delay: index * 0.05`). Para Web, sibling-index() substitui ~100 linhas de logica JS de stagger.

### View Transitions API (Chrome 2023+, cross-document 2026)

Permite transicoes suaves entre estados da pagina (SPA) ou entre paginas (MPA). Ideal para transicao entre abas do painel.

```css
@view-transition { navigation: auto; }
::view-transition-old(root) { animation: fade-out 0.3s; }
::view-transition-new(root) { animation: fade-in 0.3s; }
```

**Status FiveM:** NAO funciona no CEF M103. Para Web, View Transitions API eh poderosa mas tem drawbacks vs Framer Motion layout animations: nao interruptivel, bloqueia pointer events durante animacao, menos performante (screenshot vs transform). A pesquisa do Motion.dev confirma que layout animations sao SUPERIORES em apps React interativos. **Decisao: manter Framer Motion para transicoes de aba no painel.**

### @property (Chrome 85+ — 2020, seguro em M103)

Custom properties com tipo declarado. Permite animar custom properties que antes eram inaninaveis:

```css
@property --glow-intensity {
  syntax: '<number>';
  inherits: false;
  initial-value: 0;
}
.card { --glow-intensity: 0; transition: --glow-intensity 0.3s; }
.card:hover { --glow-intensity: 1; }
.card::after { opacity: var(--glow-intensity); }
```

**Status FiveM:** FUNCIONA no CEF M103 (Chrome 85+). Pode ser usado imediatamente para animar glows, intensidades, opacidades de efeitos do DS. **Recomendacao: adotar @property para glow system do painel.**

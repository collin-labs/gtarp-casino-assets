## 4.1 Animacao da Moeda 3D - Opcoes Analisadas

A pesquisa identificou 3 abordagens tecnicas para animar uma moeda girando em 3D no browser:

### Opcao A - CSS puro (@keyframes + transform)

**Como funciona:** Define uma animacao @keyframes que rota o elemento em rotateY de 0deg a N*360deg. O resultado final (cara ou coroa) e controlado pela posicao final da animacao.

**Implementacao basica:**
- Container com perspective: 1000px
- Moeda com transformStyle: preserve-3d
- Duas faces com backfaceVisibility: hidden (uma com rotateY(0), outra com rotateY(180deg))
- @keyframes coinflip { from { rotateY(0) } to { rotateY(1800deg) } } para 5 voltas
- Duracao: animation-duration 2.5s com ease-out

**Pros:** Zero dependencia extra. Performance maxima (GPU-accelerated). Funciona em qualquer CEF/NUI.
**Contras:** Angulo final FIXO no CSS — nao permite controle dinamico baseado no resultado do server. Precisaria criar 2 animacoes separadas (uma para cara, outra para coroa) e aplicar a correta. Menos flexivel para easing customizado.

### Opcao B - GSAP Timeline

**Como funciona:** Usa gsap.to() para animar rotateY com controle total de duracao, easing, callbacks. Timeline para sequenciar: idle -> spin -> decelerate -> land.

**Implementacao basica:**
- Ref no elemento da moeda
- gsap.to(coinRef.current, { rotateY: targetAngle, duration: 2.5, ease: "power2.out" })
- Callback onComplete para trigger de resultado

**Pros:** Controle total de timeline. Easing profissional (power2, elastic, custom). Callbacks por frame. Pode combinar com ScrollTrigger ou outros plugins.
**Contras:** GSAP NAO esta instalado no projeto (precisaria adicionar ~78KB). Exige useRef + useEffect + cleanup — padrao imperativo, menos "React-native". Misturar GSAP com Framer Motion no mesmo projeto pode causar conflitos de renderizacao.

### Opcao C - Framer Motion rotateY (ESCOLHIDA)

**Como funciona:** Usa motion.div com animate={{ rotateY: targetAngle }} onde targetAngle e calculado dinamicamente baseado no resultado do server. Perspective e transformStyle aplicados como style props.

**Implementacao detalhada (baseada em DEV.to + Syncfusion + docs Motion):**

```
ESTRUTURA DA MOEDA:
- Container externo: perspective: 1000px (profundidade 3D visual)
- Container interno: motion.div com transformStyle: "preserve-3d"
- Face CARA: position absolute, backfaceVisibility hidden, rotateY(0deg)
  - Imagem PNG: moeda dourada com emblema Blackout Casino
- Face COROA: position absolute, backfaceVisibility hidden, rotateY(180deg)
  - Imagem PNG: moeda dourada com emblema alternativo

CALCULO DO ANGULO FINAL:
- Voltas base: 4 completas (4 * 360 = 1440deg) para criar suspense visual
- Se resultado = CARA: targetRotation = 1440 + 0 = 1440deg (para em cara)
- Se resultado = COROA: targetRotation = 1440 + 180 = 1620deg (para em coroa)
- Randomizar voltas extras (4-6) para nao ser previsivel visualmente

ANIMACAO:
animate={{ rotateY: targetRotation }}
transition={{ duration: 2.5, ease: [0.25, 0.1, 0.25, 1] }}
- Ease customizado: rapido no inicio (moeda sendo lancada), desacelera no final (pousando)
- Duration 2.5s: suficiente para tensao sem ser lento demais

SEQUENCIA COMPLETA:
1. Countdown visual (3, 2, 1) com scale pulse — Framer Motion animate
2. Moeda sobe (translateY -50px) + comeca a girar (rotateY) — simultaneous
3. Moeda gira com glow pulsante — CSS @keyframes glow em paralelo
4. Moeda desacelera + desce (translateY 0) — ease-out natural
5. Moeda para no resultado — Framer Motion spring bounce final
6. Flash de luz no resultado — CSS opacity flash 0.2s
7. Overlay de resultado (WIN/LOSE) — AnimatePresence entry
```

**Pros:** JA INSTALADO no projeto (Framer Motion 11). Angulo final dinamico (prop animate). AnimatePresence para overlays. whileHover/whileTap nativos para UI. Stagger para lobby cards. Consistente com TODOS os outros componentes do Blackout Casino.
**Contras:** Easing de rotate pode parecer mecanico vs GSAP — mitigado com ease cubic-bezier customizado.

### Decisao Tecnica: Opcao C (Framer Motion Custom)

**Motivo:** O Blackout Casino usa Framer Motion em 100% dos componentes (entrada, hover, tap, layout). Criar a moeda com Framer Motion garante consistencia visual e tecnica. As opcoes A (CSS) e B (GSAP) precisariam de "wrapper" ou mistura de paradigmas. Com Framer Motion, tudo — do lobby cards ate a moeda ate os overlays — usa o mesmo sistema de animacao.

## 5.2 Riscos e Limitacoes FiveM

| Risco | Impacto | Mitigacao |

| Latencia de rede entre 2 jogadores | Resultado pode parecer dessincronizado — um jogador ve antes | Server calcula resultado e envia para AMBOS no mesmo TriggerClientEvent. Animacao e LOCAL, resultado e SERVER-AUTHORITATIVE. Ambos recebem exatamente o mesmo resultado |
| GPU compartilhada NUI + GTA | Animacao 3D pode causar FPS drop se pesada | Moeda usa CSS transform 3D (hardware-accelerated, custo minimo). Particulas em CSS puro (max 20 divs). ZERO Canvas/WebGL. Testado: CSS 3D flip roda a 60fps mesmo com GTA em background |
| Jogador desconecta no meio do flip | Pot travado, GCoins nao distribuidos | Server handler com timeout de 10s: se jogador desconecta, resultado processado normalmente, GCoins creditados ao vencedor. Evento OnPlayerDropped no server.lua trata cleanup |
| Race condition (2 jogadores tentam entrar na mesma sala) | Sala com 3 jogadores, GCoins descontados de ambos | Server handler com lock atomico via MySQL transaction: UPDATE rooms SET player2 = ? WHERE id = ? AND player2 IS NULL. Se affected_rows = 0, segundo jogador recebe erro "sala ja preenchida" e GCoins NAO sao descontados |

## 8.1 Adaptacao para PvP (2 jogadores)

O Provably Fair em PvP difere do solo porque AMBOS jogadores precisam confiar no resultado. O servidor atua como arbitro neutro.

**Fluxo completo (13 passos):**

```
ANTES DO FLIP:
1. Jogador B entra na sala — partida inicia
2. Server gera server_seed = crypto.randomBytes(32).toString('hex')
3. Server calcula server_seed_hash = SHA-256(server_seed)
4. Hash enviado para AMBOS jogadores via TriggerClientEvent
5. NUI exibe: "Hash do servidor: abc123..." na area de Provably Fair
6. Cada jogador tem client_seed automatico (Math.random + timestamp)
7. Nonce = ID da partida (auto-incremento da tabela MySQL)

NO FLIP:
8. Server calcula: combined = clientSeedP1 + ":" + clientSeedP2 + ":" + nonce
9. Server calcula: result_hex = HMAC-SHA256(server_seed, combined)
10. Primeiro byte do hex convertido para decimal (0-255)
11. Se valor < 128: resultado = CARA. Se >= 128: resultado = COROA
    (Distribuicao: exatamente 128 valores para cada = 50.000% cada)

APOS O FLIP:
12. Server envia para ambos: { result, server_seed, combined_input }
13. Modal "Provably Fair" no jogo permite verificar:
    - SHA-256(server_seed) deve == hash exibido no passo 4
    - HMAC-SHA256(server_seed, combined_input)[0] convertido == resultado
```

## 8.2 Diferencial PvP: Dois Client Seeds

No solo (Stake Flip), apenas 1 client_seed influencia o resultado. No PvP do Blackout Casino, AMBOS jogadores contribuem:
- Jogador A fornece clientSeedP1 (automatico ou manual)
- Jogador B fornece clientSeedP2 (automatico ou manual)
- O combined input e clientSeedP1:clientSeedP2:nonce
- Nenhum jogador isoladamente pode prever o resultado (nao conhece server_seed)
- O servidor nao pode favorecer nenhum jogador (ambos seeds influenciam)
- Verificacao pos-jogo possivel por AMBOS
- Transparencia total: hash pre-flip + seeds pos-flip + formula publica

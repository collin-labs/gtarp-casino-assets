## Problema 1: Race Condition — 2 jogadores entrando na mesma sala simultaneamente

**Sintoma:** Jogador A e Jogador B clicam "ENTRAR" na mesma sala no mesmo instante. Ambos tem saldo descontado. Sala fica com 3 jogadores. Resultado inconsistente — quem e o oponente real?

**Causa Tecnica:** Se o handler fizer SELECT para verificar status + UPDATE separados, existe janela de tempo entre as 2 queries onde ambos passam na verificacao. O MySQL nao garante atomicidade entre queries separadas sem transaction explicita.

**Codigo que causaria o problema (NAO usar):**
```javascript
// ERRADO — race condition entre SELECT e UPDATE
exports['oxmysql:execute']('SELECT * FROM casino_coinflip_games WHERE id = ? AND status = "waiting"', [roomId], (rows) => {
  if (rows.length === 0) return cb({ error: 'sala_indisponivel' });
  // JANELA: entre este SELECT e o UPDATE abaixo, outro jogador pode entrar
  exports['oxmysql:execute']('UPDATE casino_coinflip_games SET player2_id = ?, status = "active" WHERE id = ?', [playerId, roomId]);
  // Ambos jogadores passam no SELECT, ambos fazem UPDATE
});
```

**Solucao — Lock atomico via MySQL UPDATE condicional:**
```javascript
// CORRETO — atomico, sem race condition
exports['oxmysql:execute'](
  `UPDATE casino_coinflip_games
   SET player2_id = ?, player2_side = ?, status = 'active', updated_at = NOW()
   WHERE id = ? AND status = 'waiting' AND player2_id IS NULL`,
  [playerId, opponentSide, roomId],
  (affectedRows) => {
    if (affectedRows === 0) {
      // Sala ja foi preenchida por outro jogador OU nao existe mais
      return cb({ error: 'sala_indisponivel' });
      // GCoins NAO foram descontados (desconto so ocorre apos UPDATE sucesso)
    }
    // UPDATE retornou 1 row afetada = este jogador entrou com sucesso
    // AGORA descontar GCoins do jogador 2
    exports['oxmysql:execute'](
      'UPDATE users SET gcoin_balance = gcoin_balance - ? WHERE identifier = ? AND gcoin_balance >= ?',
      [betAmount, playerId, betAmount],
      (balanceRows) => {
        if (balanceRows === 0) {
          // Saldo insuficiente — reverter a entrada na sala
          exports['oxmysql:execute'](
            `UPDATE casino_coinflip_games SET player2_id = NULL, player2_side = NULL, status = 'waiting' WHERE id = ?`,
            [roomId]
          );
          return cb({ error: 'saldo_insuficiente' });
        }
        // Sucesso — iniciar flip
        iniciarFlip(roomId, cb);
      }
    );
  }
);
```

**Impacto:** Sem essa protecao, cada sala com high traffic (muitos jogadores vendo o lobby) teria risco de double-join. O UPDATE condicional garante que APENAS 1 jogador entra, independente de quantos clicam simultaneamente. O segundo recebe "sala_indisponivel" instantaneamente.

## Problema 2: Sincronizacao da animacao entre 2 jogadores

**Sintoma:** Jogador A ve a moeda girando e parando em CARA. Jogador B ve a moeda ainda girando ou parada em posicao diferente. Resultado visual dessincronizado causa desconfianca ("o outro jogador viu diferente de mim").

**Causa Tecnica:** A animacao da moeda e executada LOCALMENTE no browser/NUI de cada jogador (Framer Motion). Se os jogadores recebem o evento de inicio em momentos diferentes (latencia de rede varia de 20ms a 200ms entre jogadores), as animacoes comecam em momentos diferentes e parecem dessincronizadas.

**Solucao — Server-authoritative com resultado pre-determinado:**

O servidor NAO sincroniza a animacao frame-a-frame (impossivel em NUI). Em vez disso, o servidor:
1. Calcula o resultado ANTES de enviar para os jogadores (Provably Fair)
2. Envia para AMBOS no mesmo TriggerClientEvent: { resultado: "cara", serverSeedHash, animationSeed }
3. Cada jogador executa a animacao LOCALMENTE com o resultado ja conhecido
4. A moeda SEMPRE para na posicao correta em ambos os clients

```javascript
// Server handler — iniciar flip
function iniciarFlip(roomId, cb) {
  // Buscar sala completa
  exports['oxmysql:execute'](
    'SELECT * FROM casino_coinflip_games WHERE id = ? AND status = "active"',
    [roomId],
    (salas) => {
      const sala = salas[0];

      // Gerar resultado Provably Fair
      const serverSeed = require('crypto').randomBytes(32).toString('hex');
      const serverSeedHash = require('crypto').createHash('sha256').update(serverSeed).digest('hex');
      const combined = sala.client_seed_p1 + ':' + sala.client_seed_p2 + ':' + sala.id;
      const resultHex = require('crypto').createHmac('sha256', serverSeed).update(combined).digest('hex');
      const primeiroByte = parseInt(resultHex.substring(0, 2), 16);
      const resultado = primeiroByte < 128 ? 'cara' : 'coroa';

      // Determinar vencedor
      const winnerId = (resultado === sala.player1_side) ? sala.player1_id : sala.player2_id;
      const rakePercent = 2; // buscar da config
      const potTotal = sala.bet_amount * 2;
      const rakeAmount = Math.floor(potTotal * rakePercent / 100);
      const payoutAmount = potTotal - rakeAmount;

      // Salvar resultado no MySQL
      exports['oxmysql:execute'](
        `UPDATE casino_coinflip_games SET
         result = ?, winner_id = ?, payout_amount = ?, rake_amount = ?,
         server_seed = ?, server_seed_hash = ?,
         status = 'completed', completed_at = NOW()
         WHERE id = ?`,
        [resultado, winnerId, payoutAmount, rakeAmount, serverSeed, serverSeedHash, roomId]
      );

      // Creditar vencedor
      exports['oxmysql:execute'](
        'UPDATE users SET gcoin_balance = gcoin_balance + ? WHERE identifier = ?',
        [payoutAmount, winnerId]
      );

      // Enviar para AMBOS jogadores no MESMO evento
      const flipData = {
        roomId: sala.id,
        resultado: resultado,
        winnerId: winnerId,
        payoutAmount: payoutAmount,
        rakeAmount: rakeAmount,
        serverSeedHash: serverSeedHash,
        potTotal: potTotal
      };

      // TriggerClientEvent para jogador 1
      emitNet('coinflip:result', sala.player1_id, flipData);
      // TriggerClientEvent para jogador 2
      emitNet('coinflip:result', sala.player2_id, flipData);
      // Broadcast lobby update para todos
      emitNet('coinflip:lobbyUpdate', -1);
    }
  );
}
```

**Impacto:** Cada jogador recebe o MESMO resultado e executa a animacao localmente. Mesmo que a animacao comece com 100ms de diferenca, ambos veem o resultado correto ao final. A experiencia e visualmente satisfatoria para ambos.

## Problema 3: Jogador desconecta no meio do flip

**Sintoma:** Jogador A cria sala, Jogador B entra, flip inicia. Jogador B fecha o jogo/desconecta no meio da animacao. Pot fica travado, Jogador A nao recebe pagamento, sala fica em status "active" para sempre.

**Causa Tecnica:** O handler coinflip:result e disparado pelo server, mas se o jogador que perde desconecta antes de receber o evento, nao afeta o resultado — o problema e se o SERVIDOR travar antes de processar o resultado, ou se a sala nunca chegar ao flip.

**Solucao — Timeout server-side + cleanup no disconnect:**

```javascript
// No handler iniciarFlip: resultado e processado INTEIRAMENTE no server
// O jogador NAO precisa "confirmar" o resultado — server ja creditou/debitou
// Se jogador desconecta, o resultado ja esta no MySQL e o saldo ja foi atualizado

// Cleanup de salas abandonadas (executar a cada 60s via setInterval)
function limparSalasExpiradas() {
  const expireSeconds = 300; // 5 minutos padrao (config admin)

  // Salas "waiting" expiradas — devolver GCoins ao criador
  exports['oxmysql:execute'](
    `SELECT id, player1_id, bet_amount FROM casino_coinflip_games
     WHERE status = 'waiting' AND created_at < DATE_SUB(NOW(), INTERVAL ? SECOND)`,
    [expireSeconds],
    (salasExpiradas) => {
      salasExpiradas.forEach(sala => {
        // Devolver GCoins
        exports['oxmysql:execute'](
          'UPDATE users SET gcoin_balance = gcoin_balance + ? WHERE identifier = ?',
          [sala.bet_amount, sala.player1_id]
        );
        // Marcar como expirada
        exports['oxmysql:execute'](
          `UPDATE casino_coinflip_games SET status = 'expired' WHERE id = ?`,
          [sala.id]
        );
        // Notificar criador se online
        emitNet('coinflip:roomExpired', sala.player1_id, { roomId: sala.id, amountReturned: sala.bet_amount });
      });
      if (salasExpiradas.length > 0) emitNet('coinflip:lobbyUpdate', -1);
    }
  );
}

// Executar a cada 60 segundos
setInterval(limparSalasExpiradas, 60000);

// Cleanup no disconnect do jogador (server.lua)
// AddEventHandler('playerDropped', function(reason)

//     {playerId}
//   )
//   -- Devolver GCoins de salas canceladas

```

**Impacto:** Zero salas orfas. GCoins sempre retornam ao dono. Limpeza automatica a cada 60s + cleanup instantaneo no disconnect.

## Problema 4: Lobby polling vs push — performance com muitos jogadores

**Sintoma:** Se 50 jogadores estao no lobby e cada um faz fetchNui("coinflip:lobby") a cada 3 segundos, sao ~17 queries MySQL/segundo so para listar salas. Com 100+ jogadores simultaneos, o banco sofre.

**Causa Tecnica:** Polling (cliente pede updates periodicamente) escala linearmente com o numero de jogadores. Cada poll = 1 SELECT no MySQL. Com muitos jogadores, isso vira gargalo.

**Solucao — Push server-driven com cache local:**

Em vez de cada jogador fazer polling, o server ENVIA updates via TriggerClientEvent broadcast quando algo muda:

```javascript
// Server mantém cache do lobby em memoria (nao no MySQL)
let lobbyCache = [];
let lobbyCacheTimestamp = 0;

function atualizarLobbyCache() {
  exports['oxmysql:execute'](
    `SELECT g.id, g.player1_id, g.player1_side, g.bet_amount, g.status, g.created_at
     FROM casino_coinflip_games g
     WHERE g.status = 'waiting'
     ORDER BY g.created_at DESC
     LIMIT 50`,
    [],
    (salas) => {
      lobbyCache = salas;
      lobbyCacheTimestamp = Date.now();
    }
  );
}

// Atualizar cache a cada 5s (1 query, nao N queries)
setInterval(atualizarLobbyCache, 5000);

// Quando sala cria/entra/cancela/expira: atualizar cache + broadcast
function broadcastLobby() {
  atualizarLobbyCache();
  // Enviar para TODOS os jogadores que estao no NUI do coinflip
  emitNet('coinflip:lobbyUpdate', -1, lobbyCache);
}

// Handler de lobby: retorna cache em memoria (ZERO MySQL query)
onNet('coinflip:lobby', (cb) => {
  cb(lobbyCache);
});
```

**Impacto:** Com cache: 1 query MySQL a cada 5s (independente do numero de jogadores) + broadcasts instantaneos quando algo muda. Sem cache: N queries/3s onde N = jogadores online. Reducao de ~99% da carga no MySQL para o lobby.

## Problema 5: Provably Fair PvP — geracao de client seeds simultanea

**Sintoma:** No modelo PvP, ambos jogadores precisam ter client seeds ANTES do flip. Mas o Jogador 2 entra na sala e o flip inicia imediatamente — nao ha tempo para trocar seeds manualmente.

**Causa Tecnica:** No modelo solo (Stake), o jogador define o client seed antes de apostar. No PvP, o Jogador 2 entra e o flip ja comeca. Se o server esperar o Jogador 2 definir seed, cria delay inaceitavel. Se usar seed automatico, o jogador pode reclamar que nao escolheu.

**Solucao — Client seed automatico com opcao de customizacao pre-sessao:**

```javascript
// No NUI (React): gerar client seed automatico no mount do componente
function gerarClientSeed() {
  const array = new Uint8Array(16);
  crypto.getRandomValues(array);
  return Array.from(array, byte => byte.toString(16).padStart(2, '0')).join('');
}

// Componente CoinflipGame.tsx
const [clientSeed, setClientSeed] = useState(() => gerarClientSeed());

// O jogador pode customizar o seed A QUALQUER MOMENTO via modal Settings
// Mas o seed ativo no momento da entrada na sala e o que sera usado

// Ao criar sala: enviar clientSeed junto
fetchNui('coinflip:create', { amount, side, clientSeed });

// Ao entrar na sala: enviar clientSeed junto
fetchNui('coinflip:join', { roomId, clientSeed });

// Server recebe e armazena ambos seeds na partida
// UPDATE casino_coinflip_games SET client_seed_p1 = ?, client_seed_p2 = ?
```

**Impacto:** Zero delay na entrada. Seed automatico garante que SEMPRE existe um client seed (jogador nao pode alegar "nao tinha seed"). Modal de customizacao disponivel para jogadores que querem controlar — mas e OPCIONAL, nao bloqueia o fluxo.

### Queries Admin Uteis

```sql
-- Listar salas ativas (lobby)
SELECT id, player1_id, player1_side, bet_amount, created_at
FROM casino_coinflip_games WHERE status = 'waiting' ORDER BY created_at DESC;

-- Revenue por periodo (rake acumulado)
SELECT DATE(completed_at) as dia, SUM(rake_amount) as rake_total,
       COUNT(*) as partidas, SUM(pot_total) as volume_total
FROM casino_coinflip_games WHERE status = 'completed'
  AND completed_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY DATE(completed_at) ORDER BY dia DESC;

-- Leaderboard (top 20 por profit)
SELECT player_id, COUNT(*) as partidas,
  SUM(CASE WHEN winner_id = player_id THEN 1 ELSE 0 END) as wins,
  SUM(CASE WHEN winner_id = player_id THEN payout_amount - bet_amount ELSE -bet_amount END) as profit
FROM (
  SELECT player1_id as player_id, winner_id, payout_amount, bet_amount
  FROM casino_coinflip_games WHERE status = 'completed'
  UNION ALL
  SELECT player2_id, winner_id, payout_amount, bet_amount
  FROM casino_coinflip_games WHERE status = 'completed'
) t GROUP BY player_id ORDER BY profit DESC LIMIT 20;

-- Verificar fairness (distribuicao cara vs coroa)
SELECT result, COUNT(*) as total,
  ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM casino_coinflip_games WHERE status='completed'), 2) as pct
FROM casino_coinflip_games WHERE status = 'completed' GROUP BY result;
```

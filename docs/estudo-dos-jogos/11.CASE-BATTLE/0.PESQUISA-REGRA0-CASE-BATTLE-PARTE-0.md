## O que a pesquisa trouxe de concreto

### 1.1 Mecânica Fundamental
Case Battle é uma competição PvP onde 2-4 jogadores abrem caixas idênticas simultaneamente. O jogador cujos itens totalizam o maior valor vence e leva TODOS os itens dos outros jogadores. A mecânica central é:

1. **Criação:** Um jogador seleciona 1-50 caixas (rounds) e define o formato (1v1, 2v2, 1v1v1v1)
2. **Lobby:** A batalha aparece na lista pública. Outros jogadores pagam a mesma entry fee para entrar
3. **Execução:** Quando todos os slots são preenchidos, todas as caixas são abertas SIMULTANEAMENTE para todos
4. **Revelação:** Os itens são revelados round por round, com animação de reveal strip (carrossel horizontal parando no item)
5. **Resultado:** O jogador com maior valor total GANHA. Em empate, um tiebreaker (roll Provably Fair) decide

### 2.1 Reveal Strip — Implementação
A animação de reveal (carrossel parando no item) pode ser feita com:

**Opção A: CSS translateX com transition** (mais simples, boa performance)
```css
/* Strip com N itens lado a lado, translate para posição do item sorteado */
.reveal-strip {
  display: flex;
  transform: translateX(-${targetPosition}px);
  transition: transform 4s cubic-bezier(0.15, 0.85, 0.35, 1); /* ease-wheel do DS */
}
```

**Opção B: GSAP timeline** (mais controle, som sincronizado)
```javascript
gsap.to(stripRef.current, {
  x: -targetPosition,
  duration: 4,
  ease: "power3.out",
  onUpdate: () => { /* tick sound em cada item passando */ },
  onComplete: () => { /* highlight do item + som de resultado */ }
});
```

**Recomendação para o projeto:** Opção B (GSAP) — permite sincronizar som de "tick" a cada item que passa, ease customizado `--ease-wheel` do DS, e controle preciso do ponto de parada.

## Algoritmo Padrão (2025/2026)
```
1. SERVER SEED: string aleatória gerada pelo servidor (ex: crypto.randomBytes(32))
2. SERVER SEED HASH: SHA256(serverSeed) → mostrado ao jogador ANTES da rodada
3. CLIENT SEED: string definida pelo jogador (ou gerada pelo browser)
4. NONCE: contador incremental por jogador (evita repetição)
5. RESULTADO: HMAC_SHA256(serverSeed, `${clientSeed}-${nonce}`) → primeiro 8 chars hex → decimal → modulo pelo range

Após a rodada:
6. REVELAR serverSeed (jogador pode verificar que SHA256 bate com o hash mostrado antes)
7. Jogador pode REPRODUZIR o resultado usando os 4 valores
```

**O que JÁ existe no backend:**
O handler 10 em `cases.js` usa:
```javascript
const hash = crypto.createHash('sha256')
  .update(`${SERVER_SEED}:${userId}:${caseId}:${nonce}:battle:${battleId}`)
  .digest('hex');
const roll = (parseInt(hash.substring(0, 8), 16) / 0xFFFFFFFF) * 100;
```

**O que FALTA (para Provably Fair real):**
- Permitir o jogador definir seu client seed
- Mostrar o hash do server seed ANTES da rodada
- Revelar o server seed APÓS a rodada
- Modal de verificação com código reproduzível
- Nonce por jogador (atualmente usa `Date.now()` que não é determinístico)

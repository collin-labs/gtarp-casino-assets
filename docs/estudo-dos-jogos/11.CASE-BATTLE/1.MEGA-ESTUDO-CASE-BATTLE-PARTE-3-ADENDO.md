# =============================================
# A1. PROBLEMA 1 — STRIP ANIMATION PERFORMANCE NO NUI
# =============================================

## Sintoma

No formato 4-way FFA (1v1v1v1), o CaseBattleGame renderiza 4 colunas
simultaneas, cada uma com um BattleRevealStrip animando via CSS
`transform: translateX()`. A animacao de cada strip move ~40 item cards
ao longo de 3.5-5 segundos com easing curve. Quando os 4 strips animam
ao mesmo tempo, o NUI (CEF Chromium embarcado no FiveM) precisa processar
4 translateX independentes por frame, competindo com a GPU que ja esta
renderizando o GTA V em 30-60fps.

## Causa tecnica

CSS `transform: translateX()` e GPU-accelerated — o compositor do Chromium
cria uma layer separada para cada elemento com `transform` e a GPU move
essa layer sem reflow. Porem, no NUI do FiveM, a GPU e COMPARTILHADA com
o GTA. Cada layer adicional consome VRAM e bandwidth do compositor.

**Dados concretos medidos na pesquisa (Eixo 2, §3.3.3):**
- 1 strip translateX: ~2-4ms por frame (dentro do budget de 16ms para 60fps)
- 4 strips simultaneos: ~8-12ms por frame (margem apertada)
- Layout recalculation ao atualizar valor acumulado: +1-2ms por update
- Total estimado 4-way: ~10-14ms por frame durante reveal

Se o GTA estiver sob pressao (cidade cheia, explosoes, muitos jogadores),
o frame budget restante para o NUI pode cair de 16ms para 8-10ms,
causando frame drops visiveis na animacao do strip.

## Solucao multi-camada

**Camada 1 — will-change antes da animacao:**
Aplicar `will-change: transform` em cada strip container ANTES de iniciar
a animacao. Isso forca o compositor a criar a layer antecipadamente,
evitando o custo de layer creation durante a animacao. Remover `will-change`
APOS a animacao completar (evita consumo permanente de VRAM).

**Camada 2 — Deteccao de performance com fallback:**
```javascript
// battlePerformance.ts — monitor de FPS para Case Battle
let lastFrameTime = performance.now();
let lowFpsFrames = 0;

export function checkBattlePerformance(): boolean {
  const now = performance.now();
  const delta = now - lastFrameTime;
  lastFrameTime = now;

  // Abaixo de 30fps = frame > 33ms
  if (delta > 33) lowFpsFrames++;
  else lowFpsFrames = Math.max(0, lowFpsFrames - 1);

  // Se 5+ frames seguidos abaixo de 30fps = performance ruim
  return lowFpsFrames < 5;
}

// Uso no CaseBattleGame
const performanceOk = checkBattlePerformance();
const stripDuration = performanceOk ? 4000 : 2500; // ms — mais rapido se lento
const stripEasing = performanceOk
  ? [0.15, 0.85, 0.35, 1.0]  // deceleration longa e suave
  : [0.25, 1.0, 0.5, 1.0];   // deceleration curta (menos frames)
```

**Camada 3 — Fallback visual para 6-player (V2):**
Se no futuro implementar 6-way FFA, NAO renderizar 6 strips simultaneos.
Em vez disso: renderizar 2 strips visiveis por vez com navegacao horizontal
(swipe ou setas) para ver os outros jogadores. Isso mantem o budget em
~4-8ms/frame mesmo com 6 jogadores.

**Impacto:** ~25 linhas de performance monitor. Zero dependencia extra.
Funciona transparentemente — jogador nem percebe que o sistema adaptou.

# =============================================
# A2. PROBLEMA 2 — PROVABLY FAIR SEM PRE-COMMITMENT
# =============================================

## Sintoma

O handler `casino:cases:battle_join` em `server/handlers/cases.js`
(linha 496-498) gera o resultado via SHA-256, mas o hash do server seed
NAO e exibido ao jogador ANTES da batalha. O jogador nao tem como
verificar que o resultado nao foi manipulado apos a aposta.

## Codigo REAL atual com o problema (cases.js L496-500)

```javascript
// ATUAL — server/handlers/cases.js linha 496-500
const nonce = Date.now() + ci * 1000 + player.user_id;
const hash = crypto.createHash('sha256')
  .update(`${SERVER_SEED}:${player.user_id}:${csId}:${nonce}:battle:${battleId}`)
  .digest('hex');
const roll = (parseInt(hash.substring(0, 8), 16) / 0xFFFFFFFF) * 100;
```

## 5 problemas especificos neste codigo

**Problema 2a — SERVER_SEED e GLOBAL (nao por batalha).**
O `SERVER_SEED` e definido uma unica vez no topo do handler (linha 8):
`const SERVER_SEED = crypto.randomBytes(32).toString('hex');`
Se este seed fosse revelado (para verificacao de UMA batalha), TODAS as
batalhas do servidor ficariam comprometidas. O padrao CSGOEmpire gera
um seed NOVO para CADA batalha.

**Problema 2b — Nonce usa Date.now() que NAO e deterministico.**
`Date.now()` muda a cada milissegundo. Se o jogador tentar verificar
o resultado 1 segundo depois, o nonce sera diferente e o resultado
nao sera reproduzivel. O padrao e usar nonce FIXO: roundNumber + playerPosition.

**Problema 2c — Sem pre-commitment (hash nao exibido antes).**
O jogador NUNCA ve o hash do server seed antes da batalha. O servidor
PODE trocar o seed apos ver os client seeds e gerar um resultado
desfavoravel. A pesquisa no GitHub (provably_fixed) demonstrou esta
vulnerabilidade com scripts executaveis.

**Problema 2d — Sem revelacao do seed apos batalha.**
Nao existe endpoint nem logica para revelar o server seed ao jogador
apos a batalha. Sem revelacao, nao existe verificacao possivel.

**Problema 2e — Sem client seed do jogador.**
O jogador nao fornece seu proprio seed. Toda a aleatoriedade vem do
servidor, eliminando o principio fundamental do Provably Fair: o jogador
CONTRIBUI para a aleatoriedade, impedindo manipulacao unilateral.

## Solucao: adaptacao do padrao CSGOEmpire (pesquisa REGRA 0)

A pesquisa encontrou o codigo fonte REAL que CSGOEmpire publica para
verificacao de Case Battles (help.csgoempire.com/en/articles/9030391).
Adaptacao para JS do Blackout Casino:

```javascript
// caseBattleFairness.js — adaptacao do padrao CSGOEmpire
const crypto = require('crypto');

const MAX_HEX_SEGMENTS = 6;
const HEX_SEGMENT_SIZE = 2;
const BASE = 256;

// Gerar seed POR BATALHA (nao global)
function generateBattleSeed() {
  const seed = crypto.randomBytes(16).toString('hex'); // 32 chars
  const seedHash = crypto.createHash('sha256').update(seed).digest('hex');
  return { seed, seedHash };
}

// Converter hash em valor decimal 0.0-1.0 (padrao CSGOEmpire)
function calculateDecimalValue(preResult) {
  let decimalValue = 0;
  for (let i = 0; i < MAX_HEX_SEGMENTS; i++) {
    const hexValue = preResult.substring(HEX_SEGMENT_SIZE * i, HEX_SEGMENT_SIZE * (i + 1));
    decimalValue += parseInt(hexValue, 16) / Math.pow(BASE, i + 1);
  }
  return decimalValue;
}

// Resultado PF para um jogador em um round
function getBattleResult(serverSeed, combinedSeed, roundNumber, playerPosition, ticketQty) {
  const stringToHash = `${combinedSeed}-${roundNumber}-${playerPosition}`;
  const preResult = crypto.createHmac('sha256', serverSeed)
    .update(stringToHash).digest('hex');
  const decimal = calculateDecimalValue(preResult);
  const ticket = Math.floor(decimal * ticketQty) + 1;
  return { preResult, ticket };
}

// Tiebreaker: sem playerPosition (padrao CSGOEmpire)
function getTiebreakerResult(serverSeed, combinedSeed, totalRounds, numPlayers) {
  const roundNumber = totalRounds + 1;
  const stringToHash = `${combinedSeed}-${roundNumber}`;
  const preResult = crypto.createHmac('sha256', serverSeed)
    .update(stringToHash).digest('hex');
  const decimal = calculateDecimalValue(preResult);
  return Math.floor(decimal * numPlayers) + 1;
}

module.exports = { generateBattleSeed, getBattleResult, getTiebreakerResult };
```

**O que muda no handler battle_join:**
1. Ao CRIAR batalha: `generateBattleSeed()` → armazena `seed` e `seedHash` na tabela
2. Ao jogador ENTRAR: envia `seedHash` via callback (pre-commitment)
3. Ao INICIAR: usa `getBattleResult()` com `combinedSeed` = sorted(battle_id + player_ids)
4. Ao FINALIZAR: inclui `seed` (revelado) no resultado enviado ao NUI
5. Modal de verificacao: jogador insere seeds → NUI calcula localmente → compara

**Impacto:** ~60 linhas de modulo novo + ~30 linhas de edicao no handler existente.
Corrige uma vulnerabilidade de seguranca REAL que a pesquisa identificou.

# A4. SQL EXECUTAVEL — ALTERs + SEEDS + QUERIES ADMIN
# =============================================

```sql
-- =============================================
-- BLACKOUT CASINO — CASE BATTLE #11 — UPGRADE SQL
-- Complemento ao casino_cases_schema.sql existente
-- Copiar-colar executavel no MySQL/MariaDB
-- =============================================

-- ALTER 1: Novos campos na tabela battles
-- Campos descobertos como necessarios na pesquisa REGRA 0:
-- battle_mode (4 modos), battle_type (ffa/team), invite_code (private),
-- server_seed_hash/server_seed (Provably Fair pre-commitment),
-- expires_at (auto-cancel timeout)
ALTER TABLE casino_cases_battles
  ADD COLUMN battle_mode ENUM('regular','crazy','terminal','private')
    NOT NULL DEFAULT 'regular'
    COMMENT 'Modo de jogo: regular=maior ganha, crazy=menor ganha, terminal=ultimo round decide'
    AFTER max_players,
  ADD COLUMN battle_type ENUM('ffa','team')
    NOT NULL DEFAULT 'ffa'
    COMMENT 'Tipo: ffa=cada um por si, team=times (2v2)'
    AFTER battle_mode,
  ADD COLUMN invite_code VARCHAR(8) DEFAULT NULL
    COMMENT 'Codigo de convite para batalhas privadas. NULL=publica'
    AFTER battle_type,
  ADD COLUMN server_seed_hash VARCHAR(64) DEFAULT NULL
    COMMENT 'SHA256 do server seed — exibido ANTES da batalha (pre-commitment PF)'
    AFTER house_fee_pct,
  ADD COLUMN server_seed VARCHAR(64) DEFAULT NULL
    COMMENT 'Server seed revelado APOS a batalha (verificacao PF)'
    AFTER server_seed_hash,
  ADD COLUMN combined_seed VARCHAR(128) DEFAULT NULL
    COMMENT 'Seeds combinados dos jogadores (sorted player_ids + battle_id)'
    AFTER server_seed,
  ADD COLUMN expires_at TIMESTAMP DEFAULT NULL
    COMMENT 'Auto-cancelar se nao lotar antes deste timestamp'
    AFTER finished_at,
  ADD INDEX idx_invite (invite_code),
  ADD INDEX idx_mode (battle_mode),
  ADD INDEX idx_expires (expires_at);

-- ALTER 2: Detalhes de itens por jogador
ALTER TABLE casino_cases_battle_players
  ADD COLUMN items_detail JSON DEFAULT NULL
    COMMENT 'Array completo: [{caseIndex, itemId, name, rarity, rarity_color, value}]'
    AFTER items_won;

-- SEEDS: Batalhas de teste para desenvolvimento
INSERT INTO casino_cases_battles
  (creator_id, status, max_players, battle_mode, battle_type,
   case_sequence, entry_fee, total_pot, house_fee_pct)
VALUES
  (1, 'waiting', 2, 'regular', 'ffa',
   '[1]', 500.00, 500.00, 5.00),
  (1, 'waiting', 4, 'crazy', 'ffa',
   '[1,2]', 1500.00, 1500.00, 5.00),
  (1, 'waiting', 2, 'terminal', 'ffa',
   '[1,2,3]', 600.00, 600.00, 5.00),
  (1, 'finished', 2, 'regular', 'ffa',
   '[1]', 500.00, 1000.00, 5.00);

-- Adicionar criador como jogador nas batalhas de teste
INSERT INTO casino_cases_battle_players (battle_id, user_id, total_value, rank)
VALUES
  (LAST_INSERT_ID() - 3, 1, 0, 0),
  (LAST_INSERT_ID(), 1, 850, 2);

-- QUERIES ADMIN UTEIS

-- Listar batalhas por status com contagem de jogadores
SELECT b.id, b.battle_mode, b.battle_type, b.status, b.entry_fee, b.total_pot,
       b.max_players, COUNT(bp.id) as current_players,
       b.created_at, b.finished_at
FROM casino_cases_battles b
LEFT JOIN casino_cases_battle_players bp ON bp.battle_id = b.id
GROUP BY b.id
ORDER BY b.created_at DESC
LIMIT 50;

-- Revenue por periodo (house fee coletada)
SELECT DATE(finished_at) as dia,
       COUNT(*) as batalhas,
       SUM(total_pot) as pot_total,
       SUM(total_pot * house_fee_pct / 100) as revenue_casino,
       AVG(total_pot) as pot_medio
FROM casino_cases_battles
WHERE status = 'finished'
  AND finished_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY DATE(finished_at)
ORDER BY dia DESC;

-- Win rate por jogador (top 10)
SELECT bp.user_id,
       COUNT(*) as batalhas_jogadas,
       SUM(CASE WHEN bp.rank = 1 THEN 1 ELSE 0 END) as vitorias,
       ROUND(SUM(CASE WHEN bp.rank = 1 THEN 1 ELSE 0 END) / COUNT(*) * 100, 1) as win_rate_pct,
       SUM(bp.total_value) as valor_total_aberto
FROM casino_cases_battle_players bp
JOIN casino_cases_battles b ON b.id = bp.battle_id AND b.status = 'finished'
GROUP BY bp.user_id
HAVING batalhas_jogadas >= 5
ORDER BY win_rate_pct DESC
LIMIT 10;

-- Batalhas expiradas para auto-cancelar (rodar em cron/interval)
UPDATE casino_cases_battles
SET status = 'cancelled'
WHERE status = 'waiting'
  AND expires_at IS NOT NULL
  AND expires_at < NOW();
```

# =============================================
# A7. NOTA: GSAP → FRAMER MOTION 11
# =============================================

O Mega Estudo antigo (secao 6.3, linhas 110-157) contem pseudocodigo GSAP
para a animacao do RevealStrip:

```javascript
// ANTIGO (secao 6.3 do Mega) — GSAP (NAO USAR)
gsap.to(stripRef.current, {
  x: targetX,
  duration: 4,
  ease: "power4.out",
  onComplete: () => { /* reveal item */ }
});
```

**O projeto NAO usa GSAP.** O Blackout Casino usa Framer Motion 11 em
TODOS os componentes. A pesquisa REGRA 0 (Eixo 2, §3.1) confirmou que
FM11 e suficiente para strip animation via `motion.div` com `animate`
e `transition.ease` customizado.

**Equivalencia GSAP → FM11:**

| GSAP | Framer Motion 11 |
|------|-----------------|
| `gsap.to(ref, { x, duration, ease: "power4.out" })` | `<motion.div animate={{ x }} transition={{ duration, ease: [0.15, 0.85, 0.35, 1.0] }} />` |
| `gsap.timeline().to().to()` | Sequencia via `useAnimation().start()` com `await` |
| `onComplete` callback | `onAnimationComplete` prop |

**Acao no Roteiro:** Fase 2 (BattleRevealStrip) implementara a animacao
inteiramente com FM11, sem GSAP, sem dependencia nova.

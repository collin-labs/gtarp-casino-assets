# EIXO SOUND DESIGN — IMPLEMENTACAO TECNICA DO OSCILLATOR

## Arquitetura do Tom Ascendente

O som mais icónico do Crash e o tom que sobe conforme o multiplicador cresce.
A implementacao usa Web Audio API nativa (sem dependencias externas):

```
Arquitetura:
AudioContext → OscillatorNode → GainNode → AudioContext.destination

Parametros por tier:
  Tier 1 (1-1.99x):  frequency 220→440Hz,  type 'sine',     gain 0.15
  Tier 2 (2-4.99x):  frequency 440→660Hz,  type 'sine',     gain 0.20
  Tier 3 (5-9.99x):  frequency 660→880Hz,  type 'triangle', gain 0.25
  Tier 4 (10-99.99x): frequency 880→1320Hz, type 'triangle', gain 0.30
  Tier 5 (100x+):    frequency 1320→1760Hz, type 'sawtooth', gain 0.35
```

A frequencia nao sobe linearmente — usa `frequency.exponentialRampToValueAtTime()`
que cria a sensacao de aceleracao. O oscillator e recriado a cada mudanca de tier
(nao reutilizado, conforme best practice MDN — oscillators sao baratos).

## Anti-Click (fade in/out)

Problema: cortar um oscillator abruptamente causa "click" audivel.
Solucao documentada na pesquisa: `gainNode.gain.setTargetAtTime(0, ctx.currentTime, 0.015)`
com decay de 15ms — imperceptivel mas elimina o artefato.

No Crash: quando o jogador faz cashout, o tom nao para instantaneamente — faz fade-out
de 50ms (mais lento que o padrao 15ms para dar sensacao de "resolucao"). Quando o
crash acontece, o fade-out e de 5ms (quase instantaneo, seguido pelo som de explosao).

# ANTI-ABUSE — PROTECAO ESPECIFICA PARA CRASH

## Modelo de Ameaca

O Crash e vulneravel a 3 tipos de abuso:

**1. Bot de auto-cashout rapido:**
Programa que faz cashout automatico em multiplicadores muito baixos (1.01x-1.05x)
repetidamente. Cada cashout gera lucro minimo mas com frequencia alta, acumula.
Mitigacao: rate limit 1 bet por rodada, cooldown minimo 3s entre rounds, e
monitoramento server-side: se jogador faz cashout <1.10x em mais de 80% dos
ultimos 50 rounds, flag para revisao admin.

**2. Collusao multi-conta:**
Jogador cria 2+ contas e aposta valores diferentes com auto-cashout em
multiplicadores diferentes para cobrir mais cenarios.
Mitigacao no FiveM: identificador unico por license (Rockstar Social Club).
Cada license so pode ter 1 conta no cassino. HWID como camada extra se o
servidor usar sistema de whitelist.

**3. Timing exploit (latencia artificial):**
Jogador injeta latencia artificial para ver o crash point antes de enviar cashout.
Mitigacao: timestamp server em cada cashout. Se delta entre round_crash_time e
cashout_request_time for negativo (jogador "viu" o futuro), rejeitar cashout.
Delta max: 200ms (documentado em Mega P5-ADENDO).

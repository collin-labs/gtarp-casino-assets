# 3. EIXO 2 — TECNOLOGIA DISPONIVEL (2025-2026)
# =============================================================

O Crash tem 4 componentes tecnicos distintos: (A) a curva ascendente animada, (B) o sistema de multiplicador e timing, (C) o Provably Fair, e (D) o sistema social (feed de apostas, historico). A pesquisa revelou abordagens concretas para cada um.

## 3.1 Componente A — Curva Ascendente (Canvas 2D)

A curva do Crash e o elemento visual central. A pesquisa identificou 3 tecnicas para renderiza-la.

### Opcao A1 — Canvas 2D com requestAnimationFrame (Recomendada)

**Fontes:** MDN Canvas Animation, Bustabit open-source, repositorios GitHub crash-game

**Como funciona:**
- Canvas HTML5 ocupa a area central do jogo
- A cada frame (requestAnimationFrame), calcula-se o ponto atual da curva com base no tempo decorrido
- A curva e desenhada usando ctx.lineTo() conectando pontos do inicio ate o ponto atual
- O multiplicador e calculado como funcao exponencial do tempo: `multiplier = Math.pow(e, elapsed * rate)`
- A curva e redesenhada a cada frame (clearRect + redraw) com trail/gradient
- Grid de fundo com linhas sutis para dar referencia visual de eixos X/Y
- Multiplicador central renderizado sobre o canvas (ou em div posicionada acima)

**Tecnica da curva exponencial:**
```
// Calculo do multiplicador baseado em tempo
const GROWTH_RATE = 0.00006; // controla velocidade da subida
const elapsed = Date.now() - startTime;
const multiplier = Math.pow(Math.E, GROWTH_RATE * elapsed);

// Posicao no canvas
const x = (elapsed / totalDuration) * canvasWidth;
const y = canvasHeight - (Math.log(multiplier) / Math.log(maxMultiplier)) * canvasHeight;
```

**Performance no NUI FiveM:**
- Canvas 2D usa GPU acelerada via Chromium (CEF)
- requestAnimationFrame sincroniza com refresh rate
- Sem dependencias externas — API nativa do browser
- Impacto de FPS: minimo (<2%) — confirmado por scripts FiveM existentes que usam Canvas para roleta, slots
- Clearance: canvas simples (1 curva + grid) e muito mais leve que Canvas com fisica (Matter.js do Plinko)

**Pros:**
- Performance maxima no NUI (zero dependencias, GPU acelerada)
- Controle total sobre visual (cor da curva, grossura, glow, gradiente)
- Bem documentado (MDN, centenas de tutoriais)
- Ja validado no projeto: RouletteWheel.tsx usa Canvas 2D para a roda (288 linhas)
- Funciona em qualquer resolucao via proporcao do canvas

**Contras:**
- Precisa de logica manual para grid de fundo, labels de eixos, styling
- Canvas nao tem hover nativo em elementos individuais (nao e problema para o Crash pois a curva nao e interativa)

**Veredicto:** MELHOR OPCAO. Canvas 2D ja esta validado no projeto (RouletteWheel.tsx) e e a mesma tecnica que Stake.com e Bustabit usam. Zero risco tecnico.

### Opcao A2 — SVG com Path Animado

**Como funciona:**
- SVG path `<path d="M0,h L...">` redesenhado a cada frame via React state
- O d attribute e recalculado com os pontos da curva

**Pros:**
- SVG escala perfeitamente em qualquer resolucao
- Pode usar CSS para estilizar stroke (cor, grossura, dasharray)
- DOM-based — mais facil de integrar com Framer Motion

**Contras:**
- SVG redesenhado com muitos pontos pode ser mais lento que Canvas
- Para curvas longas (muitos pontos), o path string fica pesado
- Nao e o padrao do mercado para crash games (todos usam Canvas)

**Veredicto:** Viavel mas inferior ao Canvas 2D. Nao recomendado.

### Opcao A3 — CSS/DOM Puro (Div com Transform)

**Como funciona:**
- Uma div com width crescente e background gradiente simula a curva
- Transform: scaleX() animado

**Contras:**
- Nao consegue curva exponencial real (so linear ou pre-definida)
- Visual limitado
- Nenhum crash game real usa esse approach

**Veredicto:** Descartado. Nao viavel para a curva exponencial do Crash.

## 3.2 Componente B — Sistema de Multiplicador e Timing

**Mecanica core do multiplicador (baseada na pesquisa do Stake/Bustabit/Aviator):**

O crash point e determinado ANTES do round comecar, usando o sistema Provably Fair. O multiplicador visual sobe no client ate atingir o crash point pre-determinado. O server controla quando o crash acontece — o client apenas anima.

**Fluxo de um round:**
1. Server gera crash point via Provably Fair (ex: 2.47x)
2. Server envia sinal de inicio para todos os clients (NUI)
3. Client inicia animacao: multiplicador comeca em 1.00x, sobe exponencialmente
4. Jogador clica "Cashout" — client envia request ao server
5. Server valida: se multiplicador atual < crash point, cashout aceito. Senao, perdeu.
6. Quando multiplicador atinge crash point, server envia sinal de crash
7. Client mostra animacao de crash (curva para, tela vermelha, som de explosao)
8. 5 segundos de pausa, proximo round comeca

**Timing critico:** O multiplicador no client e no server DEVEM estar sincronizados. A pesquisa do Stake mostra que o multiplicador e funcao pura do tempo (sem randomness no crescimento — so o crash point e aleatorio). Isso significa:
- Client e server usam mesma formula: `multiplier = f(elapsed_time)`
- Se o jogador clica cashout no tempo T, o server calcula o multiplicador no tempo T e valida
- Latencia de rede pode causar discrepancia — solucao: janela de tolerancia de ~200ms

## 3.3 Componente C — Provably Fair (Implementacao para FiveM)

A pesquisa do Stake.com revelou o codigo REAL da implementacao Provably Fair:

**Algoritmo Stake.com (fonte oficial: stake.com/provably-fair/implementation):**
- Funcao byteGenerator recebe: serverSeed, clientSeed, nonce, cursor
- Usa HMAC_SHA256: `createHmac('sha256', serverSeed).update(clientSeed:nonce:currentRound)`
- Buffer de 32 bytes gerado
- 4 bytes por resultado (cursor controla qual quarteto de bytes usar)
- Para Crash: 1 resultado necessario = 4 bytes = cursor 0

**Adaptacao para FiveM (server handler JS com oxmysql):**
- Server gera serverSeed: `crypto.randomBytes(32).toString('hex')`
- Hash do serverSeed enviado ao NUI antes do round (compromisso)
- clientSeed gerado no NUI do jogador (Math.random ou crypto.getRandomValues)
- Nonce incrementa a cada round
- Crash point calculado: 4 bytes do HMAC -> float entre 0 e 1 -> convertido em multiplicador
- Formula de conversao: `crashPoint = Math.max(1, Math.floor(100 / (rawFloat * 100)) / 100)`
- House edge aplicado: se rawFloat < houseEdge (0.01 para 1%), crash imediato em 1.00x

**Diferencial Blackout Casino:** Os scripts FiveM existentes NAO implementam Provably Fair. Usam Math.random() simples no server — sem verificacao possivel. O Blackout vai implementar PF real com modal de verificacao, elevando a credibilidade.

## 3.4 Componente D — Sistema Social (Feed de Apostas + Historico)

**Feed de apostas live:**
- Lista lateral (ou inferior) mostrando: nome do jogador, valor apostado, status (apostou / sacou em Xx / perdeu)
- Atualizado em tempo real via NUI events (SendNUIMessage)
- Em FiveM, o server broadcast a lista de apostas para todos os NUIs no round

**Historico de rounds:**
- Barra horizontal no topo mostrando os ultimos 20-30 resultados
- Cada resultado e um badge com o multiplicador e cor (verde se alto, vermelho se baixo)
- Persistido no MySQL: tabela casino_crash_rounds (id, crash_point, hash, server_seed, timestamp)

**Tecnologia no FiveM:**
- Server handler JS: RegisterNuiCallback para apostas, cashout, historico
- Client Lua: RegisterNetEvent para receber broadcasts (novo round, crash, apostas de outros)
- NUI React: useEffect com listener de mensagens do FiveM para atualizar estado

## 4.2 Limitacoes do NUI/CEF no FiveM

| Limitacao | Impacto no Crash | Solucao |
|-----------|-----------------|---------|
| GPU compartilhada com GTA | Canvas 2D simples nao impacta (<2% FPS) | Canvas sem WebGL, sem shader pesado |
| Latencia NUI <-> Server | Cashout pode ter delay de 50-200ms | Janela de tolerancia no server. Timestamp do click enviado junto |
| Sem WebSocket nativo | Nao pode usar Socket.IO para real-time | Usar SendNUIMessage + RegisterNetEvent (nativos do FiveM, zero latencia extra) |
| Resolucao variavel | Canvas precisa se adaptar | Canvas dimensionado via container pai com ResizeObserver |
| Chromium version antiga | Algumas APIs modernas podem nao existir | Canvas 2D e HMAC_SHA256 existem em todas as versoes de CEF usadas no FiveM |

## 5.1 Referencias Visuais Encontradas

A pesquisa no Dribbble encontrou 100+ designs de crash game. Os padroes visuais dominantes em 2025-2026 sao:

### Padrao Visual 1 — Curva Neon em Fundo Escuro (Stake-style)
- Fundo: preto/cinza escuro (#0A0A0A a #1A1A1A) com grid sutil

- Multiplicador: fonte bold gigante (40-80px), branca ou dourada, com glow

- Fundo: gradiente azul escuro para preto
- Elemento: aviao/foguete com trail colorido

- Stars/nebula no fundo

### Padrao Visual 3 — Minimalista com Accent Color (Blaze-style)
- Fundo solido escuro
- Curva fina com cor unica
- Multiplicador central dominante
- Quase sem decoracao

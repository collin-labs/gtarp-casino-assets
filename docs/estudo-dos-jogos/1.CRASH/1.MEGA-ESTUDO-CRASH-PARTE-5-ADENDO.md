# PARTE 4B — PERFORMANCE BUDGET CANVAS NUI

## Budget de Renderizacao

O Crash roda Canvas 2D dentro do NUI Chromium 93 do FiveM, que compartilha
GPU com o GTA V. O budget precisa ser conservador para nao impactar FPS do jogo.

| Recurso | Budget max | Justificativa |
|---------|:----------:|---------------|
| Canvas redraws/frame | 1 | clearRect + drawPath + drawGrid + drawMultiplier = 1 ciclo |
| Pontos no path (curva) | 500 | Sliding window — apos 500, remove mais antigos |
| Particulas simultaneas | 30 | Cashout particles + explosion = 30 max (reciclar pool) |
| requestAnimationFrame | 60fps target, 30fps min | Se deltaTime > 33ms, pular frame de particulas |
| Tamanho Canvas | 100% do container | devicePixelRatio capped em 2 (retina nao necessario no NUI) |
| Memoria path | ~4KB por round | 500 pontos x 8 bytes (x,y float) = 4KB (garbage collected entre rounds) |
| Texto no Canvas | 2 por frame max | Multiplicador + countdown = 2 drawText. Historico via DOM |
| Gradients ativos | 3 max | Background radial + curva linear + glow radial |

**Fallback progressivo:**
Se performance detectada < 30fps por 3 frames consecutivos:
1. Reduzir path para 250 pontos (metade do budget)
2. Desligar particulas de cashout (manter apenas explosion)
3. Simplificar gradients (1 cor solida em vez de linear-gradient no path)
4. Se ainda < 30fps: desligar grid dourado (manter apenas curva e multiplicador)

## Metricas de Latencia Multiplayer

O Crash e o unico jogo do projeto onde latencia afeta gameplay diretamente
(cashout em 2.50x com 200ms de atraso pode significar crash em 2.48x):

| Etapa | Latencia esperada | Tolerancia |
|-------|:-----------------:|:----------:|
| NUI → Lua client | <5ms | N/A (local) |
| Lua client → JS server (TriggerServerEvent) | 10-50ms | N/A (rede interna) |
| JS server → MySQL (bet insert) | 5-15ms | N/A |
| Server broadcast (round start) | 10-50ms por jogador | Compensar com countdown sync |
| Cashout request → validation | 10-50ms | 200ms delta max (server timestamp) |
| Round inteiro (end-to-end) | 25-120ms | Se > 200ms, usar server value |

**Implicacao para o Roteiro:** O handler `casino:crash:cashout` deve salvar
`server_multiplier_at_request` e `client_multiplier_claimed`. Se a diferenca
for < 0.02x (2 centesimos), usar o client value (beneficio do jogador).
Se > 0.02x, usar server value.

# PARTE 6B — EDGE CASES DOCUMENTADOS

## EC-1: Crash Point 1.00x (Instant Crash)

O crash pode acontecer em 1.00x — o multiplicador nem sobe.
Probabilidade: ~4% (com house edge 4%). Todos os jogadores perdem.
- UI: Multiplicador pisca vermelho 1.00x, explosao imediata, sem curva
- Feed: Todos os bets ficam vermelhos
- Sound: Explosao grave sem o buildup do oscillator
- Comunicacao: "Crash em 1.00x eh parte do jogo — RNG verificavel via PF"

## EC-2: Desconexao durante Rising

Jogador desconecta enquanto multiplicador esta subindo e tem bet ativo.
- Se auto-cashout configurado: server executa o auto-cashout normalmente
- Se manual (sem auto-cashout): server NAO faz cashout automatico — bet e perdido
- Razao: impedir abuse de "desconectar = cashout gratis"
- Mitigacao: aviso no UI "Configure auto-cashout como seguranca"

## EC-3: Round sem Nenhuma Aposta

Nenhum jogador aposta antes do countdown terminar.
- Server: pula o round (nao processa hash chain, nao gasta nonce)
- UI: mostra "Sem apostas nesta rodada" e inicia countdown do proximo
- Nao consome hash da chain (economia de hashes)

## EC-4: Aposta Maxima Atingida

Jogador tenta apostar mais que o max configurado pelo admin.
- UI: input fica vermelho, tooltip "Maximo: X GCoin"
- API: rejeita com 400, mensagem clara
- Admin config: `max_bet` na tabela casino_crash_config

## EC-5: Multiplicador Extremo (>1000x)

Raro mas possivel — o multiplicador passa de 1000x.
- UI: Canvas continua desenhando, path comprimido horizontalmente
- Tier 5 visual ativo (roxo + particulas)
- Feed: aplausos automaticos para quem ainda esta no round
- Limitacao: o Canvas usa coordenadas float — acima de 10.000x, truncar display
  para "10.000x+" (curva continua internamente)

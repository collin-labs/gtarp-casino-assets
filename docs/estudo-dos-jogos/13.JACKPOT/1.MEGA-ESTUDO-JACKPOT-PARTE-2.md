# PARTE 4 — PROBLEMAS E LIMITAÇÕES
# ═══════════════════════════════════════════════════

## Problema 1: Sincronização de Timer entre Múltiplos Clients

**Sintoma:** Jogador A vê timer em 5s, Jogador B vê timer em 8s. Jogador B deposita e A acha que deveria estar fechado.

**Causa:** Timer rodando client-side com latência de rede variável (50-200ms FiveM). Cada client inicia o countdown em momentos ligeiramente diferentes.

**Solução:**
- Timer é **server-side authoritative**. O handler mantém o timestamp de fechamento (`closeAt = Date.now() + 60000`).
- A cada broadcast de update, enviar `timeRemaining = closeAt - Date.now()`.
- Client ajusta seu countdown local baseado no `timeRemaining` recebido.
- O CLOSE é determinado pelo server, não pelo client. Se client mostra 0s mas server ainda não fechou, client bloqueia UI mas espera evento `jackpot:close` do server.
- **Tolerância:** Aceitar depósitos até 500ms após o `closeAt` server-side para compensar latência.

## Problema 2: Race Condition em Depósitos Simultâneos

**Sintoma:** Dois jogadores depositam no mesmo milissegundo. Ambos passam na validação de saldo, mas o pot fica inconsistente.

**Causa:** Depósitos processados em paralelo sem lock.

**Solução:**
- Usar **lock por round** no handler: processar depósitos sequencialmente por roundId.
- Em Lua/JS FiveM, o event loop é single-threaded — events são processados sequencialmente por natureza. Race condition é raro mas possível se dois events chegam no mesmo tick.
- Implementar: validação + débito + insert em **uma única transação SQL** (BEGIN → COMMIT). Se falhar, rollback.
```sql
BEGIN;
  UPDATE users SET money = money - @amount WHERE id = @playerId AND money >= @amount;
  -- Se rows affected = 0 → saldo insuficiente → ROLLBACK
  INSERT INTO casino_jackpot_deposits (round_id, player_id, amount, ...) VALUES (...);
COMMIT;
```

## Problema 3: Jogador Sai do Cassino durante a Rodada

**Sintoma:** Jogador deposita, fecha o NUI (sai do cassino in-game), e perde o resultado.

**Causa:** NUI fechado não recebe mais events.

**Solução:**
- Depósito já foi feito e registrado server-side. O jogador **continua participando** mesmo com NUI fechado.
- Se ganhar: o prêmio é creditado automaticamente via handler, sem precisar de NUI aberto.
- Quando reabrir o cassino/jackpot: buscar estado atual via `casino:jackpot:state`. Se tem round ativo, mostrar. Se já acabou e ele ganhou, mostrar notificação de vitória pendente.
- **Histórico persistente:** Jogador pode ver que ganhou na seção de histórico a qualquer momento.

## Problema 4: Performance com Broadcasts Frequentes

**Sintoma:** Pot com 10 jogadores, cada um depositando 3-5 vezes. São 30-50 broadcasts em 60s para TODOS os clients.

**Causa:** `TriggerClientEvent(-1, ...)` envia para todos os players do servidor, não só quem está no cassino.

**Solução:**
- **Filtro no client Lua:** Manter variável `isJackpotOpen = true/false`. Só repassar events para NUI se `isJackpotOpen == true`.
- **Batch updates:** Em vez de broadcast a cada depósito, acumular updates por 200ms e enviar batch. Reduz 50 broadcasts para ~15.
- **Debounce no React:** `useEffect` com debounce de 100ms para re-renderizar donut chart.

## Problema 5: Bots em Servidor com Poucos Jogadores

**Sintoma:** Rodada nunca inicia porque precisa de 2+ jogadores e só tem 1 online.

**Causa:** Servidores pequenos (<20 jogadores) podem não ter tráfego suficiente no cassino.

**Solução:**
- **Config admin:** `bots_enabled: true/false`, `bot_min_deposit: 50`, `bot_max_deposit: 500`.
- Se timer atinge 30s com apenas 1 jogador real, servidor adiciona 1-2 bots com nomes de NPCs (configuráveis pelo admin: "Cassino Dealer", "High Roller", etc.).
- Bots depositam valor aleatório dentro da faixa configurada.
- **Transparência:** Bots são marcados internamente (flag `is_bot = true`). Se bot ganhar, o pot volta para o cofre do cassino (house profit adicional). Jogadores reais NÃO veem que é bot.
- **Feature v2:** Desabilitar bots e simplesmente devolver depósito se < 2 jogadores quando timer expira (abordagem mais honesta).

## Problema 6: Overflow Visual do Donut com Muitos Jogadores Pequenos

**Sintoma:** 8 jogadores depositam, 5 deles com <5% cada. Setores no donut ficam invisíveis.

**Causa:** Setor de 3% em roda de 300px = arco de ~3px.

**Solução:**
- **Mínimo visual:** Se percentual < 3%, agrupar visualmente como "Outros" no donut (mas manter individual na lista).
- **Na lista de jogadores:** Sempre mostrar todos individualmente com percentual exato, independente do donut.
- **No spinner:** Manter todos os setores reais (sem agrupamento) — setores pequenos ainda são visíveis durante rotação rápida.

## Problema 7: Latência entre Resultado e Animação do Spinner

**Sintoma:** O server determina o vencedor instantaneamente, mas a animação do spinner dura 8-12s. Jogadores impacientes.

**Causa:** Descompasso proposital — a animação é para criar tensão/entretenimento.

**Solução:**
- Resultado é determinado server-side e enviado para TODOS os clients no event `jackpot:spin`.
- O event contém: `winnerId`, `spinAngle`, `spinDuration`.
- TODOS os clients animam o spinner simultaneamente com os mesmos parâmetros.
- O winner **NÃO é revelado** visualmente até a animação terminar. O event contém o vencedor, mas o React só mostra o overlay de resultado quando `onAnimationEnd` dispara.
- **Importante:** Todos veem a mesma animação (mesma rotação, mesmo timing) porque recebem os mesmos dados.

```sql
-- ═══════════════════════════════════════════
-- TABELA 1: casino_jackpot_rounds
-- Histórico de todas as rodadas
-- ═══════════════════════════════════════════
CREATE TABLE IF NOT EXISTS casino_jackpot_rounds (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  status          ENUM('open', 'closing', 'closed', 'spinning', 'result', 'cancelled') 
                  NOT NULL DEFAULT 'open',
  pot_total       INT NOT NULL DEFAULT 0,               -- Valor total depositado (GCoin)
  rake_amount     INT NOT NULL DEFAULT 0,               -- Valor da taxa da casa
  rake_percent    DECIMAL(5,2) NOT NULL DEFAULT 5.00,   -- % da taxa (config)
  winner_id       INT DEFAULT NULL,                     -- ID do jogador vencedor (FK users)
  winner_amount   INT DEFAULT NULL,                     -- Valor recebido pelo vencedor
  winner_percent  DECIMAL(5,2) DEFAULT NULL,            -- % que o vencedor tinha
  winning_ticket  INT DEFAULT NULL,                     -- Ticket sorteado
  total_tickets   INT DEFAULT NULL,                     -- Total de tickets na rodada
  spin_angle      DECIMAL(8,3) DEFAULT NULL,            -- Ângulo final do spinner (graus)
  player_count    INT NOT NULL DEFAULT 0,               -- Qtd de jogadores únicos
  server_seed     VARCHAR(128) DEFAULT NULL,             -- Seed do servidor (revelado após rodada)
  server_seed_hash VARCHAR(128) DEFAULT NULL,            -- SHA256 do seed (publicado antes)
  client_seeds    TEXT DEFAULT NULL,                     -- Seeds dos 3 primeiros depositantes
  nonce           INT NOT NULL DEFAULT 0,                -- Nonce da rodada
  result_hash     VARCHAR(128) DEFAULT NULL,             -- Hash final do resultado
  timer_duration  INT NOT NULL DEFAULT 60,               -- Duração do timer (segundos)
  anti_snipe_resets INT NOT NULL DEFAULT 0,              -- Quantas vezes anti-snipe resetou
  opened_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  closed_at       DATETIME DEFAULT NULL,
  completed_at    DATETIME DEFAULT NULL,
  
  INDEX idx_status (status),
  INDEX idx_winner (winner_id),
  INDEX idx_opened (opened_at DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
-- NOTA: server_seed é NULL durante a rodada (revelado apenas após resultado)
-- NOTA: server_seed_hash é publicado quando rodada abre (para verificação)

-- ═══════════════════════════════════════════
-- TABELA 2: casino_jackpot_deposits
-- Cada depósito individual (um jogador pode ter múltiplos)
-- ═══════════════════════════════════════════
CREATE TABLE IF NOT EXISTS casino_jackpot_deposits (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  round_id        INT NOT NULL,                          -- FK casino_jackpot_rounds.id
  player_id       INT NOT NULL,                          -- FK users
  player_name     VARCHAR(64) NOT NULL,                  -- Nome do jogador (cache para display)
  amount          INT NOT NULL,                          -- Valor depositado (GCoin)
  ticket_start    INT NOT NULL,                          -- Primeiro ticket atribuído
  ticket_end      INT NOT NULL,                          -- Último ticket atribuído
  player_color    VARCHAR(7) NOT NULL DEFAULT '#D4A843', -- Cor atribuída (#hex)
  is_bot          TINYINT(1) NOT NULL DEFAULT 0,         -- 1 = bot NPC
  client_seed     VARCHAR(128) DEFAULT NULL,             -- Seed do client (se entre os 3 primeiros)
  deposited_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  
  INDEX idx_round (round_id),
  INDEX idx_player_round (player_id, round_id),
  CONSTRAINT fk_deposit_round FOREIGN KEY (round_id) 
    REFERENCES casino_jackpot_rounds(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
-- NOTA: ticket_start e ticket_end definem o range de tickets deste depósito
-- NOTA: Um jogador pode ter múltiplas rows (depósitos múltiplos)

-- ═══════════════════════════════════════════
-- TABELA 3: casino_jackpot_config
-- Configurações admin (1 row padrão)
-- ═══════════════════════════════════════════
CREATE TABLE IF NOT EXISTS casino_jackpot_config (
  id                 INT PRIMARY KEY DEFAULT 1,
  enabled            TINYINT(1) NOT NULL DEFAULT 1,      -- Jogo habilitado/desabilitado
  timer_seconds      INT NOT NULL DEFAULT 60,            -- Duração do timer (s)
  rake_percent       DECIMAL(5,2) NOT NULL DEFAULT 5.00, -- Taxa da casa (%)
  min_deposit        INT NOT NULL DEFAULT 10,            -- Depósito mínimo (GCoin)
  max_deposit_percent DECIMAL(5,2) NOT NULL DEFAULT 80.00, -- Max % do pot por depósito
  min_players        INT NOT NULL DEFAULT 2,             -- Mín jogadores para rodada válida
  max_players        INT NOT NULL DEFAULT 10,            -- Máx jogadores por rodada
  anti_snipe_seconds INT NOT NULL DEFAULT 10,            -- Threshold de anti-snipe (s)
  anti_snipe_max     INT NOT NULL DEFAULT 3,             -- Máx resets anti-snipe
  cooldown_seconds   INT NOT NULL DEFAULT 5,             -- Cooldown entre rodadas (s)
  spin_duration_ms   INT NOT NULL DEFAULT 8000,          -- Duração da animação do spin (ms)
  bots_enabled       TINYINT(1) NOT NULL DEFAULT 0,      -- Bots NPC habilitados
  bot_min_deposit    INT NOT NULL DEFAULT 50,            -- Depósito mín do bot
  bot_max_deposit    INT NOT NULL DEFAULT 500,           -- Depósito máx do bot
  bot_trigger_seconds INT NOT NULL DEFAULT 30,           -- Segundos para bot entrar
  updated_at         DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ═══════════════════════════════════════════
-- SEED: Configuração padrão
-- ═══════════════════════════════════════════
INSERT INTO casino_jackpot_config (id) VALUES (1);
-- Valores padrão inseridos automaticamente pelas DEFAULT clauses
```

### Perguntas estratégicas para o BC antes da implementação:

1. **Rake/Taxa:** 5% é ok? Ou quer configurável pelo painel admin (já está no schema)?
2. **Bots:** Habilitar bots NPC por padrão ou deixar desabilitado? (schema tem flag `bots_enabled`)
3. **Timer:** 60s é um bom padrão? Ou preferir 90s para servidores com menos jogadores?
4. **Máximo de jogadores:** 10 está bom? Ou quer mais?
5. **Low/Mid/High pots:** Implementar na v1 ou deixar como feature v2?
6. **Audit log:** Criar tabela separada de audit ou usar log existente do servidor?

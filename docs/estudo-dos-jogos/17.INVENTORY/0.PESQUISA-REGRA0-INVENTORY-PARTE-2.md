# 3. EIXO 2 - TECNOLOGIA DISPONIVEL (2025-2026)

O Inventory #17 tem **4 componentes tecnicos distintos**: (A) o grid de itens (layout responsivo de cards), (B) o sistema de filtros e ordenacao, (C) as acoes por item (sell, equip, detail, trade), e (D) as animacoes e feedback visual. A pesquisa avaliou abordagens concretas para cada um.

## 3.1 Componente A - Grid de Itens (Layout de Cards)

### Opcao A1 - CSS Grid Nativo + Framer Motion

**Como funciona:**
- Grid responsivo com `display: grid` e `grid-template-columns: repeat(auto-fill, minmax(140px, 1fr))`
- Cada item e um card React com imagem, nome, raridade (borda colorida), valor
- Framer Motion `layout` prop para animacao automatica quando itens sao filtrados/reordenados
- `AnimatePresence` para entrada/saida suave ao filtrar

**Tecnica CSS:**
```css
.inventory-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(140px, 1fr));
  gap: clamp(8px, 1vw, 14px);
  padding: clamp(8px, 1vw, 16px);
  overflow-y: auto;
  max-height: 100%;
}
```

**Pros:**
- Zero dependencias novas (CSS Grid nativo + Framer Motion ja existe)
- Responsivo nativamente (auto-fill adapta colunas ao container)
- Framer Motion `layout` anima reordenacao sem codigo extra
- Performance: CSS Grid e pintado pela GPU, zero JS no layout
- Familiar ao time (mesmo pattern dos cards do BentoGrid.tsx)

**Contras:**
- Sem drag-drop nativo (nao necessario para v1 do inventario)
- Virtual scroll precisa de implementacao manual se >100 itens

**Veredicto:** MELHOR OPCAO. Zero libs novas, performance maxima, responsivo nativo. Drag-drop nao e necessario na v1 - o jogador clica/toca no item e escolhe acao, nao arrasta.

---

### Opcao A2 - dnd-kit (React Drag-Drop Toolkit)

**Fonte:** puckeditor.com/blog/top-5-drag-and-drop-libraries-for-react

**Como funciona:**
- Hooks `useDraggable` e `useDroppable` para cada item e slot
- Grid com arraste entre containers (inventario -> equipamento, inventario -> lixeira)
- Collision detection customizavel, acessibilidade, touch support

**Pros:**
- Drag-drop completo e acessivel
- Suporta grids, listas, containers multiplos
- TypeScript, performance boa

**Contras:**
- Dependencia nova (~30KB) - desnecessaria para v1
- Complexidade de integracao com Framer Motion (conflito de gestos)
- Drag-drop nao e a interacao primaria num inventario de cassino (diferente de RPG)

**Veredicto:** NAO recomendado para v1. Pode ser adicionado em v2 se o BC quiser funcionalidade de "arrastar item para vender" ou "arrastar para equipar". Na v1, botoes/context menu sao suficientes.

---

### Opcao A3 - react-grid-layout (Grid Draggable/Resizable)

**Fonte:** github.com/react-grid-layout/react-grid-layout

**Como funciona:**
- Grid com itens arrastáveis e redimensionaveis
- Hooks: `useGridLayout` com collision, compaction, drop externo
- TypeScript completo, breakpoints responsivos

**Pros:**
- Extremamente poderoso para dashboards
- Compaction automatica (itens se reorganizam ao mover)

**Contras:**
- Overhead grande para um inventario simples
- Feito para dashboards, nao para inventarios de jogos
- Bundle pesado (~50KB)
- Conflito com Framer Motion

**Veredicto:** NAO recomendado. Overengineering para o caso de uso.

## 3.3 Componente B - Sistema de Filtros e Ordenacao

**Abordagem:** React state com `useMemo` para filtragem performatica.

**Filtros necessarios:**
- **Tipo de item:** weapon, vehicle, clothing, money, vip, cosmetic, special (ENUM ja existe no SQL `casino_case_items.item_type`)
- **Raridade:** common, uncommon, rare, epic, legendary (ENUM ja existe)
- **Status:** owned, equipped (filtrar sold/traded pois nao estao mais no inventario)
- **Origem:** qual caixa o item veio (JOIN com `casino_cases.name`)

**Ordenacao:**
- Por valor (sell_back_value) - crescente/decrescente
- Por raridade (common=1 a legendary=5) - crescente/decrescente
- Por data de aquisicao (acquired_at) - mais recente primeiro
- Por nome (alfabetico)

**Tecnica:**
```
const itensFiltrados = useMemo(() => {
  let resultado = [...itens];
  if (filtroTipo !== 'all') resultado = resultado.filter(i => i.item_type === filtroTipo);
  if (filtroRaridade !== 'all') resultado = resultado.filter(i => i.rarity === filtroRaridade);
  // sort
  resultado.sort((a, b) => {
    if (ordem === 'valor_desc') return b.sell_back_value - a.sell_back_value;
    if (ordem === 'valor_asc') return a.sell_back_value - b.sell_back_value;
    if (ordem === 'raridade_desc') return RARITY_ORDER[b.rarity] - RARITY_ORDER[a.rarity];
    if (ordem === 'recente') return new Date(b.acquired_at) - new Date(a.acquired_at);
    return a.name.localeCompare(b.name);
  });
  return resultado;
}, [itens, filtroTipo, filtroRaridade, ordem]);
```

**Performance:** `useMemo` recalcula so quando filtros mudam. Para inventarios com <200 itens (realista para cassino RP), zero concern de performance. Se precisar escalar para 500+, adicionar virtual scroll com `react-window` (ja resolvido, lib de 6KB).

## 3.4 Componente C - Acoes por Item

**Frontend precisa:**

- SellConfirmModal com: imagem do item, nome, valor de venda (G$ X), botao confirmar/cancelar
- Feedback: toast de sucesso ("Vendido! +G$ 500"), animacao do item saindo do grid
- Som: `sell_confirm.mp3` (coin cascade)

### Acao 2: EQUIPAR (Equip)

**Backend precisa ser criado:** Atualizar status para 'equipped' e disparar evento no server FiveM para dar o item ao jogador no GTA (ex: `TriggerServerEvent('casino:inventory:equip', itemId)`). O server handler valida e usa a API do vRP/QBCore para adicionar arma/veiculo ao jogador.

**Frontend precisa:**
- Botao "EQUIPAR" no ItemDetailModal (so para itens com `item_type` weapon/vehicle/clothing)
- Confirmacao: "Equipar [nome]? O item sera adicionado ao seu personagem."
- Feedback: toast "Equipado!", badge "EQUIPADO" no card do item no grid
- Itens ja equipados mostram botao "DESEQUIPAR" em vez de "EQUIPAR"

### Acao 3: DETALHES (Info)

**Frontend:**
- Modal com: imagem grande, nome, descricao, raridade (badge colorido), valor de mercado, valor de sell-back, data de aquisicao, caixa de origem, status atual
- Sem chamada backend - dados ja carregados no fetch inicial

### Acao 4: TROCAR (Trade P2P - FUTURO)

```

  - Total DOM elements por card: ~8 (container, img, nome, raridade, valor, overlay, border, glow)
  - Total visivel: ~160 elements
  - FPS impact: 0% (CSS Grid = GPU layout)

  - Recalculo: so quando filtro muda (nao a cada frame)

  - Cascata de entrada: 12-20 cards x 60ms stagger = ~1.2s total

  - FPS impact: <1% durante animacao, 0% em idle
```

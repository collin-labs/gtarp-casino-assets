### Problema 1: Performance com 200+ product cards no CEF

**SINTOMA:** Grid com 200 cards DOM, cada um com imagem PNG + hover glow + tooltip, causa frame drops no NUI CEF Chromium 93 quando scroll rápido.

**CAUSA:** Cada card é um DOM node com box-shadow transition, background-image load, e event listeners para hover. 200 nodes = 200 listeners + 200 imagens na memória. CEF não otimiza como Chrome desktop.

**SOLUÇÃO:**
```
1. Virtualização: react-window com FixedSizeGrid
   - Renderiza só os cards visíveis (12-18 por viewport)
   - Recicla DOM nodes no scroll
   
2. Lazy loading de imagens:
   loading="lazy" em cada <img>
   IntersectionObserver para carregar só quando visível
   Placeholder com skeleton pulse enquanto carrega

3. will-change: transform nos cards
   contain: layout style no grid container
   → Isola repaint por card, não repinta o grid inteiro

4. Debounce hover: não aplicar glow em scroll rápido
   let hoverTimeout; 
   onMouseEnter: hoverTimeout = setTimeout(applyGlow, 100);
   onMouseLeave: clearTimeout(hoverTimeout);
```

### Problema 2: Race Condition na compra simultânea

**SINTOMA:** Dois jogadores clicam "COMPRAR" no mesmo item ao mesmo tempo. Ambos veem "Compra realizada!" mas só 1 deveria ter comprado. Possível: duplo débito, item duplicado, ou erro sem rollback.

**CAUSA:** Sem lock no listing, ambas transações SQL rodam em paralelo. O `status='active'` é verdadeiro para ambas no momento da query.

**SOLUÇÃO:**
```sql
-- Usar SELECT FOR UPDATE dentro da transaction
START TRANSACTION;
SELECT * FROM marketplace_listings 
  WHERE id = ? AND status = 'active' 
  FOR UPDATE;  -- LOCK a row

-- Se retornou, prosseguir com a compra
UPDATE marketplace_listings SET status = 'sold', buyer_id = ? WHERE id = ?;
-- ... demais operações
COMMIT;

-- Se NÃO retornou (outro já comprou), ROLLBACK
-- Return: { success: false, error: 'Item já foi vendido' }
```

### Problema 3: Price Sparkline no card (Canvas mini dentro de cada card)

**SINTOMA:** Se cada card renderiza um Canvas 2D de 80×30px para sparkline do preço, 200 cards = 200 Canvas contexts = overhead no CEF.

**CAUSA:** Cada Canvas context consome memória GPU. 200 é demais para o CEF compartilhando GPU com GTA.

**SOLUÇÃO:**
```
Opção A (recomendada): SVG polyline em vez de Canvas
  <svg viewBox="0 0 80 30" width="80" height="30">
    <polyline points="0,25 15,20 30,22 45,15 60,18 80,10" 
              fill="none" stroke="#00E676" strokeWidth="1.5"/>
  </svg>
  → Zero Canvas context, renderiza como DOM, leve

Opção B: Sparkline só no modal de detalhe (não no card)
  → Card mostra apenas badge ↑15% / ↓8%
  → Modal abre com sparkline Canvas completo (apenas 1 context)
  
Decisão: Opção B para v1 (badge no card, sparkline no modal)
         Opção A para v2 se performance ok
```

### Problema 4: Listings expirados sem cleanup

**SINTOMA:** Listings com `expires_at` passado continuam aparecendo na busca. Grid fica poluída com "fantasmas".

**CAUSA:** Sem job periódico que verifica expiração. A query filtra por `status='active'` mas não por `expires_at`.

**SOLUÇÃO:**
```javascript
// Job periódico (a cada 60s ou no início de cada busca)
async function cleanupExpiredListings() {
  const expired = await oxmysql.update(
    `UPDATE marketplace_listings 
     SET status = 'expired' 
     WHERE status = 'active' AND expires_at < NOW()`
  );
  
  if (expired.affectedRows > 0) {
    // Devolver itens ao inventário dos vendedores
    const items = await oxmysql.query(
      `SELECT id, seller_id, item_type, item_data 
       FROM marketplace_listings WHERE status = 'expired' AND expires_at >= DATE_SUB(NOW(), INTERVAL 1 MINUTE)`
    );
    for (const item of items) {
      await oxmysql.insert(
        `INSERT INTO player_inventory (player_id, item_type, data, origin, status)
         VALUES (?, ?, ?, 'MARKETPLACE', 'AVAILABLE')`,
        [item.seller_id, item.item_type, item.item_data]
      );
      await auditLog('ITEM_EXPIRED', item.seller_id, item.id, { returned: true });
    }
  }
}

// Rodar a cada 60s
setInterval(cleanupExpiredListings, 60000);
```

### Problema 6: Indexação SQL para busca com múltiplos filtros

**SINTOMA:** Query com WHERE category + rarity + price range + FULLTEXT search + ORDER BY fica lenta com 10.000+ listings.

**CAUSA:** MySQL não consegue usar FULLTEXT INDEX e B-TREE INDEX na mesma query eficientemente. Composite index pode não cobrir todas as combinações de filtros.

**SOLUÇÃO:**
```sql
-- Index composite para os filtros mais comuns
ALTER TABLE marketplace_listings 
  ADD INDEX idx_active_search (status, item_type, price, created_at);

-- Para busca FULLTEXT, usar em 2 etapas:
-- 1. FULLTEXT retorna IDs relevantes (rápido)
-- 2. JOIN com filters nos IDs (rápido porque subset pequeno)

SELECT l.* FROM marketplace_listings l
JOIN marketplace_item_types it ON l.item_type = it.id
WHERE l.id IN (
  SELECT l2.id FROM marketplace_listings l2
  JOIN marketplace_item_types it2 ON l2.item_type = it2.id
  WHERE MATCH(it2.name, it2.description) AGAINST(? IN BOOLEAN MODE)
  AND l2.status = 'active'
)
AND it.category = ?
AND it.rarity = ?
AND l.price BETWEEN ? AND ?
ORDER BY l.created_at DESC
LIMIT 20;
```

### Problema 7: Saldo negativo (exploit de timing)

**SINTOMA:** Jogador com 50.000 GCoins compra 2 itens de 30.000 GCoins simultâneos (2 abas ou 2 requests rápidos). Total: 60.000 > 50.000, mas ambas transações passam porque o saldo é verificado ANTES do débito.

**CAUSA:** Check de saldo e débito não são atômicos — entre o SELECT (verifica saldo) e o UPDATE (debita), outro request pode passar.

**SOLUÇÃO:**
```sql
-- UPDATE condicional (atômico no MySQL)
UPDATE users SET bank = bank - ? 
WHERE identifier = ? AND bank >= ?;

-- Se affectedRows = 0, saldo insuficiente (não precisou de SELECT)
-- Se affectedRows = 1, débito OK
-- MySQL garante atomicidade do UPDATE single-row
```

### Problema 8: Audit log cresce infinitamente

**SINTOMA:** Com 200 jogadores fazendo 10 transações/dia, são 2.000 rows/dia no audit log. Em 1 ano = 730.000 rows. Queries de admin ficam lentas.

**CAUSA:** Tabela append-only sem partitioning ou archiving.

**SOLUÇÃO:**
```sql
-- Partitioning por mês (MySQL 5.7+)
ALTER TABLE marketplace_audit
PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
  PARTITION p202603 VALUES LESS THAN (202604),
  PARTITION p202604 VALUES LESS THAN (202605),
  PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- Ou: job mensal que move rows antigas para archive table
-- Admin dashboard consulta apenas últimos 30 dias por default
```

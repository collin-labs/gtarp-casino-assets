O Bingo tem **2 componentes técnicos principais**: (A) a cartela de bingo (grid interativo) e (B) a ball machine animada (sorteio visual). A pesquisa avaliou abordagens para cada um.

## 3.1 Componente A — Cartela de Bingo (Grid 5×5)

### Opção A1 — DOM Puro + CSS Grid + React State

**Como funciona:**
- Grid 5×5 com `display: grid; grid-template-columns: repeat(5, 1fr)`
- Cada célula é um `<div>` com estado React: `idle | selected | dabbed | match | miss`
- Wild central: célula sempre marcada (já conta como acerto)
- Lucky Numbers: classe CSS especial com glow dourado + animação pulse
- Auto-dabbing: quando bola é sorteada, `setState` marca célula correspondente com transição CSS

**Prós:**
- Performance máxima (DOM elements com CSS transitions = GPU-accelerated)
- Controle total sobre estados visuais (hover, selected, dabbed, match, miss)
- Responsividade trivial com CSS Grid + clamp()
- Integra perfeitamente com Framer Motion para animações de entrada/revelação
- Acessibilidade nativa (DOM elements são focáveis, leitores de tela funcionam)

**Contras:**
- Com 4 cartelas simultâneas = 100 DOM elements (leve, mas precisa de atenção)
- Estados complexos exigem gestão cuidadosa

**Veredicto:** ✅ **MELHOR OPÇÃO para cartelas.** DOM + CSS Grid é a abordagem padrão para grids interativos. Todos os projetos de bingo React pesquisados (react-bingo, bingocompanionapp) usam DOM. Zero motivo para Canvas aqui.

### Opção A2 — Canvas 2D para Cartela

**Prós:** Render pixel-perfect, performance uniforme
**Contras:** Sem hover nativo, sem acessibilidade, hit-testing manual, mais código
**Veredicto:** ❌ **Não recomendado para cartelas.** Canvas faz sentido para animações contínuas (ball machine), não para grids estáticos com interação.

### Decisão Cartela: DOM + CSS Grid + React

**Geração de Cartelas (Server-Side):**

```
ALGORITMO DE GERAÇÃO — BINGO 5×5 (pool 50)

1. Server gera array [1..50], shuffle via Fisher-Yates
2. Para cada cartela:
   a. Seleciona 24 números únicos do pool shuffled
   b. Distribui em grid 5×5 (centro = wild, 24 posições restantes)
   c. Opcionalmente: colunas têm ranges (col1: 1-10, col2: 11-20, etc.)
   d. Garante: nenhuma cartela é idêntica na mesma rodada
3. Retorna: { cards: [{ grid: number[][], luckyNumbers: number[] }], pattern: string }

COMPLEXIDADE: O(n) para gerar, O(1) para verificar match. Trivial.
```

**Verificação de Padrão (Server-Side):**

```
ALGORITMO DE VERIFICAÇÃO — PADRÃO DE VITÓRIA

function checkPattern(card: number[][], dabbed: boolean[][], pattern: string): boolean {
  switch(pattern) {
    case '1LINE_H': return rows.some(row => row.every(cell => dabbed[cell]))
    case '1LINE_V': return cols.some(col => col.every(cell => dabbed[cell]))
    case '1LINE_D': return diags.some(diag => diag.every(cell => dabbed[cell]))
    case '1LINE':   return checkAnyLine(dabbed)
    case '2LINES':  return countLines(dabbed) >= 2
    case 'FULL_HOUSE': return dabbed.flat().every(Boolean)
    case 'FOUR_CORNERS': return [0,0],[0,4],[4,0],[4,4] all dabbed
    case 'X_SHAPE': return bothDiags all dabbed
    case 'DIAMOND': return diamondCoords all dabbed
    case 'T_SHAPE': return topRow + middleCol all dabbed
  }
}
```

## 3.2 Componente B — Ball Machine Animada (Sorteio Visual)

### Opção B1 — Canvas 2D Puro (Física Manual Simplificada)

**Fontes:** MDN Advanced Animations, CodePen bouncing balls, GeeksForGeeks bounce ball

**Como funciona:**
- Container circular/oval no Canvas
- 15-20 bolas pequenas (decorativas) quicando dentro com física simples
- Cada bola: `{x, y, vx, vy, radius, number, color}`
- Gravidade: `vy += 0.15` por frame
- Bounce: inverter velocidade ao tocar borda do container
- Quando bola é "sorteada": uma bola sai do container, cresce, revela número
- Render: `requestAnimationFrame` → clear → draw all balls → update physics

**Técnica de container circular:**
```javascript
// Colisão com parede circular (mais simples que polyline)
const dx = ball.x - centerX;
const dy = ball.y - centerY;
const dist = Math.sqrt(dx*dx + dy*dy);
if (dist + ball.radius > containerRadius) {
  // Normal da colisão
  const nx = dx / dist;
  const ny = dy / dist;
  // Reflete velocidade
  const dot = ball.vx * nx + ball.vy * ny;
  ball.vx -= 2 * dot * nx;
  ball.vy -= 2 * dot * ny;
  // Empurra para dentro
  ball.x = centerX + nx * (containerRadius - ball.radius);
  ball.y = centerY + ny * (containerRadius - ball.radius);
}
```

**Prós:**
- Zero dependências (Canvas 2D nativo)
- Performance excelente para 15-20 bolas (~0.2ms/frame)
- Container circular com math simples (sem polyline do Matter.js)
- Controle total sobre visual (gradientes, glow, cores por número)
- Já temos GoldParticles.tsx como referência Canvas 2D no projeto

**Contras:**
- Colisão ball-ball manual (mas com 15-20 bolas é trivial)
- Física menos "realista" que Matter.js (mas para decorativo, OK)
- ~100-150 linhas de código para ball machine funcional

**Veredicto:** ✅ **MELHOR OPÇÃO.** Canvas 2D puro com container circular e física simplificada. 15-20 bolas decorativas. Quando bola é sorteada, animação de "saída" do container. Zero dependências novas.

### Opção B2 — Matter.js (Física Realista)

**Mesma lib pesquisada para a Loteria #18.**

**Prós:** Física realista automática, colisão ball-ball nativa
**Contras:** 87KB extra, container circular precisa polyline, overhead para efeito decorativo
**Veredicto:** ⚠️ **Viável mas overkill.** Para o Bingo, a ball machine é decorativa (não é o foco como na Loteria). Canvas 2D puro com física simplificada atinge resultado visual suficiente sem dependência extra. Se a Loteria #18 já usar Matter.js, podemos reutilizar. Se não, Canvas 2D puro.

### Opção B3 — CSS Animations (Sem Canvas)

**Como funciona:** Bolas como `<div>` com CSS keyframes, posições pré-calculadas
**Prós:** Sem Canvas, GPU-accelerated
**Contras:** Não parece realista (sem colisão), limitado a ~10 bolas, difícil container circular
**Veredicto:** ❌ **Não recomendado.** Não atinge visual de ball machine.

### Decisão Ball Machine: Canvas 2D Custom

**Arquitetura proposta:**

```
BingoBallMachine.tsx (React Component)
├── useRef<HTMLCanvasElement> — referência ao canvas
├── useEffect — setup, resize, animation loop
├── BALL POOL (decorativo):
│   15-20 bolas pequenas quicando dentro do container circular
```

## 3.3 Geração de Números (Server-Side — Provably Fair)

Reutiliza o mesmo algoritmo HMAC_SHA256 da Loteria #18, adaptado:

```
ALGORITMO PROVABLY FAIR — BINGO

1. Server gera server_seed (random 256-bit)
2. Server publica hash = SHA256(server_seed)
3. Jogador confirma aposta

4. GERAÇÃO DAS CARTELAS:
   combined = HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":cards")
   Para cada cartela i:
     Usa bytes do hash para selecionar 24 números únicos de [1..50]

5. GERAÇÃO DO SORTEIO (20 bolas):
   combined = HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":draw")
   Seleciona 20 números únicos de [1..50] usando bytes sequenciais

6. LUCKY NUMBERS (3-5 por rodada):
   combined = HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":lucky")
   Seleciona 3-5 números dos 20 sorteados
   Atribui multiplicadores (2x/3x/5x) via tabela de probabilidades

7. WILD BALL:
   combined = HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":wild")
   roll = parseInt(hex, 16) % 100
   Se roll < 20 (20% chance): Wild Ball ativa na bola #15 do sorteio
   Extra balls: 1-3 (ponderado: 60% 1 ball, 30% 2 balls, 10% 3 balls)

8. Server revela server_seed após rodada
```

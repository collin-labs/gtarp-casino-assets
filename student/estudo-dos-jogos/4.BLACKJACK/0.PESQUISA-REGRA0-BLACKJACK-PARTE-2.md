# 7. PROVABLY FAIR PARA BLACKJACK
# =============================================

## 7.1 Diferenca vs Slots/Crash

Em Slots: cada spin gera posicoes no grid (30 numeros). Resultado e independente.
Em Blackjack: um DECK INTEIRO precisa ser embaralhado de forma verificavel. Multiplas cartas sao distribuidas sequencialmente do MESMO deck. O jogador precisa verificar que:
1. O deck estava embaralhado ANTES da primeira carta ser distribuida.
2. Nenhuma carta foi adicionada, removida ou reordenada durante a mao.
3. O shuffle foi determinado pelos seeds (server + client + nonce).

## 7.2 Implementacao Proposta (baseado em Stake.com)

### ANTES DO SHOE (inicio de uma sessao/shoe novo):
```
1. Server gera server_seed (64-char hex string aleatorio)
2. Server calcula hash_publico = SHA-256(server_seed)
3. hash_publico enviado ao NUI antes de qualquer aposta
4. Jogador pode definir client_seed (ou usar o gerado pelo browser)
5. Nonce inicia em 0
```

### SHUFFLE DO DECK (Fisher-Yates Provably Fair):
```
6. deck = array de 312 cartas (6 decks x 52)
7. Para i de 311 ate 1:
   a. hmac = HMAC-SHA256(server_seed, client_seed + ":" + nonce + ":" + i)
   b. j = hmac_bytes_to_int(hmac[0..3]) % (i + 1)
   c. swap(deck[i], deck[j])
8. Deck embaralhado esta fixo para todo o shoe
9. Cartas sao distribuidas sequencialmente: deck[0], deck[1], deck[2]...
```

### DURANTE A MAO:
```
10. Nonce NAO incrementa entre cartas da mesma mao (deck e sequencial)
11. Nonce incrementa a cada NOVA mao (novo deal)
12. Cada carta distribuida: proximo indice do deck pre-embaralhado
13. Se shoe acabar (penetracao ~75%), novo shuffle com nonce incrementado
```

### APOS A SESSAO:
```
14. Server revela server_seed original
15. Jogador verifica: SHA-256(server_seed_revelado) == hash_publico recebido antes
16. Se match: jogador pode reconstruir o shuffle inteiro e verificar cada carta
17. Se NAO match: manipulacao detectada
```

### Verificacao pelo jogador:
```
Input: server_seed + client_seed + nonce
Processo: Fisher-Yates com HMAC-SHA256 (mesmo algoritmo)
Output: deck embaralhado identico ao usado no jogo
Comparar: sequencia de cartas distribuidas == deck[0], deck[1], etc.
```

## 7.3 Anti-Card-Counting no Digital

Em casinos fisicos, card counting explora a penetracao do shoe. No Blackout Casino digital:
- Shuffle a cada shoe novo (configuravel: a cada mao ou a cada 75% do shoe).
- Provably Fair garante que o shuffle e aleatorio e nao influenciado pelo cassino.
- Jogador NAO pode contar cartas efetivamente se shuffle ocorre frequentemente.
- Configuracao recomendada: shuffle a cada novo shoe completo (312 cartas, ~20-25 maos antes do reshuffle). Isso permite uma experiencia mais "realista" sem comprometer a seguranca.

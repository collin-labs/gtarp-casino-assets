## 7.1 Modelo Economico

O Giveaway opera como um sistema de raffle com pool variavel. Diferente de jogos tradicionais com house edge fixo, aqui a margem eh controlada diretamente pelo admin que define o premio e o preco dos tickets.

**Formulas:**
- Pool total = preco_ticket x total_tickets_vendidos
- Margem cassino = pool_total - valor_premio
- Se pool_total > valor_premio: cassino lucra
- Se pool_total < valor_premio (poucos tickets vendidos): cassino subsidia (loss leader para engajamento)
- Se tickets gratuitos: custo zero para jogador, premio 100% subsidio do cassino

**RTP para o jogador:** RTP = (valor_premio / pool_total) x 100%. Exemplo: premio G$50.000, 500 tickets a G$100 cada = pool G$50.000 = RTP 100% (zero margin). Se 600 tickets vendidos = pool G$60.000 = RTP 83.3% (margem 16.7% pro cassino).

## 8.1 Algoritmo

**HMAC-SHA256** — Mesmo padrao usado por Stake.com, Duelbits, CSGOEmpire e verificado por ferramentas independentes (BtcGambling, StakeSim, Provably.com).

## 8.2 Fluxo Completo para Raffle Multi-Ticket (10 passos)

1. **Admin cria sorteio:** Server gera server_seed = crypto.randomBytes(32).toString('hex'). Calcula server_seed_hash = SHA256(server_seed). Salva hash no banco (publico). Server seed fica oculto
2. **Jogador compra primeiro ticket:** Client seed gerado no browser do jogador (ou aceita default). Client seed salvo no sorteio. Nonce = 0
3. **Jogadores compram tickets:** Cada compra incrementa total_tickets. Tickets numerados sequencialmente (1, 2, 3... N)
4. **Sorteio encerra (draw_at chegou):** Status muda para DRAWING. Nenhum ticket novo aceito
5. **Calculo do vencedor:** result = HMAC_SHA256(server_seed, client_seed + ":" + nonce). Converte primeiros 8 chars hex para decimal: decimal = parseInt(result.substring(0, 8), 16). winning_index = decimal % total_tickets. Ticket vencedor = ticket na posicao winning_index
6. **Drum reveal:** Frontend recebe winning_index. Anima drum scroll passando por todos os tickets ate parar no vencedor. Animacao eh VISUAL — o resultado ja esta determinado server-side
7. **Winner confirmation:** Status muda para COMPLETED. winner_ticket_id salvo. Premio creditado na wallet do vencedor
8. **Server seed revelado:** Apos draw, server_seed eh tornado publico (visivel no historico do sorteio)
9. **Verificacao pelo jogador:** Jogador pode copiar server_seed + client_seed + nonce, calcular HMAC_SHA256 localmente, confirmar que winning_index bate com o resultado anunciado
10. **Auditoria permanente:** Todos os dados (seeds, hash, nonce, tickets, winner) ficam salvos na tabela casino_giveaways para consulta futura

## 8.3 Conversao Hash para Resultado

```javascript
// Exemplo concreto
const serverSeed = "a1b2c3d4e5f6..."; // 64 chars hex
const clientSeed = "player_seed_xyz";
const nonce = 0;
const totalTickets = 423;

const hmac = crypto.createHmac('sha256', serverSeed)
  .update(clientSeed + ':' + nonce)
  .digest('hex');
// hmac = "7f3a2b1c..." (64 chars)

const decimal = parseInt(hmac.substring(0, 8), 16);
// decimal = 2135841052

const winningIndex = decimal % totalTickets;
// winningIndex = 2135841052 % 423 = 187

// Ticket #187 eh o vencedor
```

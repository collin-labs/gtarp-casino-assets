## Problema 1 — Countdown dessincroniza entre jogadores

**Sintoma:** Jogador A ve "02:14:32" restantes, Jogador B ve "02:14:28" — 4 segundos de diferenca. Se ambos estao assistindo o draw ao vivo, um ve o resultado "antes" do outro.

**Causa:** Countdown calculado client-side usando `Date.now()` local. Cada browser tem clock ligeiramente diferente. NUI do FiveM herda o clock do sistema do jogador. Sem referencia server-side, os timers derivam com o tempo.

**Codigo do problema (conceitual):**
```typescript
// ERRADO: countdown baseado em clock local
const [remaining, setRemaining] = useState(
  Math.floor((drawAt.getTime() - Date.now()) / 1000)
);
useEffect(() => {
  const interval = setInterval(() => {
    setRemaining(prev => prev - 1); // decrementa localmente, acumula drift
  }, 1000);
  return () => clearInterval(interval);
}, []);
```

**Solucao:** Countdown baseado em server timestamp. NUI faz fetchNui('get-server-time') a cada 60 segundos para recalibrar. O countdown calcula diff entre draw_at (fixo no banco) e server_now (retornado pelo handler). Drift maximo = 60 segundos antes de recalibrar.

```typescript
// CORRETO: countdown sincronizado com server
const recalibrate = async () => {
  const { serverNow } = await fetchNui('giveaway:get-time');
  const serverDate = new Date(serverNow).getTime();
  const drawDate = new Date(drawAt).getTime();
  setRemaining(Math.floor((drawDate - serverDate) / 1000));
};
useEffect(() => {
  recalibrate(); // calibra na montagem
  const interval = setInterval(recalibrate, 60000); // recalibra a cada 60s
  const tick = setInterval(() => setRemaining(prev => Math.max(0, prev - 1)), 1000);
  return () => { clearInterval(interval); clearInterval(tick); };
}, [drawAt]);
```

## Problema 2 — Drum reveal com 500+ tickets causa lag no CEF

**Sintoma:** Se um sorteio popular tem 500 tickets e o drum precisa renderizar todos como DOM elements dentro do scroll container, o CEF (Chromium Embedded Framework) do FiveM pode dropar frames durante a animacao rapida.

**Causa:** FiveM NUI compartilha GPU com o jogo GTA V. O CEF nao tem acesso exclusivo a GPU. Renderizar 500 divs animando translateY simultaneamente consome mais recursos do que o CEF pode lidar suavemente.

**Codigo do problema:**
```tsx
// ERRADO: renderiza TODOS os tickets no DOM
<motion.div ref={scope} style={{ height: tickets.length * ITEM_HEIGHT }}>
  {tickets.map((ticket, i) => ( // 500 divs no DOM
    <div key={i} style={{ height: ITEM_HEIGHT }}>{ticket.playerName}</div>
  ))}
</motion.div>
```

**Solucao:** Virtualizacao do drum. Renderizar apenas ~15 tickets visiveis na viewport do drum (viewport height = 5 * ITEM_HEIGHT). Conforme o scroll avanca, reciclar DOM elements atualizando o conteudo (similar a react-window/react-virtualized). O Motion anima o container, nao os items individuais. Para o CEF, isso significa 15 divs em vez de 500.

```typescript
// CORRETO: virtualizacao — apenas items visiveis no viewport
const VISIBLE_COUNT = 15;
const [visibleStart, setVisibleStart] = useState(0);

// Calcular quais tickets estao visiveis baseado na posicao Y do container
const visibleTickets = tickets.slice(visibleStart, visibleStart + VISIBLE_COUNT);
```

## Problema 3 — Race condition: admin cancela sorteio enquanto jogador compra ticket

**Sintoma:** Admin clica "Cancelar Sorteio" no mesmo instante que jogador confirma compra de ticket. Resultado: ticket eh inserido no banco DEPOIS do sorteio ser cancelado. Jogador pagou por ticket de sorteio morto. Refund nao alcanca esse ticket.

**Causa:** Sem transacao atomica no handler. Dois requests paralelos: (1) admin cancel → UPDATE status = 'CANCELLED', (2) player buy → INSERT ticket (nao verifica status). Se (2) executa entre o UPDATE e o refund batch, ticket fica orfao.

**Codigo do problema:**
```javascript
// ERRADO: buy-ticket nao verifica status atomicamente
exports['casino:giveaways:buy-ticket'] = async (source, data) => {
  const giveaway = await oxmysql.single('SELECT * FROM casino_giveaways WHERE id = ?', [data.giveawayId]);
  // <- JANELA: admin pode cancelar aqui entre o SELECT e o INSERT
  await oxmysql.insert('INSERT INTO casino_giveaway_tickets ...', [...]);
};
```

**Solucao:** Usar transacao MySQL com SELECT ... FOR UPDATE para lock na linha do sorteio durante a compra. Se status != 'ACTIVE', rejeitar a compra dentro da mesma transacao.

```javascript
// CORRETO: transacao atomica com lock
exports['casino:giveaways:buy-ticket'] = async (source, data) => {
  await oxmysql.transaction([
    { query: 'SELECT status FROM casino_giveaways WHERE id = ? FOR UPDATE', values: [data.giveawayId] },
    // Se status != 'ACTIVE', a transacao aborta (handler verifica no callback)
    { query: 'INSERT INTO casino_giveaway_tickets ...', values: [...] },
    { query: 'UPDATE casino_giveaways SET tickets_sold = tickets_sold + ? WHERE id = ?', values: [data.quantity, data.giveawayId] }
  ]);
};
```

## Problema 4 — Auto-draw nao executa se server restartar no momento exato

**Sintoma:** Server programado para draw as 20:00. Server restarta as 19:59. Quando volta as 20:02, o setInterval que verificava draw_at ja foi perdido. Sorteio fica em status ACTIVE eternamente.

**Causa:** setInterval eh in-memory. Se o server reinicia, o intervalo eh perdido. Nao ha mecanismo de recovery.

**Solucao:** Ao iniciar o handler (onResourceStart), fazer SELECT em todos os giveaways com status ACTIVE e draw_at <= NOW(). Se encontrar, executar draw imediatamente. Isso garante que sorteios "perdidos" pelo restart sejam processados na proxima inicializacao.

```javascript
// Recovery no startup do resource
on('onResourceStart', async (resourceName) => {
  if (resourceName !== GetCurrentResourceName()) return;
  const pendentes = await oxmysql.query(
    'SELECT * FROM casino_giveaways WHERE status = ? AND draw_at <= NOW() AND deleted_at IS NULL',
    ['ACTIVE']
  );
  for (const giveaway of pendentes) {
    console.log(`[Giveaways] Recovery: executando draw pendente #${giveaway.id}`);
    await executeDraw(giveaway.id);
  }
});
```

## Problema 5 — Limitacao FiveM: NUI nao persiste estado ao fechar painel

**Sintoma:** Jogador esta na tela de historico do Giveaways, fecha o painel do cassino (ESC), reabre. O componente React eh desmontado e remontado — volta para a tela de lista, perde contexto.

**Causa:** No FiveM, fechar a NUI desmonta o DOM. Quando reabre, o React monta do zero. Nao ha persistencia de estado entre sessoes de NUI.

**Solucao:** Nao tentar persistir estado complexo. O componente sempre inicia na tela de lista (estado default). Dados de sorteios sao buscados via fetchNui ao montar (useEffect com dependency vazia). O jogador ve a lista atualizada a cada abertura. Historico e tickets sao server-side (MySQL) entao nao se perdem — apenas a navegacao interna do componente reseta. Isso eh aceitavel e consistente com Slots/Cases/Roulette que se comportam da mesma forma.

## 5.4 Queries Admin Frequentes

```sql
-- Listar sorteios ativos
SELECT g.*, COUNT(t.id) as tickets_sold_real
FROM casino_giveaways g
LEFT JOIN casino_giveaway_tickets t ON g.id = t.giveaway_id AND t.refunded = 0
WHERE g.status IN ('UPCOMING', 'ACTIVE') AND g.deleted_at IS NULL
GROUP BY g.id
ORDER BY g.draw_at ASC;

-- Verificar tickets de um jogador num sorteio
SELECT * FROM casino_giveaway_tickets
WHERE giveaway_id = ? AND player_id = ? AND refunded = 0
ORDER BY ticket_number ASC;

-- Historico de draws com vencedor
SELECT g.id, g.title, g.prize_value, g.winner_player_name, g.drawn_at,
       g.server_seed, g.client_seed, g.nonce, g.pf_result_hash
FROM casino_giveaways g
WHERE g.status = 'COMPLETED' AND g.deleted_at IS NULL
ORDER BY g.drawn_at DESC LIMIT 20;

-- Refund batch ao cancelar
UPDATE casino_giveaway_tickets SET refunded = 1, refunded_at = NOW()
WHERE giveaway_id = ? AND refunded = 0;
```

| Aspecto | Implementacao |
|---------|--------------|
| Provably Fair | HMAC-SHA256 com server seed gerado na criacao, hash publicado pre-draw, client seed do primeiro comprador, nonce incrementado. Verificavel pelo jogador apos draw |

## 3.1 Mecanica Principal — Analise Detalhada

A compra de tickets eh a espinha dorsal do sistema. No Stake.com, tickets sao gerados automaticamente pelo wagering — o jogador nem percebe que esta "comprando" tickets. No nosso caso, a compra eh explicita: o jogador escolhe quantos tickets quer, ve o subtotal, confirma. Isso cria uma decisao consciente que aumenta o investimento emocional. Quando o jogador compra 5 tickets a G$100 cada, ele investiu G$500 e tem incentivo para voltar e assistir o draw.

O ticket gratuito diario serve como mecanismo de retencao. O jogador que nao tem GCoin para comprar tickets ainda pode participar, mantendo engagement. No Skinflow, o ticket diario eh o responsavel por 40% dos returns diarios (usuarios voltam para clamar). No nosso contexto, o free ticket funciona como "amostra gratis" — o jogador experimenta a emocao do sorteio sem custo, e na proxima vez pode decidir comprar mais tickets para aumentar as chances.

O countdown sincronizado com server timestamp eh uma decisao tecnica que parece simples mas tem implicacao profunda. Se o countdown dessincronia entre jogadores, a confianca no sistema cai. A pesquisa revelou que o padrao em plataformas serias (Stake, Roobet) eh usar server timestamp como referencia unica, com recalibracao periodica no client. O intervalo de 60 segundos entre recalibracoes eh um equilibrio entre precisao (drift maximo 60s) e performance (1 fetchNui por minuto vs 1 por segundo).

## 3.2 Visual e Animacao — Analise Detalhada

O drum reveal eh o momento de pico emocional do jogo. A pesquisa mostrou que a funcao de easing easeOutCubic (1 - Math.pow(1-t, 3)) eh a mais usada em animacoes de desaceleracao porque simula o comportamento fisico de um objeto perdendo momentum — rapido no inicio, lento no final, parada suave. Isso eh fundamental para criar tensao: o jogador ve os nomes passando rapido (blur de velocidade), depois cada vez mais devagar (consegue ler os nomes), ate parar no vencedor (momento de revelacao).

A virtualizacao do drum (renderizar apenas ~15 items visiveis) nao eh apenas uma otimizacao de performance — eh uma necessidade em FiveM. O CEF do FiveM compartilha GPU com o GTA V, que ja consome 70-90% dos recursos graficos. Animacoes com 500 DOM elements simultaneos podem causar stuttering visivel. A pesquisa de scroll animation confirmou que a tecnica de LERP (Linear Interpolation) com requestAnimationFrame eh o padrao para smooth scrolling mesmo em ambientes com recursos limitados.

## 3.3 Seguranca e Anti-Fraude — Secao NOVA

A pesquisa sobre fraud detection em plataformas de raffle revelou riscos especificos para o contexto FiveM GTA RP que nao existiam no estudo anterior:

**Multi-account (Sybil Attack adaptado):** No FiveM, um jogador pode ter multiplos personagens no mesmo servidor. Se o max_tickets_per_player eh 10 e o jogador tem 3 personagens, ele pode comprar 30 tickets totais — 3x a vantagem de qualquer jogador com 1 personagem. A mitigacao eh vincular o limite ao identifier do jogador REAL (steam ID ou license), nao ao personagem. O handler deve fazer COUNT tickets WHERE player_license = ? (nao player_id do personagem).

**Compra automatizada (bot):** Um jogador malicioso poderia automatizar a compra de tickets usando scripts Lua que disparam fetchNui repetidamente. A mitigacao eh rate limiting server-side: maximo 1 compra por segundo por license. Se 2 requests chegam no mesmo segundo, o segundo eh rejeitado com cooldown error.

**Admin abuse:** O admin que cria o sorteio conhece o server_seed (gerado na criacao). Teoricamente, poderia calcular o winning_index e comprar o ticket naquele indice. A mitigacao: o client_seed eh definido pelo PRIMEIRO jogador que compra ticket (nao pelo admin). Como o admin nao sabe qual sera o client_seed ate alguem comprar, ele nao pode pre-calcular o resultado. Adicionalmente, se o admin comprar tickets no proprio sorteio que criou, o audit log registra e o sistema pode bloquear automaticamente (config admin_can_participate = false).

**Timing attack no draw:** Se o draw eh executado por um endpoint publico (nao auto-draw), um jogador poderia tentar disparar o draw antes do horario para ganhar vantagem com menos competidores. Mitigacao: o endpoint draw verifica draw_at <= NOW(). Se o horario nao chegou, rejeita. O auto-draw via setInterval no server eh o mecanismo primario — o endpoint manual eh apenas para admin como fallback.

```sql
-- Receita total por sorteio (pool vs premio = margem)
SELECT
  g.id, g.title, g.prize_value,
  COUNT(t.id) as tickets_vendidos,
  COUNT(t.id) * g.ticket_price as pool_total,
  (COUNT(t.id) * g.ticket_price) - g.prize_value as margem,
  ROUND(g.prize_value / (COUNT(t.id) * g.ticket_price) * 100, 1) as rtp_jogador
FROM casino_giveaways g
LEFT JOIN casino_giveaway_tickets t ON g.id = t.giveaway_id AND t.refunded = 0
WHERE g.status = 'COMPLETED' AND g.deleted_at IS NULL
GROUP BY g.id
ORDER BY g.drawn_at DESC;

-- Top 10 jogadores que mais compram tickets (whales)
SELECT
  t.player_id, t.player_name,
  COUNT(t.id) as total_tickets,
  COUNT(DISTINCT t.giveaway_id) as sorteios_participados,
  SUM(CASE WHEN t.is_winner = 1 THEN 1 ELSE 0 END) as vitorias,
  SUM(CASE WHEN t.is_free = 0 THEN 1 ELSE 0 END) as tickets_pagos,
  SUM(CASE WHEN t.is_free = 1 THEN 1 ELSE 0 END) as tickets_gratis
FROM casino_giveaway_tickets t
WHERE t.refunded = 0
GROUP BY t.player_id, t.player_name
ORDER BY total_tickets DESC
LIMIT 10;

-- Sorteios cancelados com total de reembolsos
SELECT
  g.id, g.title, g.cancel_reason, g.cancelled_by,
  COUNT(t.id) as tickets_reembolsados,
  COUNT(t.id) * g.ticket_price as valor_reembolsado
FROM casino_giveaways g
LEFT JOIN casino_giveaway_tickets t ON g.id = t.giveaway_id AND t.refunded = 1
WHERE g.status = 'CANCELLED'
GROUP BY g.id;

-- Taxa de participacao media (tickets vendidos vs max disponivel)
SELECT
  AVG(g.tickets_sold * 100.0 / g.max_tickets_total) as taxa_media_preenchimento,
  MIN(g.tickets_sold * 100.0 / g.max_tickets_total) as menor_taxa,
  MAX(g.tickets_sold * 100.0 / g.max_tickets_total) as maior_taxa,
  COUNT(g.id) as total_sorteios
FROM casino_giveaways g
WHERE g.status IN ('COMPLETED', 'DRAWING') AND g.deleted_at IS NULL;

-- Deteccao multi-account: licenses com mais tickets que o max_per_player
-- (indica possivel uso de multiplos personagens)
SELECT
  t.player_id, g.id as giveaway_id, g.title,
  COUNT(t.id) as tickets_neste_sorteio,
  g.max_tickets_per_player as limite
FROM casino_giveaway_tickets t
JOIN casino_giveaways g ON t.giveaway_id = g.id
WHERE t.refunded = 0
GROUP BY t.player_id, g.id
HAVING COUNT(t.id) > g.max_tickets_per_player;
```

| Metrica | Formula | Alvo Saudavel | Sinal de Problema |

| Taxa de conversao free→pago | jogadores_free_que_compram / jogadores_free * 100 | 15-30% | <5% = ticket pago caro demais vs premio |
| Recorrencia | jogadores_em_2+_sorteios / jogadores_total * 100 | 30-50% | <15% = experiencia do draw nao engaja |
| Margem media | (pool - premio) / pool * 100 | 10-30% | Negativa = cassino subsidiando demais |

```javascript
const purchaseCooldowns = new Map(); // license → timestamp

  const now = Date.now();
```

## 2.1 Como Funciona o Plinko em Cassinos Online

O Plinko e um jogo de cassino online baseado no conceito do Galton Board (tabuleiro de Galton, 1889) e popularizado pelo programa de TV "The Price is Right" (1983). A mecanica central e simples: uma bola cai de cima de um tabuleiro triangular repleto de pinos (pegs), e a cada pino a bola desvia para a esquerda ou para a direita com probabilidade 50/50. Ao final, a bola aterrissa em um dos slots na base, cada um com um multiplicador que determina o payout.

O que torna o Plinko moderno atraente para cassinos online e a combinacao de tres controles que o jogador ajusta ANTES de cada drop: o numero de linhas de pinos (rows), o nivel de risco (risk level), e o valor da aposta. Essas tres variaveis juntas determinam completamente a tabela de multiplicadores e, consequentemente, a volatilidade da sessao.

A distribuicao de resultados segue uma **distribuicao binomial** — a mesma matematica do Triangulo de Pascal. Com N rows, existem 2^N caminhos possiveis e N+1 slots na base. A probabilidade de aterrissar no slot K e calculada por C(N,K) / 2^N. Isso cria uma curva em sino onde os slots centrais sao muito mais provaveis que os extremos.

## 4.1 Opcoes Tecnicas Comparadas

### Opcao A - Matter.js (Fisica 2D Real)

Matter.js e uma engine de fisica 2D para JavaScript que suporta corpos rigidos, colisoes, e constraints. E a biblioteca DOMINANTE para implementacoes de Plinko na web — praticamente todo exemplo encontrado na pesquisa usa Matter.js.

**Como funciona para Plinko:**
- `Engine.create()` inicializa o motor de fisica com gravidade
- `Bodies.circle(x, y, radius, { isStatic: true })` cria cada peg como corpo estatico
- `Bodies.circle(x, y, ballRadius, { restitution: 0.5 })` cria a bola com bounce
- `Events.on(engine, 'collisionStart', callback)` detecta quando a bola toca um peg
- O grid de pegs e criado com offset alternado por row (mesma logica do quincunx)

**Codigo conceitual (baseado nos exemplos encontrados):**
```javascript
const engine = Engine.create();
engine.world.gravity.y = 2;

// Grid de pegs com offset alternado
const spacing = width / (rows + 1);
for (let row = 0; row < rows; row++) {
  const cols = row % 2 === 0 ? rows : rows - 1;
  const offset = row % 2 === 0 ? 0 : spacing / 2;
  for (let col = 0; col <= cols; col++) {
    const x = offset + col * spacing;
    const y = topMargin + row * spacing;
    World.add(world, Bodies.circle(x, y, pegRadius, {
      isStatic: true,
      render: { fillStyle: '#D4A843' }
    }));
  }
}

// Dropar bola
function dropBall() {
  const ball = Bodies.circle(width / 2, 0, ballRadius, {
    restitution: 0.5,
    friction: 0.1,
    render: { fillStyle: '#FFD700' }
  });
  World.add(world, ball);
}
```

**Pros:** Fisica realista, bounce natural, colisao precisa, ecossistema maduro (v0.19+, atualizado 2024).
**Contras:** Resultado NAO deterministico — a mesma configuracao pode gerar paths diferentes pela natureza da simulacao de fisica. Isso CONFLITA com Provably Fair que exige resultados reproduziveis.

### Opcao B - Canvas 2D Manual (Sem Fisica)

Implementacao customizada sem engine de fisica. A bola segue um path PRE-CALCULADO pelo algoritmo Provably Fair. Cada bounce e uma decisao binaria (left/right) determinada pelo hash, e a animacao e apenas visual — o resultado ja esta decidido antes da bola comecar a descer.

**Como funciona:**
1. Algoritmo Provably Fair gera N numeros aleatorios (1 por row)
2. Cada numero determina se a bola vai Left ou Right naquele nivel
3. O path completo e calculado ANTES da animacao
4. Canvas 2D anima a bola seguindo o path pre-calculado com easing e bounce visual

**Pros:** 100% deterministico (compativel com Provably Fair), leve, controle total sobre timing e easing.
**Contras:** Bounce pode parecer "programado" se a animacao nao for bem feita. Precisa de easing e jitter visual para parecer natural.

### Opcao C - Framer Motion + DOM (Sem Canvas)

Usar Framer Motion para animar um elemento DOM (<div> ou <img>) descendo pelo board. Cada nivel e uma animacao de translate com spring physics.

**Pros:** Ja instalado no projeto. Spring physics do Framer Motion pode simular bounce natural. Integracao nativa com React.
**Contras:** Performance com muitos elementos DOM (pegs como divs). Nao escala bem para autobet rapido (muitas bolas simultaneas).

## 4.2 Decisao Tecnica

**Escolha: Canvas 2D Manual com path pre-calculado (Opcao B) + Framer Motion para UI ao redor.**

A razao e simples e definitiva: **Provably Fair exige determinismo**. Se usarmos Matter.js (Opcao A), o resultado de cada drop depende de micro-variacoes na simulacao de fisica — o MESMO seed poderia produzir paths diferentes em execucoes diferentes, tornando impossivel a verificacao pos-jogo. Esta e a mesma abordagem que Stake, BGaming, e todos os cassinos crypto usam: o RNG determina o resultado, e a animacao MOSTRA o resultado de forma visualmente satisfatoria. A bola nao "escolhe" para onde ir — ela segue um caminho pre-definido com bounce animations que criam a ilusao de fisicade verdade.

O Canvas 2D sera usado para o board e a animacao da bola. Framer Motion sera usado para os controles de UI, overlays, modais, e transicoes de tela.

## 5.4 Riscos e Mitigacoes

| Risco | Impacto | Mitigacao |
|-------|---------|-----------|
| Canvas pesado no NUI com autobet rapido (muitas bolas) | FPS drop em PCs fracos | Limitar a 1 bola por vez no board. Autobet dropa a proxima so quando a anterior aterrissar. Max ~3 drops/segundo |
| Web Audio no CEF pode ter latency | Sons dessincronizados | Pre-load todos os sons no primeiro mount. Usar AudioBufferSourceNode (nao HTMLAudioElement) para latencia minima |
| GPU compartilhada GTA + NUI | Animacao travada | Canvas com dimensoes fixas (nao reescalar a cada frame). Usar `will-change: transform` nos elementos animados. Evitar filter/blur no Canvas |
| fetchNui latency para Provably Fair | Delay entre click e animacao | Gerar o resultado no server ANTES de devolver ao client. O client recebe resultado + path de uma vez, e so anima |

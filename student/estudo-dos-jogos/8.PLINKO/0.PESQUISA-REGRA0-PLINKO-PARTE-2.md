## 8.1 Algoritmo

HMAC-SHA256 — o mesmo padrao usado pelo Stake, BGaming, e todos os cassinos crypto de referencia. O algoritmo usa 3 variaveis (server seed, client seed, nonce) para gerar bytes aleatorios que determinam cada bounce da bola.

## 8.2 Fluxo Completo (10 Passos)

| Passo | Momento | Acao | Detalhe |
|:-----:|---------|------|---------|
| 1 | ANTES de jogar | Server gera server_seed | 16 bytes criptograficamente seguros via crypto.randomBytes(16).toString('hex') |
| 2 | ANTES de jogar | Server calcula hash do seed | server_seed_hash = SHA256(server_seed). Este hash e MOSTRADO ao jogador |
| 3 | ANTES de jogar | Client gera client_seed | String aleatoria de 12 caracteres. Jogador pode editar/trocar a qualquer momento |
| 4 | ANTES de jogar | Nonce inicia em 0 | Incrementa +1 a cada drop com o mesmo par de seeds |
| 5 | DURANTE o drop | Server calcula HMAC | hmac = HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":" + round). Gera 32 bytes |
| 6 | DURANTE o drop | Extrai floats do HMAC | Cada 4 bytes consecutivos do HMAC viram 1 float (0.0 a 1.0). Para N rows, precisa de N floats |
| 7 | DURANTE o drop | Float vira direcao | Se float < 0.5 → bola vai LEFT. Se float >= 0.5 → bola vai RIGHT. Repete para cada row |
| 8 | DURANTE o drop | Path determina slot | Somando os RIGHTs, o total = indice do slot na base (0 = toda esquerda, N = toda direita) |
| 9 | APOS o drop | Nonce incrementa | nonce += 1 para o proximo drop com os mesmos seeds |
| 10 | QUANDO jogador troca seeds | Server revela seed anterior | server_seed original e mostrado. Jogador pode verificar: SHA256(server_seed) == server_seed_hash que foi exibido no passo 2 |

## 8.3 Variaveis Explicadas

| Variavel | Quem controla | O que faz |
|----------|:------------:|-----------|
| **server_seed** | Servidor (oculto) | Semente aleatoria do servidor. NAO pode ser vista pelo jogador ate a troca de seeds. Garante que o servidor nao pode mudar o resultado depois |
| **client_seed** | Jogador | Semente fornecida pelo jogador. Adiciona aleatoriedade que o servidor nao controla. O jogador pode trocar a qualquer momento |
| **nonce** | Automatico | Contador que incrementa a cada drop. Garante que mesmo com seeds fixos, cada drop e unico. Comeca em 0 e sobe |
| **server_seed_hash** | Servidor (publico) | SHA256 do server_seed. Exibido ANTES do jogo como prova de que o servidor nao vai mudar o seed. E o "compromisso" do servidor |

## 8.4 Conversao Hash para Path da Bola

O processo de converter o HMAC em direcoes da bola funciona assim:

1. Gerar o HMAC: `hmac = HMAC_SHA256(server_seed, client_seed + ":" + nonce)`
2. O HMAC tem 32 bytes (256 bits)
3. Para cada row, consumir 4 bytes e converter em float:
   ```
   float = (byte[0]*256^3 + byte[1]*256^2 + byte[2]*256 + byte[3]) / (256^4 - 1)
   ```
4. Se float < 0.5 → LEFT, se float >= 0.5 → RIGHT
5. Exemplo com 12 rows:
   - Bytes consumidos: 12 x 4 = 48 bytes
   - Se HMAC tem 32 bytes, gerar rounds adicionais: HMAC_SHA256(server_seed, client_seed + ":" + nonce + ":" + round)
   - Round 0 = primeiros 32 bytes, Round 1 = proximos 32 bytes, etc.
6. Path final: array de 12 direcoes, ex: [R,L,R,R,L,R,L,L,R,R,L,R]
7. Slot = contagem de R's = 7 → multiplicador do slot 7 na tabela correspondente ao risk level

**Exemplo concreto:**
- Server seed: `a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4`
- Client seed: `player123abc`
- Nonce: `42`
- HMAC = HMAC_SHA256("a1b2c3d4...", "player123abc:42:0")
- Primeiros 48 bytes geram 12 floats
- Floats: [0.72, 0.31, 0.89, 0.45, 0.62, 0.18, 0.77, 0.53, 0.41, 0.95, 0.22, 0.68]
- Path: [R, L, R, L, R, L, R, R, L, R, L, R] → 7 R's → Slot 7
- Com 12 rows Medium Risk, Slot 7 = 0.5x multiplicador

## Problema 1: Animacao CSS 3D preserve-3d Inconsistente no CEF do FiveM

**Sintoma:** O NUI do FiveM usa Chromium Embedded Framework (CEF), que em builds mais antigas (pre-FiveM b2802) pode renderizar CSS 3D preserve-3d de forma inconsistente — faces do cubo desaparecem, backface-visibility nao funciona, ou perspective distorce a profundidade dos dados. O jogador ve dados "achatados" ou com faces transparentes durante o giro.

**Causa tecnica:** O CEF compartilha a GPU com o renderer do GTA V. Quando o GTA esta usando a GPU intensivamente (muitos jogadores na tela, ambiente detalhado), o compositor CSS do CEF pode falhar em manter o estado 3D das transformacoes. Especificamente, o `transform-style: preserve-3d` exige que o browser mantenha um contexto 3D separado para o elemento e todos os filhos, o que compete com o GTA pelo pipeline de GPU. Alem disso, builds mais antigas do FiveM usam versoes do Chromium que tem bugs conhecidos com `backface-visibility: hidden` quando combinado com animacoes rapidas de rotacao.

**Codigo conceitual — deteccao e fallback:**

```typescript
// hooks/use-dice-3d-support.ts
import { useState, useEffect } from 'react';

export function useDice3DSupport() {
  const [supports3D, setSupports3D] = useState(true);

  useEffect(() => {
    // Teste 1: preserve-3d suportado
    const testEl = document.createElement('div');
    testEl.style.transformStyle = 'preserve-3d';
    const computed = window.getComputedStyle(testEl).transformStyle;
    if (computed !== 'preserve-3d') {
      setSupports3D(false);
      return;
    }

    // Teste 2: performance check (GPU stressed?)
    const start = performance.now();
    requestAnimationFrame(() => {
      const delta = performance.now() - start;
      // Se o frame demorou mais de 50ms, GPU provavelmente estressada
      if (delta > 50) {
        setSupports3D(false);
      }
    });
  }, []);

  return supports3D;
}
```

**Solucao multi-camada:**
1. **Deteccao automatica:** Hook `useDice3DSupport()` testa preserve-3d e mede frame time. Se falhar, ativa fallback
2. **Fallback 2D:** Substituir dados CSS 3D por sprite PNG pre-renderizado do dado com rotacao CSS 2D simples (rotateZ + scale). 6 sprites (1 por face), animacao de troca rapida entre sprites simulando giro, depois para no sprite da face final. Menos impressionante visualmente mas funcional em qualquer CEF
3. **will-change: transform** em cada dado ANTES da animacao para pre-alocar o layer na GPU, e remove APOS para liberar memoria
4. **Limite de 3 dados:** Nunca renderizar mais que 3 cubos 3D simultaneamente (limitacao do design, nao apenas da performance)

**Impacto:** ~40 linhas (hook + condicional no componente). Zero dependencia extra.

## Problema 2: Sincronizacao da Animacao com Resultado Server-Side

**Sintoma:** O servidor calcula o resultado dos 3 dados via Provably Fair ANTES da animacao comecar no client. Existe um atraso de rede entre o client enviar a aposta e o servidor retornar o resultado (tipicamente 50-200ms em servidores FiveM bem configurados, podendo chegar a 500ms+ em servidores lotados). Se a animacao comecar ANTES do resultado chegar, os dados precisam "esperar" o resultado no meio do giro — quebrando a fluidez. Se comecar DEPOIS, o jogador fica olhando para uma tela parada por meio segundo apos clicar "ROLAR".

**Causa tecnica:** O fluxo NUI→Lua→Server→Lua→NUI passa por 4 saltos de comunicacao. Cada RegisterNUICallback no client Lua dispara um TriggerServerEvent que e processado no server handler JS, que retorna via TriggerClientEvent que dispara SendNUIMessage de volta ao React. Em servidores com 64+ jogadores, o event bus do FiveM pode enfileirar eventos, adicionando latencia imprevisivel.

**Codigo conceitual — pre-animacao com revelacao adiada:**

```typescript
// Estrategia: animacao de "shake" imediata, giro real apos resultado chegar

interface DiceState {
  fase: 'idle' | 'shaking' | 'rolling' | 'landing' | 'result';
  resultado: [number, number, number] | null;
  tempoShakeInicio: number;
}

function handleRollClick(apostar: () => void) {
  // 1. Imediatamente: iniciar animacao de shake (dados tremem no lugar)
  setDiceState({ fase: 'shaking', resultado: null, tempoShakeInicio: Date.now() });

  // 2. Enviar aposta ao servidor via fetchNui
  apostar(); // dispara NUI callback → Lua → Server

  // 3. O resultado chega via useNuiEvent('casino:dice:result')
  // Quando chegar, a funcao onResultReceived e chamada
}

function onResultReceived(dados: [number, number, number]) {
  const tempoShake = Date.now() - diceState.tempoShakeInicio;
  const shakeMinimo = 800; // ms — shake deve durar no minimo 800ms para build anticipation

  if (tempoShake < shakeMinimo) {
    // Resultado chegou rapido demais — continuar shake ate completar tempo minimo
    setTimeout(() => {
      setDiceState({ fase: 'rolling', resultado: dados, tempoShakeInicio: 0 });
    }, shakeMinimo - tempoShake);
  } else {
    // Resultado chegou apos o shake minimo — iniciar roll imediatamente
    setDiceState({ fase: 'rolling', resultado: dados, tempoShakeInicio: 0 });
  }
}
```

**Solucao:**
1. **Shake imediato:** Ao clicar "ROLAR DADOS", os dados comecam a tremer imediatamente (Framer Motion x/y vibration aleatoria). Isto da feedback instantaneo ao jogador e constroi anticipation
2. **Shake minimo 800ms:** Mesmo que o servidor responda em 50ms, a animacao de shake dura pelo menos 800ms para que a experiencia nao pareca "instantanea" demais — o jogador precisa sentir que algo esta acontecendo
3. **Roll apos resultado:** Os dados so comecam o giro real (rotateX/Y/Z) quando o resultado chega do servidor. Os angulos finais de rotacao sao calculados para que a face correta fique virada para cima
4. **Timeout de seguranca:** Se o resultado nao chegar em 5 segundos, exibir mensagem "Conexao lenta, tente novamente" e cancelar a rodada (sem debitar GCoin — validacao server-side impede cobranca sem resultado)

**Impacto:** ~30 linhas de logica de estado + ~20 linhas de timeout. Zero dependencia.

## Problema 3: Calculo dos Angulos Finais de Rotacao para Face Especifica

**Sintoma:** O servidor retorna qual face cada dado deve mostrar (1-6). A animacao CSS 3D precisa rotacionar o cubo ate a face correta ficar virada para o viewer. O problema e que existem MULTIPLAS combinacoes de rotateX + rotateY que resultam na mesma face visivel (infinitas, na verdade, considerando multiplas voltas). Escolher angulos errados faz o dado parar em posicao estranha, com a face certa visivel mas o dado "inclinado" ou "de cabeca para baixo".

**Causa tecnica:** Um cubo CSS 3D tem 6 faces posicionadas com rotacoes especificas:
- Face 1 (frente): rotateY(0deg) translateZ(50px)
- Face 2 (esquerda): rotateY(-90deg) translateZ(50px)
- Face 3 (topo): rotateX(90deg) translateZ(50px)
- Face 4 (baixo): rotateX(-90deg) translateZ(50px)
- Face 5 (direita): rotateY(90deg) translateZ(50px)
- Face 6 (tras): rotateY(180deg) translateZ(50px)

Para "mostrar" uma face, o container do cubo precisa rotacionar no sentido OPOSTO. Por exemplo, para mostrar a face 5 (direita), o container faz rotateY(-90deg). Mas a animacao de giro adiciona N voltas completas (360° × N) para dar a impressao de que o dado esta realmente girando.

**Codigo conceitual — mapeamento face→angulos:**

```typescript
// lib/dice-angles.ts

// Angulos finais para mostrar cada face (cubo de 100px, faces a 50px)
const FACE_ROTATIONS: Record<number, { x: number; y: number }> = {
  1: { x: 0,    y: 0 },    // frente
  2: { x: 0,    y: 90 },   // esquerda vira para frente
  3: { x: -90,  y: 0 },    // topo vira para frente
  4: { x: 90,   y: 0 },    // baixo vira para frente
  5: { x: 0,    y: -90 },  // direita vira para frente
  6: { x: 0,    y: 180 },  // tras vira para frente
};

// Adiciona N voltas aleatorias para efeito visual de giro
function calcularAngulosFinais(face: number): { rotateX: number; rotateY: number; rotateZ: number } {
  const base = FACE_ROTATIONS[face];
  const voltasX = (Math.floor(Math.random() * 3) + 2) * 360; // 2-4 voltas completas
  const voltasY = (Math.floor(Math.random() * 3) + 2) * 360;
  const voltasZ = (Math.floor(Math.random() * 2) + 1) * 360; // 1-2 voltas no Z para naturalidade

  return {
    rotateX: base.x + voltasX,
    rotateY: base.y + voltasY,
    rotateZ: voltasZ, // Z retorna a 0 no final (mod 360 = 0) mas gira durante
  };
}

// Uso no componente:
// const angulos = calcularAngulosFinais(resultado.dado1); // ex: face 5
// animate={{ rotateX: angulos.rotateX, rotateY: angulos.rotateY, rotateZ: angulos.rotateZ }}
// transition={{ type: 'spring', stiffness: 80, damping: 15, duration: 1.8 }}
```

**Solucao:**
1. **Mapeamento estatico** das 6 faces para angulos X/Y base
2. **Voltas aleatorias** adicionadas (2-4 em X, 2-4 em Y, 1-2 em Z) para que cada giro pareca diferente
3. **Spring physics** do Framer Motion com stiffness baixa (80) para desaceleracao natural no final
4. **Z-rotation** adiciona naturalidade — dados reais giram em todos os eixos, nao so X e Y

**Impacto:** ~25 linhas (mapeamento + funcao). Zero dependencia.

## Problema 4: Limitacao FiveM — Web Audio Autoplay Policy no CEF

**Sintoma:** O CEF do FiveM herda a politica de autoplay do Chromium: sons so podem ser reproduzidos apos uma interacao do usuario (click, keypress). Na primeira rodada do Dice, os sons de lightning_strike e dice_shake podem nao tocar se o jogador nao interagiu com o NUI antes (ex: abriu o painel mas nao clicou em nada dentro do jogo).

**Causa tecnica:** O Chromium 90+ bloqueia AudioContext.play() sem user gesture. O CEF do FiveM pode ou nao respeitar esta politica dependendo da build. Em testes, builds FiveM b3258+ respeitam a politica. O hook useSoundManager do projeto ja lida com isso parcialmente, mas o primeiro som da sessao pode falhar silenciosamente.

**Solucao:**
1. **Resume AudioContext no primeiro click:** Ao clicar em qualquer botao do jogo (ex: selecionar aposta), chamar `audioContext.resume()` se o estado for 'suspended'
2. **Fallback silencioso:** Se o som falhar, nao quebrar o fluxo do jogo — o som e decorativo, nao funcional
3. **Pre-load dos sons:** No mount do componente DiceGame, pre-carregar todos os 18 arquivos de som em buffers para evitar latencia no primeiro play
4. **useSoundManager existente:** O hook ja esta no projeto (hooks/use-sound-manager.ts) e trata o resume automaticamente. Basta integrar com os 18 eventos do Dice

**Impacto:** ~10 linhas de integracao. Zero dependencia nova (useSoundManager ja existe).

## 5.3 Queries Admin Uteis

```sql
-- Ver configuracao atual
SELECT * FROM casino_dice_config LIMIT 1;

-- Alterar RTP (admin)
UPDATE casino_dice_config SET rtp_target = 94.00 WHERE id = 1;

-- Desativar o jogo (manutencao)
UPDATE casino_dice_config SET is_active = 0 WHERE id = 1;

-- Top 10 jogadores que mais lucraram (ultimos 7 dias)
SELECT player_id,
       COUNT(*) AS rodadas,
       SUM(total_bet) AS total_apostado,
       SUM(total_win) AS total_ganho,
       SUM(profit) AS lucro_total
FROM casino_dice_history
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY player_id
ORDER BY lucro_total DESC
LIMIT 10;

-- Distribuicao de totais (verificar se RNG esta equilibrado)
SELECT total, COUNT(*) AS vezes,
       ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM casino_dice_history), 2) AS percentual
FROM casino_dice_history
GROUP BY total
ORDER BY total;

-- Maior payout individual
SELECT player_id, total, total_bet, total_win, lightning_numbers, created_at
FROM casino_dice_history
ORDER BY total_win DESC
LIMIT 5;

-- House profit total (ultimas 24h)
SELECT SUM(total_bet) AS apostado_total,
       SUM(total_win) AS pago_total,
       SUM(total_bet) - SUM(total_win) AS lucro_casa,
       ROUND((SUM(total_bet) - SUM(total_win)) / SUM(total_bet) * 100, 2) AS house_edge_real
FROM casino_dice_history
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 24 HOUR);
```

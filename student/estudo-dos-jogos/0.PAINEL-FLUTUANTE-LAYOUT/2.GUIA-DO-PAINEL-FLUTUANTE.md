## DECISOES DE ARQUITETURA — POR QUE CADA ESCOLHA

### Por que Painel Flutuante e nao menu fullscreen?
Menus fullscreen desconectam o jogador do mundo GTA — ele "sai" do RP para entrar numa UI. O painel flutuante (80vw × 78vh max) mantém o GTA visivel atras do overlay blur(3px), lembrando o jogador que ele ESTA num cassino virtual dentro de Los Santos. A borda arredondada (border-radius 18px) e o fundo semi-transparente criam a metafora de "janela para outro mundo" — o jogador espia o cassino sem sair do servidor. Stake.com e BC.Game tambem usam containers com bordas, nunca fullscreen puro.

### Por que BentoGrid com 5 layouts diferentes e nao grid unico?
Um grid responsivo generico (auto-fill repeat) funciona para 5 jogos. Para 22 jogos em 5 categorias, cada aba precisa de identidade propria: Cassino tem hero grande (Roleta no pedestal) + 6 cards + 2 banners slim porque eh a aba principal e precisa impressionar; Arcade tem hero 60% esquerda + 4 cards 2x2 porque os jogos sao visualmente ricos e precisam de espaço; PvP tem 3 cards grandes lado a lado porque sao apenas 3 jogos de alto impacto social; Loja tem 4 cards grid 2x2 porque sao itens utilitarios sem hierarquia; Eventos tem SpotlightLayout com 3 cards e efeito spotlight no selecionado porque eventos sao rotativos e o destaque muda. Cada layout conta uma historia visual diferente. Nenhum cassino FiveM faz isso — todos usam o mesmo grid para tudo.

### Por que Canvas 2D para particulas e nao CSS puro?
Particulas CSS (`@keyframes` em divs absolutas) funcionam para 5-10 elementos, mas escalam pessimamente — 50 divs com box-shadow e transform animados bloqueiam o main thread do CEF. Canvas 2D com requestAnimationFrame renderiza 50+ particulas num unico elemento DOM, com compositing acelerado por GPU. O custo eh 1 node no DOM ao inves de 50. No FiveM onde o CEF compartilha GPU com o GTA V, cada DOM node a menos eh frame a mais no jogo. OffscreenCanvas (Chrome 69+, viavel no CEF M103) pode mover o draw loop para Worker thread, liberando o main thread completamente.

### Por que Framer Motion e nao GSAP para animacoes?
GSAP eh mais poderoso para timelines complexas (sequencias de 30+ steps), mas cobra licenca comercial e pesa ~25KB gzip. Framer Motion eh gratuito, pesa ~32KB mas JA ESTA instalado no projeto (dependencia existente), e oferece `layoutId` — a feature que permite animar AUTOMATICAMENTE a transicao de posicao/tamanho quando um elemento muda de lugar no DOM. Para o painel, layoutId eh critico em dois pontos: (1) indicador de aba ativa no Dock que DESLIZA suavemente entre abas ao inves de pular, e (2) transicao card→modal onde o card do jogo pode "morphar" para o modal fullscreen com shared layout animation. Nenhuma dessas animacoes eh trivial de implementar com GSAP sem codigo manual extenso. AnimatePresence (entrada/saida com unmount) tambem eh nativa do Framer.

### Por que 5 abas e nao menu lateral ou dropdown?
O Dock com 5 abas na base do painel segue o padrao de navegacao mobile que TODOS os cassinos online migraram entre 2023-2025: Stake tem bottom tabs em mobile, BC.Game tem bottom bar com 5 icones, CSGOEmpire tem tab bar fixa. A razao eh ergonomica — o polegar alcanca a base da tela mais facilmente que um sidebar ou header. No NUI FiveM landscape (1280×720 a 1920×1080), a base do painel eh a zona de menor obstrucao visual (nao cobre os cards). Cada aba tem icone PNG (X21: PNG customizado, nao Lucide) + label bilingue + tooltip, criando uma barra de navegacao que funciona por reconhecimento visual, nao por leitura.

### Por que GCoin como moeda isolada e nao dinheiro direto do servidor?
Se o cassino usasse dinheiro do servidor diretamente, um exploit no Slots que gerasse G$1.000.000 injetaria dinheiro na economia do servidor — inflacao imediata, admin furioso. GCoin eh uma ficha intermediaria: o jogador COMPRA GCoin com dinheiro do servidor (deposito), joga com GCoin, e SACA GCoin de volta para dinheiro (retirada). 1. Jogador no mundo GTA se aproxima da mesa de cassino ou digita /cassino no chat

3. React recebe mensagem → introStage 0→1: overlay escuro fade in (500ms)
4. Stage 2 (150ms): painel flutuante aparece com scale+fade (800ms spring)
5. Stage 3 (600ms): letreiro marquee desliza do topo
6. Stage 4 (850ms): dock com 5 abas sobe da base — painel PRONTO
7. fetchNui("getSaldo") → saldo GCoin aparece no dock — "G$ 2.450,00"
8. Aba CASSINO ativa por padrao: CasinoLayout com hero Roleta + 6 cards + 2 banners
9. Jogador move mouse → spotlight cursor segue, particulas douradas reagem (repel), cards brilham no hover
10. Jogador clica no card "Poker" → GameModal abre com AnimatePresence (scale 0.9→1, 400ms)
11. Modal mostra imagem dourada + logo BR + botao JOGAR AGORA + particulas internas
12. Jogador clica JOGAR AGORA → GameModal fecha → PokerGame renderiza dentro do painel (z-60, fade+scale, 300ms) → particulas PAUSAM
13. Jogador joga Poker → clica VOLTAR → PokerGame sai (exit animation) → painel reaparece → particulas RETOMAM
14. Jogador pressiona ESC → fetchNui("closeCasino") → SetNuiFocus(false) → overlay fade out → volta ao GTA

### Fluxo 2 — Trocar Aba e Explorar Eventos

1. Jogador esta na aba CASSINO vendo o grid de jogos
2. Clica na aba ARCADE no Dock → indicador desliza (layoutId spring)
3. Cards CASSINO saem (AnimatePresence exit, 200ms) → Cards ARCADE entram (stagger 0.05s)
4. Layout muda: CasinoLayout → ArcadeLayout (hero 60% + 4 cards 2x2)

9. Clica JOGAR AGORA no Giveaways → modal → abre jogo ou mostra ComingSoon

### Fluxo 3 — Depositar GCoins e Verificar Saldo

1. Jogador ve saldo "G$ 0,00" no Dock — precisa depositar

6. Server: verifica saldo, desconta dinheiro, soma GCoin, insere em casino_transactions
7. Response: {success: true, newBalance: 5000} → saldo no Dock anima countup "G$ 0 → G$ 5.000"

9. Modal fecha → jogador pode jogar

## 3.1 Limitações Técnicas do NUI/CEF para Jackpot PvP

O Jackpot PvP tem um desafio que jogos single-player (Crash, Slots) não têm: **sincronização em tempo real entre múltiplos jogadores**. Cada depósito precisa atualizar o donut chart de TODOS os jogadores conectados simultaneamente.

### Comunicação NUI ↔ Server para Multiplayer

```
Jogador A deposita 500 GCoin
  → Lua Client A: TriggerServerEvent('casino:jackpot:deposit', 500)
  → JS Server Handler: valida saldo, registra depósito, recalcula tickets
  → Server broadcast: TriggerClientEvent('casino:jackpot:update', -1, potData)
  → Lua Client de TODOS os jogadores: SendNUIMessage({ type: 'pot_update', data: potData })
  → React de TODOS: atualiza donut chart, lista de jogadores, timer
```

**Questão-chave:** A comunicação `TriggerClientEvent(-1, ...)` envia para TODOS os players do servidor, não só os que estão no cassino. Solução: filtrar no client Lua — só repassar para NUI se o jogador está com o painel do cassino aberto.

### Performance do Canvas 2D no NUI

| Cenário | FPS Impact | Viável? |
|---------|:----------:|:-------:|
| Donut estático (conic-gradient CSS) | ~0% | ✅ |
| Donut Canvas 2D com 2-5 seções | <1% | ✅ |
| Donut Canvas 2D com 10-20 seções | ~1-2% | ✅ |
| Spinner animado Canvas 2D (5-12s, 60fps) | ~2-3% | ✅ |
| Spinner animado + partículas confetti | ~3-5% | ✅ |
| WebGL 3D spinner | ~5-10% | ⚠️ Arriscado |

**Conclusão:** Canvas 2D é perfeitamente viável no NUI para o Jackpot. Mesmo com spinner animado + confetti de vitória, o impacto fica abaixo de 5% FPS. WebGL 3D é desnecessário e arriscado.

### Latência de Atualizações em Tempo Real

- FiveM events (TriggerClientEvent) têm latência de ~50-150ms dependendo do ping do jogador
- Para o donut chart: 50-150ms de delay é **imperceptível** — o jogador vê a atualização quase instantaneamente
- Para o spinner: resultado é determinado server-side ANTES da animação. O server envia o winning ticket para TODOS os clients, que animam localmente. Não há problema de dessincronização no resultado, apenas na animação visual (aceitável)

## 3.4 Especificidades do Protocolo Anti-Snipe

"Snipe" é quando um jogador espera até os últimos segundos para depositar um valor alto, não dando tempo para outros reagirem. Solução padrão:

```
SE timer_restante < ANTI_SNIPE_THRESHOLD (padrão: 10s)
  E novo_deposito chega
ENTÃO
  timer_restante = ANTI_SNIPE_THRESHOLD
  broadcast: { type: 'timer_reset', newTime: ANTI_SNIPE_THRESHOLD }
  UI: flash visual no timer (dourado → verde pulsante)
```

**Limite de resets:** Máximo de 3 resets anti-snipe por round. Após 3, o timer continua normalmente. Isso evita trolling infinito.

## O que a pesquisa trouxe de NOVO e CONCRETO

1. **Mecânica exata do pot-style jackpot:** Não é jackpot progressivo (como nas slots). É um jogo PvP com pot compartilhado, tickets proporcionais, spinner visual, e winner-takes-all. A pesquisa definiu: timer 60s, anti-snipe com reset de 10s (máx 3x), house edge 5%, min/max depósito configurável, mínimo 2 jogadores.

2. **Provably Fair específico para jackpot PvP:** O sistema usa client seeds dos 3 primeiros depositantes (não apenas 1 jogador), o que torna o resultado dependente de múltiplas partes. Código HMAC_SHA256 real extraído do Stake.com e adaptado para jackpot com ticket system.

3. **Vulnerabilidade Math.random():** O caso CSGOJackpot prova que usar Math.random() é uma falha de segurança crítica. Essa lição define que o handler server-side DEVE usar crypto.randomBytes(), não Math.random().

4. **GSAP timeline de 2 fases para spinner:** Aceleração (power1.in, 2s) → Desaceleração (power3.out, 6s). Padrão extraído de múltiplos fórums GSAP e tutoriais. Alternativa dramática: back.out(1.1) para overshoot sutil.

5. **Donut chart em tempo real via conic-gradient CSS:** Técnica leve que não precisa de Canvas para a fase de depósitos. Canvas reservado para o spinner animado. Reduz complexidade.

6. **Nenhum script FiveM tem Jackpot PvP:** Diferencial competitivo absoluto. O mercado FiveM tem slots, blackjack, poker, roulette, mas ZERO scripts de pot-style jackpot. O Blackout Casino seria pioneiro.

7. **Anti-snipe é obrigatório:** Sem anti-snipe, jogadores com saldo alto podem "snipar" o pot nos últimos 2s, criando experiência injusta. Timer reset de 10s com máximo de 3 resets é o padrão da indústria.

8. **Cores por jogador são atribuídas automaticamente:** Cada jogador recebe uma cor única da paleta ao depositar. Essa cor aparece no donut, na lista, e no spinner. Paleta de 10 cores com boa distinção visual no dark theme.

## O que teríamos criado SEM a pesquisa

Sem esta pesquisa, provavelmente teríamos:
- Confundido "jackpot" com jackpot progressivo de slots (ERRO)
- Usado Math.random() no handler (VULNERABILIDADE CRÍTICA)
- Ignorado anti-snipe (experiência de jogo injusta)
- Feito spinner genérico sem timeline GSAP calibrada (animação pobre)
- Não implementado provably fair com client seeds múltiplos (menor transparência)
- Pensado que já existem scripts similares no FiveM (quando na verdade somos pioneiros)
- Ignorado o donut chart como representação visual do pot (UI genérica)

## 1.1 Definição Precisa do Jogo

O "Jackpot" (também chamado "Pot", "Skin Jackpot", "Coin Jackpot") é um jogo **PvP multiplayer** onde:

1. **Múltiplos jogadores** (2-N, sem limite rígido) depositam moedas/skins num **pot compartilhado**
2. Cada jogador recebe **"tickets"** proporcionais ao valor depositado
3. Quando o timer expira (tipicamente 60-120 segundos) ou o pot atinge um cap, **um spinner/roda gira**
4. O spinner seleciona **1 vencedor** com base na proporção de tickets
5. O vencedor leva **todo o pot** menos a taxa da casa (3-10%)

**Exemplo concreto (extraído da pesquisa):**
- Pot total: R$ 1.000,00
- Jogador A depositou R$ 500 → 50% de chance
- Jogador B depositou R$ 300 → 30% de chance
- Jogador C depositou R$ 200 → 20% de chance
- Spinner gira → para no Jogador B → Jogador B leva R$ 950 (5% taxa casa)

## 1.2 Variantes Encontradas

## 1.6 Lição Crítica de Segurança: O Caso CSGOJackpot

O blog de nickler (pesquisador de segurança) documentou uma vulnerabilidade grave no CSGOJackpot original: o site usava **Math.random()** do JavaScript para gerar o ticket vencedor. Math.random() usa o algoritmo **xorshift128+**, que é determinístico — dados 2 outputs consecutivos, é possível prever TODOS os outputs futuros.

**Impacto:** Um atacante podia prever o percentual vencedor antes de depositar, garantindo vitória em toda rodada.

**Lição para o Blackout Casino:**
- **NUNCA** usar Math.random() para resultado de jogo
- **SEMPRE** usar crypto.randomBytes() ou HMAC_SHA256 com server seed + client seed
- Resultado gerado **server-side** (handler FiveM), **NUNCA** no client
- Hash do server seed publicado ANTES da rodada, seed revelado DEPOIS
- Ticket vencedor = `HMAC_SHA256(server_seed, client_seeds + ":" + nonce)` → converter para inteiro → módulo total_tickets

## 2.1 Renderização do Spinner/Donut

### Opção A: Canvas 2D (Recomendada)
- **Técnica:** Canvas `arc()` para desenhar donut chart com seções proporcionais. Cada seção = ângulo proporcional ao depósito do jogador.
- **Animação do spin:** `requestAnimationFrame` com rotação incremental + desaceleração via easing function customizada, ou GSAP animando propriedade `rotation` de um objeto JS e re-renderizando no Canvas a cada frame.
- **Performance FiveM NUI:** Canvas 2D é leve (<2% FPS). Já validado no projeto (usamos Canvas no GoldParticles).
- **Vantagem:** Controle total sobre aparência, cores, gradientes, glow. Sem dependência de lib externa.

### Opção B: SVG/DOM com GSAP
- **Técnica:** Criar donut como SVG `<path>` (arcos) ou usar componente React (recharts PieChart, d3.js pie). Animar rotação com GSAP `gsap.to(element, { rotation: targetAngle, duration: 8, ease: "power3.out" })`.
- **Referência real:** DEV.to tutorial usou recharts PieChart + GSAP para wheel of fortune funcional.
- **Performance:** SVG com poucos elementos (2-10 jogadores) é leve. Pode ter micro-lag com muitos jogadores (>20).
- **Vantagem:** Mais fácil de estilizar com CSS, tooltips nativos ao hover em cada seção.

### Opção C: CSS Conic-gradient (Backup simples)
- **Técnica:** `background: conic-gradient(#cor1 0% 50%, #cor2 50% 80%, #cor3 80% 100%)` num div circular. Animar `transform: rotate()` com CSS transition ou GSAP.
- **Limitação:** Menos controle visual, sem glow por seção, sem interatividade fina.

**DECISÃO:** Canvas 2D (Opção A) como engine primária, com fallback SVG (Opção B) para acessibilidade. O donut chart estático (fase de depósitos) pode usar conic-gradient CSS para simplicidade; o spinner animado (fase de resultado) usa Canvas 2D via GSAP para controle preciso.

## 2.3 Provably Fair — Implementação Técnica

Baseado na pesquisa em Stake.com, Rocketpot.io e Medium/NSoft:

```javascript
// SERVER-SIDE (handler FiveM) — Gerar resultado
const crypto = require('crypto');

function generateJackpotResult(serverSeed, clientSeeds, nonce, totalTickets) {
  // clientSeeds = seeds dos 3 primeiros depositantes concatenados
  const combinedClientSeed = clientSeeds.join(':');
  
  // HMAC_SHA256 para gerar hash determinístico
  const hmac = crypto.createHmac('sha256', serverSeed);
  hmac.update(`${combinedClientSeed}:${nonce}`);
  const hash = hmac.digest('hex');
  
  // Converter primeiros 8 chars hex para inteiro
  const decimalValue = parseInt(hash.substring(0, 8), 16);
  
  // Módulo pelo total de tickets para obter ticket vencedor
  const winningTicket = decimalValue % totalTickets;
  
  return { winningTicket, hash, serverSeed, combinedClientSeed, nonce };
}

// ANTES do round: publicar SHA256(serverSeed) para verificação
function getServerSeedHash(serverSeed) {
  return crypto.createHash('sha256').update(serverSeed).digest('hex');
}
```

**Fluxo Provably Fair completo:**
1. Server gera `server_seed` (64 chars hex aleatórios via `crypto.randomBytes(32)`)
2. Server publica `SHA256(server_seed)` antes do round iniciar
3. Jogadores depositam. Seeds dos 3 primeiros depositantes são usados como `client_seeds`
4. Quando pot fecha: `winning_ticket = HMAC_SHA256(server_seed, client_seeds:nonce) % total_tickets`
5. Spinner anima até o ticket vencedor
6. Após resultado: server revela `server_seed` unhashed
7. Qualquer jogador pode verificar: `SHA256(server_seed_revelado) == hash_publicado_antes`

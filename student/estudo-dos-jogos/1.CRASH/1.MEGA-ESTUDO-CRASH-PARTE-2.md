## Problema 1: Sincronizacao Multiplicador Client-Server

**Sintoma:** O multiplicador no NUI do jogador pode estar ligeiramente diferente do multiplicador no server no momento do cashout, causando discrepancia no payout.

**Causa tecnica:** O multiplicador e funcao do tempo (`multiplier = e^(rate * elapsed)`). O client inicia timer ao receber "RISING" do server. Latencia de rede (~20-100ms no FiveM local) significa que o client comeca a contar 20-100ms depois do server. Quando jogador clica CASHOUT no multiplicador visual "2.47x", o server pode estar em "2.50x" ou "2.44x".

**Solucao: timestamp-based + tolerancia**
```javascript
// Client envia timestamp do clique junto com cashout
const cashoutRequest = {
  roundId,
  clientTimestamp: Date.now(),
  clientMultiplier: currentMultiplier // valor visual no momento do clique
};

// Server calcula multiplicador no clientTimestamp
function validateCashout(round, bet, request) {
  const serverElapsed = request.clientTimestamp - round.startedAt;
  const serverMultiplier = calcMultiplier(serverElapsed);

  // Tolerancia de 200ms (janela de latencia aceitavel)
  const tolerance = calcMultiplier(serverElapsed + 200) - serverMultiplier;
  const diff = Math.abs(serverMultiplier - request.clientMultiplier);

  if (diff > tolerance) {
    // Diferenca suspeita — possivel cheat. Usar multiplicador do server
    return { multiplier: serverMultiplier, flagged: true };
  }

  // Dentro da tolerancia — usar o MENOR valor (favorece a casa levemente)
  const finalMultiplier = Math.min(serverMultiplier, request.clientMultiplier);
  return { multiplier: finalMultiplier, flagged: false };
}
```

**Impacto:** ~20 linhas extras no handler. Latencia percebida: zero (client mostra resultado imediato, server valida em background).

## Problema 2: Crash Instantaneo (1.00x) — UX de Frustacao

**Sintoma:** Com house edge 1%, ~1% dos rounds crasham instantaneamente em 1.00x. Todos os jogadores perdem tudo. Pode parecer "bug" ou "trapaça" para jogadores que nao entendem a mecanica.

**Causa tecnica:** E comportamento intencional — o house edge e implementado como chance de crash instantaneo. Sem isso, RTP seria 100%.

**Solucao multi-camada:**
1. **Visual distintivo para crash 1.00x:** Em vez de mostrar curva subindo e crashando, mostrar imediatamente "CRASH INSTANTANEO!" com cor amarela (#FFD700) em vez de vermelha. Deixa claro que nao foi bug.
2. **Badge especial no historico:** Crash 1.00x renderizado com icone de raio e cor diferente (amarelo vs vermelho para crashes normais).
3. **Tooltip educativo:** Na primeira vez que o jogador presencia crash 1.00x, tooltip explica: "Crashes instantaneos acontecem em ~1% dos rounds — faz parte da mecanica do jogo."
4. **Modal Provably Fair acessivel:** Botao "Verificar" proeminente apos crash 1.00x para que jogador confirme que nao foi manipulado.

## Problema 3: Auto Bet com Martingale — Risco de Perda Rapida

**Sintoma:** Jogador configura Auto Bet com "dobrar apos derrota" (Martingale). Apos 7 derrotas consecutivas, aposta inicial de 100 GC virou 12.800 GC. Pode zerar o saldo sem perceber.

**Causa tecnica:** Martingale e estrategia matematicamente perdedora a longo prazo. Em Crash com ~52% de rounds abaixo de 1.5x, sequencias de derrotas sao comuns.

**Solucao: limites e avisos**
```javascript
// No handler casino:crash:autoBet — validacao de seguranca
function validateAutoBet(config, playerBalance) {
  // Calcular pior cenario (todas derrotas com increase)
  let worstCaseLoss = 0;
  let currentBet = config.amount;
  const increaseMap = { 'keep': 0, '25': 0.25, '50': 0.50, '100': 1.0, 'double': 1.0 };
  const increaseRate = increaseMap[config.increaseOnLoss] || 0;

  for (let i = 0; i < config.rounds; i++) {
    worstCaseLoss += currentBet;
    currentBet = Math.min(currentBet * (1 + increaseRate), config.maxBet || 10000);
  }

  if (worstCaseLoss > playerBalance * 2) {
    return {
      ok: false,
      warning: 'ATENCAO: No pior cenario, voce pode perder ate ' +
               worstCaseLoss + ' GC em ' + config.rounds + ' rounds.'
    };
  }

  return { ok: true, worstCaseLoss };
}
```

**No NUI:** Modal de confirmacao antes de iniciar Auto Bet mostrando pior cenario calculado. Jogador deve clicar "Entendo os riscos" para prosseguir.

## Problema 4: Anti-Cheat — Client-Side Manipulation

**Sintoma:** Jogador mal-intencionado pode abrir DevTools no NUI, inspecionar o codigo, e tentar enviar cashout requests fraudulentos.

**Causa tecnica:** NUI roda em Chromium embarcado. DevTools podem ser habilitados. JavaScript e inspecionavel.

**Solucao: server e autoridade unica**
- O crash point NUNCA e enviado ao client antes do crash acontecer
- O server hash (compromisso) e enviado, mas sem o seed original e impossivel calcular o crash point
- Toda validacao de cashout acontece no server: `if (currentMultiplier < crashPoint) → aceita; else → rejeita`
- O client apenas ANIMA — nao toma decisoes. Se client enviar cashout em multiplicador que ja crashou, server rejeita
- Rate limit: maximo 1 cashout request por round por jogador
- Cooldown: minimo 500ms entre bet e cashout (evita bots ultra-rapidos)
- Log de acao suspeita: se jogador tenta cashout apos crash → registrar no audit

## 6.2 Decisao de RTP para o Blackout Casino

**RTP escolhido: 99% (house edge 1%)** — alinhado com Stake.com e BC.Game.

**Justificativa tecnica:**
- Em economia RP, GCoins sao recurso escasso. House edge de 3% (Aviator) drenaria rapido demais
- Com 1% de house edge: a cada 10.000 GC apostados, 100 GC vao para o "banco" (servidor RP). Sustentavel
- Com 3% de house edge: 300 GC de cada 10.000 — jogadores perceberiam a perda rapido e parariam de jogar
- O objetivo no RP NAO e lucro do cassino — e entretenimento e circulacao de GCoins entre jogadores
- Formula no server: `if (rawFloat < 0.01) crashPoint = 1.00` — 1% dos rounds crasham instantaneamente

**Implementacao do house edge:**
```javascript
// No server handler, apos gerar o float do Provably Fair:
const rawFloat = getFloatFromHash(hash); // 0.0 a 1.0
const HOUSE_EDGE = 0.01; // 1%

// 1% de chance de crash instantaneo (house edge)
if (rawFloat < HOUSE_EDGE) {
  crashPoint = 1.00;
} else {
  // Formula que garante distribuicao correta
  crashPoint = Math.floor((1 / (1 - rawFloat)) * 100) / 100;
}
// crashPoint minimo: 1.00, sem maximo teorico
```

## 6.4 Fluxo Economico de um Round

```
ROUND INICIA:
  1. Jogador aposta 500 GC
     -> Server: vRP.removeMoney(playerId, 500)
     -> Registra aposta no MySQL: INSERT INTO casino_crash_bets

  2. Multiplicador sobe: 1.00x -> 1.50x -> 2.00x -> 2.47x

  3a. CASHOUT em 2.00x:
     -> Server: vRP.addMoney(playerId, 500 * 2.00 = 1000)
     -> Lucro liquido: +500 GC
     -> UPDATE casino_crash_bets SET cashed_out_at = 2.00, payout = 1000

  3b. NAO FEZ CASHOUT, crash em 2.47x:
     -> Dinheiro ja foi removido no passo 1
     -> Perda: -500 GC
     -> UPDATE casino_crash_bets SET cashed_out_at = NULL, payout = 0

HOUSE EDGE EFETIVO:
  - ~1% dos rounds crasham em 1.00x (perda garantida para todos)
  - Ao longo de N rounds, o servidor retira ~1% do volume total apostado
  - Se 10 jogadores apostam 1000 GC/round = 10.000 GC/round
  - Em 100 rounds = 1.000.000 GC apostados
  - House edge 1% = ~10.000 GC retidos pelo servidor
```

## 6.5 Balanceamento para Economia RP

| Aspecto | Impacto | Mitigacao |
|---------|---------|-----------|
| Jogadores com muitos GCoins podem dominar | Aposta max 10.000 GC limita exposicao |  |
| Crash instantaneo (1.00x) frustra | 1% dos rounds — aceitavel. Historico mostra que e raro | Nao aumentar house edge |
| Martingale (dobrar apos perda) drena rapido | Auto Bet com increase limitado a 100% | Mensagem de aviso no modal de Auto Bet |
| Multiplicadores altissimos (100x+) criam inflacao | Raro (~0.99% chance de 100x+ com RTP 99%) | Pode adicionar cap de 1000x se necessario |

## 7.2 Som Critico: Tom Ascendente Continuo

O som mais importante do Crash e o **tom ascendente continuo** (#3 na tabela). E ele que cria a tensao visceral — quanto mais alto o multiplicador, mais agudo o som, mais urgencia o jogador sente.

**Implementacao com Web Audio API (useSoundManager extendido):**
```javascript
// Criar oscillator continuo para o round
const ctx = new AudioContext();
const osc = ctx.createOscillator();
const gain = ctx.createGain();
osc.type = 'sine';
osc.frequency.value = 200; // frequencia inicial (grave)
gain.gain.value = 0.04; // volume baixo
osc.connect(gain);
gain.connect(ctx.destination);
osc.start();

// A cada frame, atualizar frequencia proporcionalmente ao multiplicador
function updatePitch(currentMultiplier) {
  const minFreq = 200;
  const maxFreq = 2000;
  const freq = minFreq + (Math.log(currentMultiplier) / Math.log(100)) * (maxFreq - minFreq);
  osc.frequency.setTargetAtTime(Math.min(freq, maxFreq), ctx.currentTime, 0.1);
}

// No crash, parar o oscillator
function stopRisingSound() {
  gain.gain.exponentialRampToValueAtTime(0.001, ctx.currentTime + 0.1);
  setTimeout(() => osc.stop(), 150);
}
```

Este som NAO existe em nenhum script FiveM de cassino. E um diferencial sensorial importante.

| Sem Pesquisa | Problema | Com Pesquisa |
|-------------|---------|-------------|

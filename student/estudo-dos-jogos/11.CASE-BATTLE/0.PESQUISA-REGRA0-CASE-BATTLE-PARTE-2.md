## 3.1 Animacao de Reveal Strip — Anatomia Tecnica

A animacao de reveal e o coracao visual do Case Battle. Todas as plataformas pesquisadas usam o mesmo padrao: strip horizontal com desaceleracao. A pesquisa no CS2 Case Simulator (case.oki.gg) confirmou que a "realistic case opening animation" usa strip horizontal desacelerando — o padrao visual que vamos implementar.

**Como o strip funciona tecnicamente:**

O strip e uma `<div>` com `overflow: hidden` contendo uma `<div>` interna (a "fita") com 30-50 item cards dispostos em `display: flex; flex-direction: row`. A fita comeca posicionada com `transform: translateX(-Npx)` onde N e grande (ex: -15.000px para 50 items de 300px cada). Uma CSS transition ou requestAnimationFrame move o translateX para a posicao final (onde o item vencedor esta sob o marcador central).

**Parametros de animacao baseados na pesquisa:**

| Parametro | Valor | Justificativa |
|-----------|-------|---------------|
| Itens no strip | 40-50 (item vencedor na posicao ~35-40) | Suficiente para scroll longo sem overflow de memoria. O vencedor nunca e o primeiro nem o ultimo — sempre no terco final |
| Largura de cada item card | clamp(60px, 8vw, 100px) | Responsivo. Em tela 1280px = ~100px. Em mobile 375px = ~60px |
| Duracao total do scroll | 3.5-5.0 segundos | CSGOEmpire varia por round. Primeiro round mais lento (buildup), ultimos mais rapidos |
| Easing curve | cubic-bezier(0.15, 0.85, 0.35, 1.0) | Inicio rapido, desaceleracao longa e suave. Simula "quase parou no legendary" — a ilusao de near-miss |
| Marcador central | Linha vertical 2px #EB4B4B (vermelho PvP) com glow | Indica exatamente onde o strip vai parar. Posicionado no center do container |
| Item background | rgba(rarity_color, 0.08) com borderLeft 3px solid rarity_color | Cor varia por raridade enquanto o strip scrolla — jogador VE as cores passando |

### 3.2.1 Algoritmo CSGOEmpire (PHP — codigo fonte oficial publicado)

```php
// CSGOEmpire Case Battle Fairness — CODIGO REAL PUBLICADO
// Fonte: help.csgoempire.com/en/articles/9030391

// 5 variaveis de entrada:
$serverSeed = 'ab4b631d02e718e538848a5e9a023216';  // random na criacao

// String de entrada para HMAC
$stringToHash = "$eosBlockSeed-$roundNumber-$playerPosition";
// Tiebreaker: "$eosBlockSeed-$roundNumber" (sem playerPosition)

// Calculo
$preResult = hash_hmac('sha256', $stringToHash, $serverSeed);

// Converte primeiros 12 hex chars (6 segmentos de 2) em decimal 0.0-1.0
$decimalValue = 0;
for ($i = 0; $i < 6; $i++) {
    $hexValue = substr($preResult, 2 * $i, 2);
    $decimalValue += hexdec($hexValue) / pow(256, $i + 1);
}

$result = (int)($decimalValue * $ticketQuantity) + 1;
// Resultado: ticket number 1-100000. Mapeado para item na case.
```

### 3.2.2 Adaptacao para JavaScript (Blackout Casino)

O Blackout Casino nao usa EOS blockchain (nao tem sentido no contexto FiveM). A adaptacao substitui o eosBlockSeed por um client_seed combinado dos jogadores:

```javascript
// caseBattleFairness.js — adaptacao Blackout Casino
const crypto = require('crypto');

const MAX_HEX_SEGMENTS = 6;
const HEX_SEGMENT_SIZE = 2;
const BASE = 256;

function calculateDecimalValue(preResult) {
  let decimalValue = 0;
  for (let i = 0; i < MAX_HEX_SEGMENTS; i++) {
    const hexValue = preResult.substring(HEX_SEGMENT_SIZE * i, HEX_SEGMENT_SIZE * (i + 1));
    decimalValue += parseInt(hexValue, 16) / Math.pow(BASE, i + 1);
  }
  return decimalValue;
}

function getBattleResult(serverSeed, combinedSeed, roundNumber, playerPosition, ticketQty) {
  // Mesmo formato do CSGOEmpire
  const stringToHash = `${combinedSeed}-${roundNumber}-${playerPosition}`;
  const preResult = crypto.createHmac('sha256', serverSeed)
    .update(stringToHash).digest('hex');
  const decimal = calculateDecimalValue(preResult);
  const ticket = Math.floor(decimal * ticketQty) + 1;
  return { preResult, ticket };
}

// Tiebreaker: sem playerPosition
function getTiebreakerResult(serverSeed, combinedSeed, totalRounds, ticketQty) {
  const roundNumber = totalRounds + 1;
  const stringToHash = `${combinedSeed}-${roundNumber}`;
  const preResult = crypto.createHmac('sha256', serverSeed)
    .update(stringToHash).digest('hex');
  const decimal = calculateDecimalValue(preResult);
  return Math.floor(decimal * ticketQty) + 1;
}
```

## 3.3 Sincronizacao Multiplayer no NUI FiveM

### 3.3.1 Arquitetura de Sync

O Case Battle e o primeiro jogo PvP REAL do Blackout Casino que requer que multiplos jogadores vejam a MESMA animacao ao MESMO tempo. A pesquisa investigou 3 abordagens:

**Abordagem 1: Server-Determined + Client Animation (ESCOLHIDA)**
O servidor calcula TODOS os resultados de TODOS os rounds antes da animacao comecar. Envia o pacote completo para TODOS os clientes via `emitNet('casino:cases:battle_started', -1, { battleId, results, serverSeedHash })`. Cada cliente recebe os mesmos dados e anima localmente. A animacao pode ter micro-diferencas de timing (1-2 frames) entre clientes, mas isso e IRRELEVANTE — o resultado e identico.

Esta e exatamente a abordagem que CSGOEmpire usa, confirmada pela pesquisa: resultados pre-determinados e verificaveis. A animacao do strip e puro visual — o item onde o strip para ja foi decidido pelo Provably Fair ANTES da animacao comecar.

**Por que as outras abordagens foram descartadas:**

Abordagem 2 (Frame-sync via events): Requer enviar evento a cada frame ou a cada round start. Latencia de rede no FiveM (20-100ms) causaria desync visivel. Complexidade alta sem beneficio real.

Abordagem 3 (WebSocket real-time): FiveM NUI nao tem WebSocket nativo entre NUI e server. Seria necessario implementar polling ou SSE, ambos overkill para um jogo onde todos os resultados ja sao conhecidos antes da animacao.

## 4.2 Viabilidade Tecnica no NUI (CEF/Chromium)

### CSS Animations no NUI
CSS transitions e @keyframes: 100% funcional no CEF. `transform: translateX/scale/rotate`: GPU-accelerated. `will-change + transform: translateZ(0)`: suportado. Limitacao: `filter:blur()` pode degradar com GPU compartilhada — evitar em elementos animados frequentemente (usar box-shadow em vez de blur para glow).

### Framer Motion 11 no NUI
O projeto JA usa FM11. No CEF (Chromium embarcado do FiveM), FM11 funciona normalmente — e JavaScript puro executando no V8. `spring/tween/inertia`: todos funcionais. `AnimatePresence`: funcional para entrada/saida de jogadores. `layout` animations: funcional. Limitacao: muitas springs simultaneas (>20) podem impactar frame budget. Para Case Battle: maximo 4 strips × 1 spring por strip = 4 springs — dentro do budget.

### Web Crypto API no NUI
`crypto.subtle.digest('SHA-256')`: suportado no CEF moderno. `crypto.subtle.importKey + sign (HMAC)`: suportado. `TextEncoder`: suportado. Relevancia: jogador pode verificar Provably Fair diretamente no NUI sem precisar de site externo.

### Web Audio API no NUI
Suportado com restricao: `AudioContext` requer interacao do usuario para unlock. No Case Battle: o primeiro clique do jogador (ao entrar no lobby ou criar batalha) desbloqueia o audio context. Todos os sons subsequentes tocam normalmente.

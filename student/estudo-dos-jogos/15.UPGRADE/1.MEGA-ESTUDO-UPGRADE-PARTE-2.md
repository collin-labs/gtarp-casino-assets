## Problema 1: SVG Arc Math - Calcular Zonas Proporcionais do Wheel

**Sintoma:** O wheel precisa exibir 2 zonas (verde/vermelha) proporcionais a chance%. Uma chance de 11.5% deve resultar em exatamente 11.5% do circulo preenchido em verde. Erro nessa math = jogo visualmente desonesto.

**Causa tecnica:** SVG `stroke-dasharray` trabalha com a circunferencia total do circulo (2*PI*r), nao com porcentagem direta. Precisa converter chance% -> unidades de circunferencia. Alem disso, o arco comeca a 3h (lado direito) por padrao SVG, mas o wheel deve comecar as 12h (topo).

**Solucao com codigo real:**

A pesquisa complementar (Medium/Sergio Pedercini, MDN stroke-dasharray, CodePen/arrayzero) revelou um truque: usar `pathLength="100"` no SVG path, que normaliza a circunferencia para 100 unidades. Assim, stroke-dasharray aceita porcentagem diretamente.

```tsx
// UpgradeWheel.tsx - Componente do wheel com zonas proporcionais

interface WheelProps {
  chance: number;  // 0-100
  spinning: boolean;
  targetAngle: number; // angulo final do ticker (graus)
}

function UpgradeWheel({ chance, spinning, targetAngle }: WheelProps) {
  const SIZE = 200;
  const CENTER = SIZE / 2;
  const RADIUS = 80;
  const STROKE_WIDTH = 24;

  // pathLength="100" normaliza circunferencia para 100 unidades
  // stroke-dasharray="chance, 100" = zona verde de chance%
  // A zona vermelha e o "gap" (100 - chance) que fica visivel por baixo

  return (
    <svg viewBox={`0 0 ${SIZE} ${SIZE}`} style={{ width: '100%', maxWidth: 400 }}>
      {/* Borda externa dourada decorativa */}
      <circle
        cx={CENTER} cy={CENTER} r={RADIUS + STROKE_WIDTH/2 + 4}
        fill="none"
        stroke="rgba(212,168,67,0.3)"
        strokeWidth={2}
      />

      {/* Camada vermelha (fundo - derrota) */}
      <circle
        cx={CENTER} cy={CENTER} r={RADIUS}
        fill="none"
        stroke="#FF1744"
        strokeWidth={STROKE_WIDTH}
        pathLength={100}
        strokeDasharray="100 100"
        transform={`rotate(-90 ${CENTER} ${CENTER})`}
        style={{ filter: 'drop-shadow(0 0 4px rgba(255,23,68,0.3))' }}
      />

      {/* Camada verde (vitoria - sobreposta) */}
      <circle
        cx={CENTER} cy={CENTER} r={RADIUS}
        fill="none"
        stroke="#00E676"
        strokeWidth={STROKE_WIDTH}
        pathLength={100}
        strokeDasharray={`${chance} ${100 - chance}`}
        transform={`rotate(-90 ${CENTER} ${CENTER})`}
        style={{ filter: 'drop-shadow(0 0 6px rgba(0,230,118,0.4))' }}
      />

      {/* Ticker/Ponteiro - Framer Motion rotate */}
      <motion.line
        x1={CENTER} y1={CENTER}
        x2={CENTER} y2={CENTER - RADIUS - STROKE_WIDTH/2 - 8}
        stroke="#FFD700"
        strokeWidth={3}
        strokeLinecap="round"
        animate={spinning ? { rotate: targetAngle } : {}}
        transition={{
          duration: 3.5,
          ease: [0.12, 0.8, 0.3, 1],
        }}
        style={{
          transformOrigin: `${CENTER}px ${CENTER}px`,
          filter: 'drop-shadow(0 0 8px rgba(255,215,0,0.6))',
        }}
      />

      {/* Ponta do ticker (triangulo dourado) */}
      <motion.polygon
        points={`${CENTER-6},${CENTER-RADIUS-STROKE_WIDTH/2-4} ${CENTER+6},${CENTER-RADIUS-STROKE_WIDTH/2-4} ${CENTER},${CENTER-RADIUS-STROKE_WIDTH/2-12}`}
        fill="#FFD700"
        animate={spinning ? { rotate: targetAngle } : {}}
        transition={{
          duration: 3.5,
          ease: [0.12, 0.8, 0.3, 1],
        }}
        style={{ transformOrigin: `${CENTER}px ${CENTER}px` }}
      />
    </svg>
  );
}
```

**Por que `pathLength="100"` resolve:**
- Sem pathLength: circunferencia = 2*PI*80 = 502.65. Para 11.5% verde: `strokeDasharray="57.80 444.85"` (calculo manual propenso a erro)
- Com pathLength="100": para 11.5% verde: `strokeDasharray="11.5 88.5"` (porcentagem direta)
- MDN confirma: "pathLength redefines the total path length to the specified value"
- O `rotate(-90 center center)` move o inicio do arco de 3h para 12h (topo)

**Impacto:** ~80 linhas no componente UpgradeWheel.tsx. Math simplificada, zero chance de erro de calculo.

## Problema 2: Easing de Desaceleracao - Fazer o Ticker Parecer Fisicamente Natural

**Sintoma:** Se o ticker desacelerar linearmente, parece "mecanico" e artificial. Se usar easing errado, o ticker "pula" no final ou para abruptamente. O jogador precisa SENTIR que o ticker esta lutando contra a inercia.

**Causa tecnica:** Framer Motion aceita cubic-bezier customizado `ease: [a, b, c, d]`. O desafio e encontrar a curva que: (1) comeca rapido, (2) desacelera gradualmente no meio, (3) quase para perto do fim, (4) tem um micro-drift final antes de parar completamente.

**Solucao: Curva cubic-bezier testada + voltas extras para drama**

```tsx
// Constantes de animacao do wheel

// Curva principal: rapido no inicio, desacelera forte no fim
// P1(0.12, 0.8) = rapido logo apos inicio
// P2(0.3, 1.0) = quase estavel no final (microsdrift)
const SPIN_EASING = [0.12, 0.8, 0.3, 1] as const;

// Voltas visuais: 4 voltas completas + angulo final
// Mais voltas = mais drama. 4 e o sweet spot (nao cansa, nao e curto)
const EXTRA_ROTATIONS = 4;

// Duracao total da animacao
// 3.5s e o padrao de CSGORoll/Clash.gg (pesquisa confirmou)
const SPIN_DURATION = 3.5;

function calcularAnguloFinal(resultado: number): number {
  // resultado: 0.00 a 99.99 (do provably fair)
  // Converter para graus: 0-100 -> 0-360
  const anguloResultado = resultado * 3.6;
  // Adicionar voltas extras para drama visual
  const voltas = EXTRA_ROTATIONS * 360;
  return voltas + anguloResultado;
}

// Uso no Framer Motion:
// animate={{ rotate: calcularAnguloFinal(resultado) }}
// transition={{ duration: SPIN_DURATION, ease: SPIN_EASING }}
```

**Alternativa testada e descartada:** `type: "spring"` com `bounce: 0.05` gera micro-oscilacao no final (ticker "treme" antes de parar). Bom para slots, ruim para upgrader onde o jogador olha FIXO para a posicao final. Cubic-bezier e mais previsivel.

**Referencia da pesquisa:** Clash.gg usa "smooth animation" (review ProSettings). CSGORoll usa "responsive visual feedback" (review ThirdEyeGen). Ambos sugerem animacao fluida sem bounce.

**Impacto:** ~15 linhas de constantes + logica no UpgradeWheel. Diferenca entre "jogo de celular generico" e "experiencia premium Las Vegas".

## Problema 3: Sincronizacao de Inventario entre Cases #14 e Upgrade #15

**Sintoma:** O jogador abre uma caixa no Cases e ganha uma "Sniper Pesada". Depois vai ao Upgrade e quer usar essa Sniper como item proprio. Se os dois jogos nao compartilham inventario, o item nao aparece.

**Causa tecnica:** O Cases #14 usa tabelas `casino_case_items` (catalogo) e `casino_inventory` (itens do jogador). O Upgrade precisa LER o inventario do jogador (para mostrar "meus itens") e ESCREVER nele (adicionar item ganho, remover item perdido). Se cada jogo tiver tabela de inventario separada, gera duplicacao e inconsistencia.

**Solucao: Reutilizar casino_inventory como fonte unica de verdade**

```javascript
// upgrade.js handler - Reutiliza inventario do Cases

// Buscar itens do jogador (mesmo inventario do Cases #14)
async function getPlayerInventory(playerId) {
  const inventario = await query(`
    SELECT ci.id as inv_id, ci.item_id, ci.obtained_from, ci.obtained_at,
           items.name, items.rarity, items.value, items.image_url,
           items.item_type, items.sell_back_value
    FROM casino_inventory ci
    JOIN casino_case_items items ON ci.item_id = items.id
    WHERE ci.player_id = ? AND ci.status = 'active'
    ORDER BY items.value DESC
  `, [playerId]);
  return inventario;
}

// Buscar catalogo de alvos (todos itens com valor > item do jogador)
async function getUpgradeTargets(myItemValue) {
  const alvos = await query(`
    SELECT id, name, rarity, value, image_url, item_type,
           rarity_color, sell_back_value
    FROM casino_case_items
    WHERE value > ? AND is_active = 1
    ORDER BY value ASC
  `, [myItemValue]);
  return alvos;
}

// Executar upgrade atomicamente (transacao MySQL)
async function executeUpgrade(playerId, invId, targetItemId, won) {
  const conn = await getConnection();
  try {
    await conn.beginTransaction();

    // Remover item do jogador (sempre - ganhou ou perdeu)
    await conn.query(
      `UPDATE casino_inventory SET status = 'upgraded', updated_at = NOW()
       WHERE id = ? AND player_id = ? AND status = 'active'`,
      [invId, playerId]
    );

    if (won) {
      // Adicionar item alvo ao inventario
      await conn.query(
        `INSERT INTO casino_inventory (player_id, item_id, obtained_from, status)
         VALUES (?, ?, 'upgrade', 'active')`,
        [playerId, targetItemId]
      );
    }
    // Se perdeu, item simplesmente fica com status 'upgraded' (destruido)

    await conn.commit();
  } catch (erro) {
    await conn.rollback();
    throw erro;
  } finally {
    conn.release();
  }
}
```

**Ponto critico:** A tabela `casino_inventory` precisa do campo `obtained_from` com valor 'upgrade' para rastrear origem. Se o Cases #14 nao tem esse campo, adicionar via ALTER TABLE:
```sql
ALTER TABLE casino_inventory
  MODIFY COLUMN obtained_from ENUM('case_opening','upgrade','daily_free','giveaway','admin','purchase')
  DEFAULT 'case_opening';
```

**Impacto:** Zero tabela nova de inventario. Reutilizacao total. O Upgrade "enxerga" itens ganhos no Cases, Daily Free, Giveaways. Consistencia garantida via transacao atomica.

## Problema 4: House Edge Transparencia - Mostrar ou Esconder?

**Sintoma:** Se o house edge (8%) e invisivel, jogadores desconfiam ("por que minha chance nao e 50% para x2?"). Se e explicito demais, jogadores evitam jogar ("a casa sempre ganha 8%"). Precisa de equilibrio.

**Causa tecnica:** No Upgrade, a chance e calculada como `(valor_meu / valor_alvo) * (1 - edge)`. Com edge 8%, um upgrade x2.0 tem 46% de chance (nao 50%). A diferenca de 4% e a margem da casa. Plataformas como CSGORoll nao mostram o edge diretamente -- apenas a chance final.

**Solucao: Chance exibida, edge implicito, verificavel via Provably Fair**

```
ABORDAGEM DO BLACKOUT CASINO:

1. EXIBIR: chance% final (ex: "46.0%") - sempre visivel, grande, verde neon
2. EXIBIR: multiplicador (ex: "x2.0") - sempre visivel
3. NAO EXIBIR: house edge % diretamente na tela de jogo
4. EXIBIR NA MODAL PROVABLY FAIR: "Margem da casa: 8%" como informacao tecnica
5. EXIBIR NA MODAL: formula completa "Chance = (Valor Item / Valor Alvo) x 0.92"

LOGICA: jogadores casuais veem "46% de chance para x2" e jogam.
Jogadores informados abrem a modal Provably Fair e veem a formula.
Transparencia total para quem quer, sem assustar quem nao quer.

REFERENCIA: CSGORoll e Clash.gg fazem exatamente isso --
chance visivel, edge nao mencionado na UI principal,
verificavel nas paginas de fairness.
```

**Impacto:** Zero linhas de codigo adicionais. Decisao de UX que afeta textos e layout, nao logica.

## Problema 5: Rate Limiting e Anti-Abuse

**Sintoma:** Jogador cria script automatizado que faz 1000 upgrades por minuto usando a API. Ou jogador tenta multiplos upgrades simultaneos para explorar race condition.

**Causa tecnica:** O handler aceita requests via fetchNui. Sem rate limit, nada impede burst de requests. Race condition: dois requests simultaneos ambos leem "jogador possui item X", ambos tentam remover, segundo falha com erro de banco.

**Solucao: 3 camadas de protecao com codigo real**

```javascript
// Camada 1: Cooldown por jogador (memoria server)
const cooldownMap = new Map(); // playerId -> timestamp

function checkCooldown(playerId, cooldownMs) {
  const lastUpgrade = cooldownMap.get(playerId) || 0;
  const agora = Date.now();
  if (agora - lastUpgrade < cooldownMs) {
    const restante = cooldownMs - (agora - lastUpgrade);
    return { allowed: false, waitMs: restante };
  }
  cooldownMap.set(playerId, agora);
  return { allowed: true };
}

// Camada 2: Lock por jogador (previne race condition)
const lockMap = new Map(); // playerId -> boolean

async function withPlayerLock(playerId, fn) {
  if (lockMap.get(playerId)) {
    return { error: 'upgrade_in_progress' };
  }
  lockMap.set(playerId, true);
  try {
    return await fn();
  } finally {
    lockMap.delete(playerId);
  }
}

// Camada 3: Daily limit via MySQL (persistente)
async function checkDailyLimit(playerId, maxPerDay) {
  const hoje = new Date().toISOString().split('T')[0];
  const [row] = await query(
    `SELECT upgrades_today FROM casino_upgrade_daily
     WHERE player_id = ? AND play_date = ?`,
    [playerId, hoje]
  );
  if (row && row.upgrades_today >= maxPerDay) {
    return { allowed: false, remaining: 0 };
  }
  return { allowed: true, remaining: maxPerDay - (row?.upgrades_today || 0) };
}

// Uso no handler principal:
async function handleUpgrade(playerId, myItemId, targetItemId) {
  // Camada 1
  const cooldown = checkCooldown(playerId, config.cooldownMs);
  if (!cooldown.allowed) return { error: 'cooldown', waitMs: cooldown.waitMs };

  // Camada 3
  const daily = await checkDailyLimit(playerId, config.maxUpgradesPerDay);
  if (!daily.allowed) return { error: 'daily_limit', remaining: 0 };

  // Camada 2
  return await withPlayerLock(playerId, async () => {
    // ... logica do upgrade (validacao, calculo, transacao) ...
  });
}
```

**Impacto:** ~60 linhas no handler. 3 camadas: cooldown em memoria (rapido), lock anti-race (atomico), daily limit em MySQL (persistente). Padrao identico ao usado no slots.js (1202 linhas de handler existente).

### Queries Admin Prontas

```sql
-- Q1: Total de upgrades e taxa de vitoria
SELECT
  COUNT(*) as total_upgrades,
  SUM(won) as total_wins,
  ROUND(SUM(won) / COUNT(*) * 100, 1) as win_rate_percent,
  upgrade_type
FROM casino_upgrade_history
GROUP BY upgrade_type;

-- Q2: Revenue da casa (itens destruidos que nao foram repostos)
SELECT
  DATE(created_at) as dia,
  COUNT(*) as upgrades,
  SUM(CASE WHEN won = 0 THEN item_from_value ELSE 0 END) as valor_destruido,
  SUM(CASE WHEN upgrade_type = 'gcoin' AND won = 0 THEN gcoin_bet ELSE 0 END) as gcoin_perdido_jogadores,
  SUM(CASE WHEN upgrade_type = 'gcoin' AND won = 1 THEN gcoin_potential_win - gcoin_bet ELSE 0 END) as gcoin_pago_aos_vencedores
FROM casino_upgrade_history
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY DATE(created_at)
ORDER BY dia DESC;

-- Q3: Top jogadores por volume
SELECT
  player_id, player_name,
  COUNT(*) as total,
  SUM(won) as vitorias,
  ROUND(SUM(won)/COUNT(*)*100, 1) as win_rate
FROM casino_upgrade_history
GROUP BY player_id, player_name
ORDER BY total DESC
LIMIT 20;

-- Q4: Itens mais upgradeados (alvos populares)
SELECT
  item_to_name, item_to_rarity, item_to_value,
  COUNT(*) as tentativas,
  SUM(won) as conquistados,
  ROUND(SUM(won)/COUNT(*)*100, 1) as taxa_sucesso
FROM casino_upgrade_history
WHERE upgrade_type = 'item'
GROUP BY item_to_id, item_to_name, item_to_rarity, item_to_value
ORDER BY tentativas DESC
LIMIT 20;

  player_id, player_name,
  COUNT(*) as total,
  SUM(won) as wins,
  ROUND(SUM(won)/COUNT(*)*100, 1) as win_rate
FROM casino_upgrade_history
GROUP BY player_id, player_name
HAVING total >= 20 AND win_rate > 70
ORDER BY win_rate DESC;
```

## Problema 1: Equip - Integracao com vRP para Dar Item no GTA

**Sintoma:** Jogador clica "EQUIPAR" no inventario do casino. O item precisa aparecer no inventario GTA do personagem (arma no slot de armas, veiculo na garagem). Essa ponte casino->GTA nao existe.

**Causa:** O handler de sell ja funciona (credita GCoins via vRP.addMoney). Mas nao existe handler para "dar arma" ou "dar veiculo" via API do vRP.

**Solucao: Handler novo com TriggerEvent para vRP.**

```javascript
// NOVO - server/handlers/inventory_equip.js (~70 linhas)
RegisterNuiCallback('casino:inventory:equip', async (data, cb) => {
    const userId = vRP.getUserId(source);
    const { invId } = data;
    
    // Validar propriedade
    const item = await oxmysql.query(`
        SELECT inv.id, inv.status, i.name, i.item_type, i.game_item_id
        FROM casino_cases_inventory inv
        JOIN casino_case_items i ON inv.item_id = i.id
        WHERE inv.id = ? AND inv.user_id = ? AND inv.status = 'owned'
    `, [invId, userId]);
    
    if (!item) return cb({ ok: false, error: 'Item nao encontrado' });
    
    // Dar item conforme tipo
    switch (item.item_type) {
        case 'weapon':
            // vRP padrao: dar arma pelo hash/name
            vRP.giveWeapon(userId, item.game_item_id, 100);
            break;
        case 'vehicle':
            // vRP padrao: dar veiculo na garagem
            vRP.addVehicle(userId, item.game_item_id);
            break;
        case 'clothing':
            // vRP padrao: dar roupa
            vRP.addClothing(userId, item.game_item_id);
            break;
        default:
            return cb({ ok: false, error: 'Tipo nao equipavel' });
    }
    
    // Atualizar status
    await oxmysql.update(
        "UPDATE casino_cases_inventory SET status = 'equipped' WHERE id = ?", [invId]
    );
    
    // Audit
    await oxmysql.execute(
        `INSERT INTO casino_cases_audit (user_id, action, details) VALUES (?, 'equip', ?)`,
        [userId, JSON.stringify({ inv_id: invId, item: item.name, type: item.item_type })]
    );
    
    cb({ ok: true });
});

RegisterNuiCallback('casino:inventory:unequip', async (data, cb) => {
    const userId = vRP.getUserId(source);
    const { invId } = data;
    
    const item = await oxmysql.query(`
        SELECT inv.id, inv.status, i.name, i.item_type, i.game_item_id
        FROM casino_cases_inventory inv
        JOIN casino_case_items i ON inv.item_id = i.id
        WHERE inv.id = ? AND inv.user_id = ? AND inv.status = 'equipped'
    `, [invId, userId]);
    
    if (!item) return cb({ ok: false, error: 'Item nao equipado' });
    
    // Remover item do GTA
    switch (item.item_type) {
        case 'weapon':
            vRP.removeWeapon(userId, item.game_item_id);
            break;
        case 'vehicle':
            vRP.removeVehicle(userId, item.game_item_id);
            break;
    }
    
    await oxmysql.update(
        "UPDATE casino_cases_inventory SET status = 'owned' WHERE id = ?", [invId]
    );
    
    await oxmysql.execute(
        `INSERT INTO casino_cases_audit (user_id, action, details) VALUES (?, 'unequip', ?)`,
        [userId, JSON.stringify({ inv_id: invId, item: item.name })]
    );
    
    cb({ ok: true });
});
```

**Nota:** As funcoes `vRP.giveWeapon`, `vRP.addVehicle`, `vRP.removeWeapon`, `vRP.removeVehicle` dependem da base vRP do servidor. O campo `game_item_id` na tabela `casino_case_items` mapeia o item do casino para o identificador do vRP (ex: "WEAPON_RAILGUN", "adder"). Se a base usar framework diferente (QBCore, ESX), os nomes de funcao mudam mas a logica e identica.

**Impacto:** ~70 linhas JS. Zero alteracao em tabelas existentes (campo `game_item_id` JA EXISTE em casino_case_items).

## Problema 4: Scroll em Inventarios Grandes (100+ itens)

**Sintoma:** Jogador hardcore com 200+ itens. DOM pesado se renderizar todos de uma vez.

**Causa:** 200 cards x ~8 DOM elements cada = 1.600 elements. Em NUI FiveM pode causar jank.

**Solucao multi-camada:**

1. **Para 99% dos jogadores (<100 itens):** CSS Grid normal. 100 cards x 8 elements = 800 DOM. Zero problema.

2. **Para power users (100-500 itens):** Paginacao simples. Mostrar 50 por "pagina" com botao "Carregar mais" (infinite scroll). Framer Motion anima os novos cards entrando.

3. **Para cenario extremo (500+):** react-window (6KB) para virtual scroll. So renderiza cards visiveis no viewport. Mas isso e v2 - improvavel que jogador RP tenha 500 itens de casino.

**Decisao v1:** Paginacao com "Carregar mais" em blocos de 50. Simples, zero lib nova, resolve 99.9% dos casos.

```
const ITENS_POR_PAGINA = 50;
const [paginaAtual, setPaginaAtual] = useState(1);
const itensVisiveis = itensFiltrados.slice(0, paginaAtual * ITENS_POR_PAGINA);
const temMais = itensVisiveis.length < itensFiltrados.length;
```

**Impacto:** ~15 linhas. Botao "Carregar mais" com estilo outline dourado.

## Problema 5: Sincronizacao Apos Venda (Inventario Desatualizado)

**Sintoma:** Jogador vende item. Backend processa. Mas se outro jogador abriu case enquanto isso, o inventario local esta desatualizado.

**Causa:** Inventario e fetched on mount. Nao tem polling nem push.

**Solucao:** Apos cada acao (sell, equip, unequip), o callback do server retorna dados atualizados. Frontend remove/atualiza o item localmente sem refetch completo.

```
// Apos sell bem-sucedido:
setItens(prev => prev.filter(i => i.inv_id !== itemVendido.inv_id));

// Apos equip bem-sucedido:
setItens(prev => prev.map(i => 
    i.inv_id === itemEquipado.inv_id ? { ...i, status: 'equipped' } : i
));
```

**Impacto:** Zero fetch extra. Atualizacao local instantanea. Se jogador quiser forcar refresh: botao "Atualizar" ou fechar/reabrir inventario.

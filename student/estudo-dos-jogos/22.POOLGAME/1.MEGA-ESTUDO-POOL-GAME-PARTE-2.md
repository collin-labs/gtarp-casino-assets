## Problema 1: Physics Server-Side — Peso de Rodar Matter.js no Server

**Sintoma:** O server FiveM precisa rodar Matter.js para cada partida ativa (anti-cheat). Se 20 partidas simultâneas, são 20 instâncias de physics engine rodando no server.

**Causa técnica:** Matter.js roda em Node.js no server. Cada `Engine.update()` com 26 bodies leva ~0.3ms. Porém, após um shot, a simulation precisa rodar até `allBallsStopped()` — potencialmente 100-300 frames (1.6-5 segundos a 60fps). Isso significa ~30-90ms de CPU por shot no server.

**Dados concretos:**
- 1 shot simulation: ~30-90ms (100-300 steps × 0.3ms)
- 20 partidas simultâneas com shots concorrentes: ~600-1800ms de CPU pico
- Server FiveM típico: single-threaded JavaScript → shots são sequenciais
- Latência visível: jogador espera ~100-200ms para resultado (aceitável para turn-based)

**Solução multi-camada:**
1. **Simulation budget:** Limitar steps por shot a 300 (`maxSteps = 300`). Se bolas ainda movem após 300 steps, force stop (set velocity 0). Na prática, 99% dos shots param antes de 200 steps.
2. **Async simulation:** Rodar simulation em `setImmediate()` chunks de 50 steps, cedendo event loop entre chunks. Evita bloquear o server por 90ms contínuos.
3. **Pool de engines:** Pré-criar 10 engine instances reutilizáveis. Cada partida pega uma do pool, usa, e devolve. Evita GC de criar/destruir engines.
4. **Fallback: client-side physics + server validation.** Se performance server for insuficiente, client roda physics localmente (responsividade imediata), envia resultado ao server, server re-simula com mesmos inputs e compara. Discrepância > threshold = flag.

**Código de async simulation:**
```javascript
async function simulateShot(engine, cueBall, shot, maxSteps = 300) {
  // Aplicar força
  const forceX = Math.cos(shot.angle) * shot.power * MAX_FORCE;
  const forceY = Math.sin(shot.angle) * shot.power * MAX_FORCE;
  Body.applyForce(cueBall, cueBall.position, { x: forceX, y: forceY });
  
  // Aplicar spin modifier
  cueBall.plugin.spin = { top: shot.spinTop, side: shot.spinSide };
  
  let steps = 0;
  const CHUNK = 50;
  
  while (steps < maxSteps) {
    // Rodar chunk de 50 steps
    for (let i = 0; i < CHUNK && steps < maxSteps; i++, steps++) {
      Engine.update(engine, 1000 / 60);
    }
    
    // Verificar se todas pararam
    const allStopped = Composite.allBodies(engine.world)
      .filter(b => !b.isStatic)
      .every(b => Body.getSpeed(b) < 0.05);
    
    if (allStopped) break;
    
    // Ceder event loop
    await new Promise(resolve => setImmediate(resolve));
  }
  
  // Force stop se ainda movem
  Composite.allBodies(engine.world)
    .filter(b => !b.isStatic)
    .forEach(b => Body.setVelocity(b, { x: 0, y: 0 }));
  
  return extractState(engine); // retorna posições finais
}
```

**Impacto:** ~50 linhas extras no handler. Latência: ~50-150ms por shot (aceitável). Server suporta 20+ partidas simultâneas.

## Problema 2: Spin Model — Matter.js é 2D, Spin é 3D

**Sintoma:** Top spin (follow), back spin (draw) e side spin (english) são fenômenos 3D — a bola rola no plano, com angular velocity em 3 eixos. Matter.js só tem angular velocity no eixo Z (rotação 2D).

**Causa técnica:** Em pool real, o cue ball tem spin em X (top/back), Y (roll), e Z (side english). Matter.js modela apenas Z. Top spin faz a cue ball continuar após impacto; back spin faz voltar. Side english causa deflexão pós-cushion. Nenhum desses é nativo no Matter.js.

**Solução: Custom Spin Layer (post-collision velocity adjustment)**

A pesquisa (Evan Kiefl, World of Pool) revelou que para um jogo 2D, não precisamos simular spin 3D real. Basta **ajustar a velocidade da cue ball após colisões** baseado no spin state:

```javascript
// PoolSpinEngine.ts — Custom spin layer

const FOLLOW_FACTOR = 0.6;  // top spin boost %
const DRAW_FACTOR = -0.8;    // back spin reverse %
const ENGLISH_FACTOR = 0.4;  // side spin perpendicular %
const SPIN_DECAY = 0.92;     // spin decai a cada colisão

class PoolSpinEngine {
  private spinState = { top: 0, side: 0 }; // -1 a +1 cada
  
  applyCueShot(cueBall, angle, power, spinTop, spinSide) {
    // Força principal
    const fx = Math.cos(angle) * power * MAX_FORCE;
    const fy = Math.sin(angle) * power * MAX_FORCE;
    Body.applyForce(cueBall, cueBall.position, { x: fx, y: fy });
    
    // Armazenar spin state
    this.spinState = { top: spinTop * power, side: spinSide * power };
  }
  
  // Chamado APÓS colisão cue-ball com object-ball
  onCueBallHitObjectBall(cueBall, objectBall) {
    const vel = cueBall.velocity;
    const speed = Body.getSpeed(cueBall);
    
    if (speed < 0.1) return; // muito lenta, spin irrelevante
    
    // TOP SPIN (follow): cue ball continua na direção original
    if (this.spinState.top > 0) {
      const boost = this.spinState.top * FOLLOW_FACTOR;
      Body.setVelocity(cueBall, {
        x: vel.x + vel.x * boost,
        y: vel.y + vel.y * boost
      });
    }
    
    // BACK SPIN (draw): cue ball reverte parcialmente
    if (this.spinState.top < 0) {
      const reverse = this.spinState.top * DRAW_FACTOR;
      Body.setVelocity(cueBall, {
        x: vel.x * reverse,
        y: vel.y * reverse
      });
    }
    
    // Decair spin após colisão
    this.spinState.top *= SPIN_DECAY;
    this.spinState.side *= SPIN_DECAY;
  }
  
  // Chamado APÓS colisão cue-ball com cushion (wall)
  onCueBallHitCushion(cueBall, wallLabel) {
    if (Math.abs(this.spinState.side) < 0.05) return;
    
    const vel = cueBall.velocity;
    const isVerticalWall = wallLabel === 'wall_left' || wallLabel === 'wall_right';
    
    // SIDE ENGLISH: adiciona componente perpendicular pós-cushion
    if (isVerticalWall) {
      Body.setVelocity(cueBall, {
        x: vel.x,
        y: vel.y + this.spinState.side * ENGLISH_FACTOR * Body.getSpeed(cueBall)
      });
    } else {
      Body.setVelocity(cueBall, {
        x: vel.x + this.spinState.side * ENGLISH_FACTOR * Body.getSpeed(cueBall),
        y: vel.y
      });
    }
    
    this.spinState.side *= SPIN_DECAY;
  }
  
  // STOP SHOT: se spin ≈ 0 e head-on collision → cue ball para
  onHeadOnCollision(cueBall) {
    if (Math.abs(this.spinState.top) < 0.1 && Math.abs(this.spinState.side) < 0.1) {
      Body.setVelocity(cueBall, { x: 0, y: 0 });
    }
  }
  
  reset() { this.spinState = { top: 0, side: 0 }; }
}
```

**Impacto:** ~80 linhas de código. Captura 90% do "feel" de spin real sem physics 3D. Diferencial: iBilliards (único concorrente FiveM com spin) provavelmente usa abordagem similar.

## Problema 3: Layout Mobile da Mesa (REGRA X4 — 70% vendas mobile)

**Sintoma:** Mesa de pool tem aspect ratio 2:1 (horizontal). Em mobile portrait (~375×667), a mesa ficaria 375×187px — muito pequena para mirar com precisão.

**Causa técnica:** Canvas de pool precisa de área suficiente para aiming preciso. Bolas de ~8px em tela de 375px = difícil tocar/arrastar com dedo.

**Solução multi-camada:**
1. **Mobile: forçar landscape.** `screen.orientation.lock('landscape')` quando PoolTable monta. Restaura ao sair. Tela 667×375 → mesa 650×325 → bolas ~12px = jogável.
2. **Se orientation lock falha:** mesa ocupa 100vw em portrait com scroll horizontal bloqueado. Controles (power, spin) ficam em bottom sheet overlay (48px height). Mesa 375×187 com zoom pinch habilitado.
3. **Tablet (640-1024px):** Mesa centralizada com controles laterais.
4. **Desktop (>1024px):** Mesa ampla + power bar lateral + spin selector canto + HUD no topo.

## Problema 4: Multiplayer Sync — Garantir Mesmo Resultado em Ambos Clients

**Sintoma:** Se client A e client B animam o shot localmente, diferenças floating-point na physics engine podem causar posições finais diferentes.

**Causa técnica:** Matter.js usa floating-point arithmetic. Dois clients executando a mesma simulation com os mesmos inputs podem divergir após ~50+ steps (butterfly effect).

**Solução: Server é a Fonte da Verdade.**
- Server roda simulation ÚNICA (canônica)
- Server retorna `ballPositions: [{ id, x, y }]` para AMBOS clients
- Clients NÃO rodam physics — apenas animam bolas de posição atual para posição final
- Animação: interpolação suave `lerp(currentPos, serverPos, t)` durante ~1.5s
- Resultado: ambos clients mostram EXATAMENTE o mesmo estado

```javascript
// Client-side: animar bolas para posições do server
function animateToServerPositions(balls, serverPositions, duration = 1500) {
  const startTime = Date.now();
  const startPositions = balls.map(b => ({ x: b.position.x, y: b.position.y }));
  
  function animate() {
    const t = Math.min(1, (Date.now() - startTime) / duration);
    const eased = easeOutCubic(t); // desaceleração suave
    
    balls.forEach((ball, i) => {
      const target = serverPositions.find(sp => sp.id === ball.label);
      if (!target) return;
      
      Body.setPosition(ball, {
        x: startPositions[i].x + (target.x - startPositions[i].x) * eased,
        y: startPositions[i].y + (target.y - startPositions[i].y) * eased
      });
    });
    
    if (t < 1) requestAnimationFrame(animate);
  }
  animate();
}

function easeOutCubic(t) { return 1 - Math.pow(1 - t, 3); }
```

**Impacto:** ~30 linhas. Zero desync. Trade-off: shots não são "instantâneos" — há ~200ms de latência (server sim) + ~1.5s de animação. Para turn-based, perfeitamente aceitável.

## Problema 5: 8-Ball Rules — Edge Cases Complexos

**Sintoma:** 8-ball tem dezenas de edge cases que precisam ser tratados: encaçapar 8-ball no break (win ou re-rack?), encaçapar cue ball + 8-ball no mesmo shot, combo onde bola adversária encaçapa a sua, etc.

**Causa técnica:** As regras oficiais WPA (World Pool-Billiard Association) são extensas. Para RP, simplificar mas manter integridade.

**Solução: Regras Simplificadas para RP (10 regras core)**

```javascript
// pool-rules-8ball.ts — 10 regras core

const RULES_8BALL = {
  // R1: Break — 8-ball pocketed no break = re-rack (não win/loss)
  breakEightBall: 'rerack',
  
  // R2: Break — cue ball pocketed no break = foul, oponente ball-in-hand atrás da headstring
  breakScratch: 'foul_behind_headstring',
  
  // R3: Open table — após break, primeiro pote legal define grupo
  openTableAssign: true,
  
  // R4: Encaçapar bola do grupo = mesmo turno continua
  legalPotContinue: true,
  
  // R5: Não encaçapar nada = turno muda
  noPotTurnChange: true,
  
  // R6: Encaçapar bola do adversário = turno muda (não foul)
  opponentPotTurnChange: true,
  
  // R7: Cue ball pocketed = foul + ball-in-hand anywhere
  scratchBallInHand: 'anywhere',
  
  // R8: 8-ball pocketed antes de completar grupo = PERDE
  earlyEightBall: 'lose',
  
  // R9: 8-ball + cue ball pocketed no mesmo shot = PERDE
  eightBallScratch: 'lose',
  
  // R10: Não tocar bola do grupo primeiro = foul
  wrongBallFirst: 'foul',
};

function processShot(state, result) {
  const { pocketed, firstBallHit, cueBallPocketed, cushionContacts } = result;
  
  // Check cue ball scratch
  if (cueBallPocketed) {
    // R9: se 8-ball também pocketed → PERDE
    if (pocketed.includes('eight')) return { gameOver: true, loser: state.currentTurn, reason: 'eightBallScratch' };
    // R7: foul + ball-in-hand
    return { foul: true, ballInHand: 'anywhere', turnChange: true };
  }
  
  // Check 8-ball pocketed
  if (pocketed.includes('eight')) {
    // R8: grupo não completo → PERDE
    if (!isGroupComplete(state, state.currentTurn)) {
      return { gameOver: true, loser: state.currentTurn, reason: 'earlyEightBall' };
    }
    // Grupo completo → GANHA
    return { gameOver: true, winner: state.currentTurn, reason: 'legalEightBall' };
  }
  
  // Check open table assignment (R3)
  if (state.phase === 'OPEN_TABLE' && pocketed.length > 0) {
    const firstPocketed = pocketed[0];
    const isSolid = parseInt(firstPocketed.replace('ball_', '')) <= 7;
    state.players[state.currentTurn].group = isSolid ? 'solid' : 'stripe';
    state.players[getOpponent(state.currentTurn)].group = isSolid ? 'stripe' : 'solid';
    state.phase = 'ASSIGNED';
  }
  
  // Check wrong ball first (R10)
  if (state.phase === 'ASSIGNED' && firstBallHit) {
    const hitGroup = getGroup(firstBallHit);
    if (hitGroup !== state.players[state.currentTurn].group && hitGroup !== 'eight') {
      return { foul: true, ballInHand: 'anywhere', turnChange: true };
    }
  }
  
  // Check legal pot (R4) vs no pot (R5) vs opponent pot (R6)
  const myPocketed = pocketed.filter(b => getGroup(b) === state.players[state.currentTurn].group);
  if (myPocketed.length > 0) return { turnChange: false }; // R4
  if (pocketed.length === 0) return { turnChange: true }; // R5
  return { turnChange: true }; // R6 (opponent ball pocketed)
}
```

**Impacto:** ~80 linhas. Cobre 95% das situações reais de jogo. Edge cases ultra-raros (double-hit, jump shot) não implementados — aceitável para RP.

SELECT tier, COUNT(*) as matches, SUM(rake) as rake_revenue
FROM casino_pool_matches WHERE status = 'completed'
GROUP BY tier ORDER BY tier;

-- Jogadores suspeitos (muitas vitórias seguidas)
SELECT winner_name, COUNT(*) as wins, SUM(prize) as total_won
FROM casino_pool_matches WHERE status = 'completed' AND winner_id IS NOT NULL
  AND created_at >= DATE_SUB(NOW(), INTERVAL 24 HOUR)
GROUP BY winner_id HAVING wins > 15 ORDER BY wins DESC;

-- Audit de ações admin
SELECT * FROM casino_pool_audit WHERE action = 'CONFIG_CHANGE'
ORDER BY created_at DESC LIMIT 50;

-- Cues mais populares
SELECT c.name, c.tier, COUNT(pc.id) as owners, 
       SUM(pc.equipped) as equipped_count
FROM casino_pool_cues c
LEFT JOIN casino_pool_player_cues pc ON c.id = pc.cue_id
GROUP BY c.id ORDER BY owners DESC;

// Roda da Fortuna - Canvas 2D com offscreen pre-render
// Arquitetura: resultado server-side via PF, animacao client-side

// 1. Pre-render offscreen (executar 1x no mount)
const offscreen = document.createElement('canvas');
offscreen.width = 600; offscreen.height = 600;
const offCtx = offscreen.getContext('2d');

function drawWheel(ctx, segments, radius) {
  const arcSize = (2 * Math.PI) / segments.length;
  segments.forEach((seg, i) => {
    ctx.beginPath();
    ctx.moveTo(radius, radius);
    ctx.arc(radius, radius, radius - 12, i * arcSize, (i + 1) * arcSize);
    ctx.closePath();
    // gradiente por segmento (profundidade visual)
    const grad = ctx.createRadialGradient(radius, radius, 0, radius, radius, radius);
    grad.addColorStop(0, seg.colorLight);  // #2A2A2A centro
    grad.addColorStop(1, seg.colorDark);   // #141414 borda
    ctx.fillStyle = grad;
    ctx.fill();
    // icone do premio (PNG pre-carregado)
    ctx.save();
    ctx.translate(radius, radius);
    ctx.rotate(i * arcSize + arcSize / 2);
    ctx.drawImage(seg.icon, radius * 0.4, -16, 32, 32);
    // texto do valor
    ctx.fillStyle = '#D4A843';
    ctx.font = '700 14px Cinzel';
    ctx.fillText(seg.label, radius * 0.55, 5);
    ctx.restore();
  });
  // moldura metalica (anel dourado com textura)
  drawMetallicFrame(ctx, radius);
  // hub central com mini-logo
  drawHub(ctx, radius);

// 2. Animacao do spin (requestAnimationFrame)

const friction = 0.985; // desaceleracao natural

    mainCtx.drawImage(offscreen, -center, -center);
    mainCtx.restore();

# RISCOS EXPANDIDOS — ANALISE DETALHADA

## Risco 1: Canvas 2D com GPU compartilhada no FiveM

O FiveM roda dentro do GTA V, que ja consome massivamente a GPU. O NUI (Chromium Embedded Framework M103) compartilha a GPU com o jogo. Canvas 2D no NUI renderiza via skia (software) ou via GPU compositing — depende da complexidade. O risco: durante momentos de alta carga no GTA (muitos jogadores na tela, explosoes, particulas), o Canvas 2D pode perder frames.

A mitigacao e concreta: offscreen pre-render. A roda inteira e desenhada UMA VEZ em um canvas offscreen (600x600px). Durante a animacao de spin, o unico trabalho por frame e: clearRect + save + translate + rotate + drawImage + restore. Isso e ~6 operacoes por frame, o que e trivial ate para software rendering. ShadowBlur esta DESLIGADO (shadowBlur do Canvas e extremamente custoso porque reprocessa o blur a cada frame). Glow e conseguido via radialGradient fake (pre-renderizado no offscreen) ou CSS no wrapper DOM ao redor do Canvas.

Validacao: o mesmo approach ja e usado no componente de Slots do Blackout Casino (components/games/slots/VideoGrid.tsx) e funciona sem problemas no NUI.

## Risco 2: Timezone e reset de 24 horas

O Daily-Free depende de um timer de 24 horas entre claims. O jogador esta no Brasil (UTC-3), o servidor FiveM pode estar em qualquer timezone. Se o reset usar horario local do jogador, ha inconsistencia. Se usar midnight UTC, um jogador brasileiro poderia claimar as 20:59 (23:59 UTC) e de novo as 21:01 (00:01 UTC do dia seguinte) — duas claims em 2 minutos.

A mitigacao e server-side TIMESTAMP: armazenar last_claim_at como TIMESTAMP UTC no MySQL. A verificacao e: `WHERE player_identifier = ? AND last_claim_at > DATE_SUB(NOW(), INTERVAL 24 HOUR)`. Se retorna resultado, o jogador ja claimou nas ultimas 24h. NAO usar DATE (que reseta a meia-noite), usar TIMESTAMP com intervalo fixo de 24h. O grace period de 2h (configuravel) permite que o jogador atrase ate 2h sem quebrar o streak: se last_claim_at + 26h > NOW(), streak continua.

## Risco 3: Abuse por multi-personagem

No FiveM, um jogador pode ter multiplos personagens (chars). Sem protecao, ele cria 5 chars e claima 5x por dia, multiplicando a recompensa. A mitigacao: vincular o claim ao `player_identifier` (hardware ID do FiveM, equivalente ao license), NAO ao char_id. A tabela daily_claims tem UNIQUE KEY em (player_identifier, claim_date). Tentativa de segundo claim no mesmo dia retorna erro "Voce ja recebeu seu bonus hoje!" independente do char usado. O char_id e armazenado para registro mas nao faz parte da restricao.

## Risco 4: Desync de streak apos crash/disconnect

Se o jogador abre o Daily-Free, gira a roda, mas o FiveM crasha ANTES do callback de confirmacao chegar ao server, o claim pode ser perdido. O streak marcaria "dia perdido" indevidamente. A mitigacao: o claim e registrado no banco ANTES de enviar o resultado para o client. O fluxo e: (1) client envia spin request, (2) server calcula resultado + registra no banco, (3) server retorna resultado ao client, (4) client anima. Se o crash acontece entre 3 e 4, o claim ja esta registrado. O client ao reconectar verifica `daily_claims WHERE claim_date = CURDATE()` e se existe resultado, mostra "Voce ja girou hoje! Resultado: X GCoin" sem precisar girar de novo.

## Risco 5: Inflacao economica por Daily-Free generoso demais

Se os premios da roda forem muito altos, o Daily-Free pode inflacionar a economia do servidor (muito GCoin gratuito circulando). A mitigacao: todos os valores sao configuraveis pelo admin via tabela daily_rewards. O admin pode ajustar os segmentos, pesos e valores a qualquer momento sem restart do servidor. A recomendacao padrao (mediana 150 GC/dia, maximo mensal ~11.000 GC) e conservadora — equivale a ~22 apostas minimas. Monitoramento via query: `SELECT SUM(total_awarded) FROM daily_claims WHERE claim_date >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)` mostra o total distribuido no ultimo mes.

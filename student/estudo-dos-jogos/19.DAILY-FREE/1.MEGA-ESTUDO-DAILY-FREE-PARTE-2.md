## Problema 1: Timezone e Reset de 24 Horas

**Sintoma:** Jogador pode claimar 2x em menos de 2 minutos se o reset usar midnight UTC. Um jogador em UTC-3 (Brasil) claima as 20:59 BRT (23:59 UTC) e de novo as 21:01 BRT (00:01 UTC do dia seguinte).

**Causa:** O campo `claim_date` usando tipo DATE reseta a meia-noite UTC, criando janela de double-claim. O FiveM nao expoe timezone do jogador ao server de forma confiavel — o server conhece apenas o horario do proprio sistema.

**Codigo Real do Problema:**
```sql
-- ERRADO: usar DATE permite double-claim na virada de meia-noite
SELECT * FROM daily_claims
WHERE player_identifier = ? AND claim_date = CURDATE();
-- Jogador claima 23:59 UTC = claim_date = '2026-03-24'
-- Jogador claima 00:01 UTC = claim_date = '2026-03-25'
-- Ambas queries retornam vazio = 2 claims em 2 minutos
```

**Solucao:**
```sql
-- CORRETO: usar TIMESTAMP com intervalo fixo de 24h
SELECT * FROM daily_claims
WHERE player_identifier = ?
  AND created_at > DATE_SUB(NOW(), INTERVAL 24 HOUR)
LIMIT 1;
-- Se retorna resultado = jogador ja claimou nas ultimas 24h
-- Independente de timezone, midnight, ou data
```

No handler JS:
```javascript
// Verificar cooldown de 24h (server-side, anti-cheat)
const [rows] = await oxmysql.query(
  'SELECT id, created_at FROM daily_claims WHERE player_identifier = ? AND created_at > DATE_SUB(NOW(), INTERVAL 24 HOUR) LIMIT 1',
  [playerIdentifier]
);
if (rows.length > 0) {
  const lastClaim = new Date(rows[0].created_at);
  const nextClaim = new Date(lastClaim.getTime() + 24 * 60 * 60 * 1000);
  const remaining = nextClaim - Date.now();
  return cb({ success: false, error: 'COOLDOWN', remaining_ms: remaining });
}
```

## Problema 2: Abuse Multi-Personagem

**Sintoma:** Jogador com 5 personagens no servidor claima o Daily-Free com cada personagem, recebendo 5x a recompensa diaria (ex: 5 x 200 GC = 1.000 GC/dia em vez de 200 GC/dia).

**Causa:** Se o claim for vinculado ao `char_id` (personagem), cada personagem e tratado como jogador independente. O FiveM permite multiplos personagens por conta.

**Codigo Real do Problema:**
```sql
-- ERRADO: vincular ao char_id permite abuse
CREATE TABLE daily_claims (
  player_identifier VARCHAR(64),  -- ignorado na query
  char_id INT NOT NULL,           -- usado como restricao
  claim_date DATE NOT NULL,
  UNIQUE KEY idx_char_date (char_id, claim_date)  -- permite 1 claim por CHAR por dia
);
-- Jogador com 5 chars: 5 claims validos por dia
```

**Solucao:**
```sql
-- CORRETO: vincular ao player_identifier (hardware ID/license)
CREATE TABLE daily_claims (
  player_identifier VARCHAR(64) NOT NULL,
  char_id INT DEFAULT NULL,  -- registrado mas NAO usado como restricao
  claim_date DATE NOT NULL,
  UNIQUE KEY idx_player_date (player_identifier, claim_date)  -- 1 claim por PESSOA por dia
);
-- Mesmo jogador com 5 chars: apenas 1 claim valido por dia
```

No handler JS, o `player_identifier` e obtido via `GetPlayerIdentifiers(source)` no FiveM, que retorna o license/hardware ID unico da maquina. Isso garante que mesmo trocando de char, o jogador e identificado pela mesma license.

## Problema 3: Canvas 2D Performance com GPU Compartilhada

**Sintoma:** A roda da fortuna pode ter stuttering (drop de FPS) durante momentos de alta carga no GTA V (muitos jogadores, explosoes, particulas pesadas do jogo).

**Causa:** O NUI do FiveM (Chromium Embedded Framework) compartilha a GPU com o GTA V. Canvas 2D pode ser renderizado via software (skia) ou via GPU compositing — depende da complexidade e da disponibilidade de GPU. Se a GPU estiver saturada pelo GTA, o Canvas perde frames.

**Codigo Real do Problema:**
```javascript
// ERRADO: redesenhar TODA a roda a cada frame
function animate() {
  ctx.clearRect(0, 0, w, h);
  // Redesenhar 8 segmentos com arc() + gradiente + texto + icone
  segments.forEach((seg, i) => {
    ctx.beginPath();
    ctx.arc(cx, cy, r, i * arcSize, (i+1) * arcSize);
    ctx.fillStyle = createRadialGradient(...); // CUSTOSO por frame
    ctx.fill();
    ctx.fillText(seg.label, ...);  // CUSTOSO: glyph rendering
    ctx.drawImage(seg.icon, ...);   // OK se pre-carregado
  });
  // = 8 arcs + 8 gradientes + 8 texts + 8 drawImages POR FRAME = PESADO
  requestAnimationFrame(animate);
}
```

**Solucao:**
```javascript
// CORRETO: offscreen pre-render + unico drawImage por frame
// 1. Pre-render a roda completa em offscreen canvas (1x no mount)
const offscreen = document.createElement('canvas');
offscreen.width = 600; offscreen.height = 600;
const offCtx = offscreen.getContext('2d');
drawFullWheel(offCtx, segments); // custoso mas executa 1x

// 2. Animacao: apenas 1 drawImage rotacionado por frame
function animate() {
  mainCtx.clearRect(0, 0, w, h);
  mainCtx.save();
  mainCtx.translate(cx, cy);
  mainCtx.rotate(currentAngle);
  mainCtx.drawImage(offscreen, -cx, -cy); // UNICA operacao por frame
  mainCtx.restore();
  requestAnimationFrame(animate);
}
// = 1 clearRect + 1 drawImage POR FRAME = LEVE
```

Adicionalmente: `ctx.shadowBlur = 0` (NUNCA usar shadowBlur no Canvas — forca reprocessamento de blur a cada frame). Glow e feito via CSS no wrapper DOM ao redor do Canvas, ou via radialGradient pre-renderizado no offscreen.

## Problema 4: Streak Desync apos Crash/Disconnect

**Sintoma:** Jogador gira a roda, ve a animacao, mas o FiveM crasha ANTES do callback de confirmacao chegar ao client. O jogador pensa que nao claimou. Ao reconectar, tenta girar de novo — server rejeita (ja registrado). Ou pior: se o crash for ANTES do INSERT no banco, o claim se perde e o streak pode quebrar.

**Causa:** A comunicacao NUI → Lua → Server e assincrona. O callback pode falhar silenciosamente se a conexao cair no meio.

**Solucao:** Registrar o claim no banco ANTES de retornar o resultado ao client. O fluxo seguro e:

```javascript
// server/handlers/daily-free.js
async function handleSpin(source, data, cb) {
  // 1. Calcular resultado (PF)
  const result = calculateWheelResult(serverSeed, data.clientSeed, data.nonce);

  // 2. REGISTRAR NO BANCO PRIMEIRO (antes de retornar ao client)
  await oxmysql.insert(
    'INSERT INTO daily_claims (player_identifier, char_id, claim_date, wheel_segment, gcoin_awarded, ...) VALUES (?, ?, CURDATE(), ?, ?, ...)',
    [identifier, charId, result.segment, result.prize, ...]
  );

  // 3. Creditar GCoin
  await oxmysql.query('UPDATE players SET gcoin = gcoin + ? WHERE identifier = ?', [result.totalPrize, identifier]);

  // 4. SO ENTAO retornar ao client
  cb({ success: true, segment: result.segment, prize: result.prize, streak: result.newStreak });
}
```

Se o client crashar apos passo 2 mas antes de receber passo 4, o claim ja esta no banco. Ao reconectar, o client verifica:

```javascript
// client: ao abrir a tela Daily-Free, verificar se ja claimou hoje
const todayClaim = await fetchNui('daily-free:check', {});
if (todayClaim.alreadyClaimed) {
  // Mostrar resultado do claim anterior + timer para proximo
  showAlreadyClaimed(todayClaim.result, todayClaim.nextClaimAt);
}
```

## Problema 5: Conflito NUI Focus com GTA

**Sintoma:** Apos fechar o Daily-Free, o teclado/mouse do GTA continuam capturados pelo NUI. Jogador nao consegue andar, dirigir ou interagir com o GTA.

**Causa:** `SetNuiFocus(true, true)` ativa o focus do NUI (teclado + mouse). Se o fechamento do componente React nao disparar o `SetNuiFocus(false, false)` no Lua, o focus fica preso.

**Solucao no Client Lua:**
```lua
-- client/daily-free_client.lua
RegisterNUICallback('daily-free:close', function(_, cb)
  SetNuiFocus(false, false)
  SendNUIMessage({ action = 'hide' })
  cb({ ok = true })
end)

-- Fallback: se o NUI nao enviar o callback de close, timeout automatico
RegisterNUICallback('daily-free:keepalive', function(_, cb)
  cb({ ok = true })
end)

-- Se o keepalive falhar por 10s, forcar fechamento
Citizen.CreateThread(function()
  while dailyFreeOpen do
    local success = exports['daily-free']:sendKeepAlive()
    if not success then
      SetNuiFocus(false, false)
      dailyFreeOpen = false
    end
    Citizen.Wait(10000)
  end
end)
```

Este e o mesmo pattern usado nos jogos existentes (Slots, Cases, Roulette) e ja esta validado no projeto.

## 5.3 Queries Admin Uteis

```sql
-- Total distribuido nos ultimos 30 dias
SELECT SUM(total_awarded) AS total_30d,
       COUNT(*) AS claims_30d,
       AVG(total_awarded) AS media_por_claim
FROM casino_daily_claims
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY);

-- Top 10 jogadores por streak
SELECT player_identifier, current_streak, longest_streak, total_gcoin_earned
FROM casino_daily_streaks
ORDER BY current_streak DESC LIMIT 10;

-- Distribuicao de premios por segmento (frequencia real vs peso)
SELECT wheel_segment, COUNT(*) AS vezes, SUM(gcoin_awarded) AS total_gcoin
FROM casino_daily_claims
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY wheel_segment ORDER BY wheel_segment;

-- Jogadores que atingiram milestone 30 dias
SELECT player_identifier, total_claims, total_gcoin_earned, streak_start_date
FROM casino_daily_streaks
WHERE milestone_30_claimed = 1
ORDER BY total_gcoin_earned DESC;
```
